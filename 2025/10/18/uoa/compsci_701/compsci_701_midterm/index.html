Creating Maintainable Software - midterm review
2025-10-18 23:16   label UoA

Creating Maintainable Software

Better maintainability → lower maintenance cost → higher overall software quality.

Maintenance:

  • corrective - remove defects
  • adaptive - change the product to a changed environment
  • preventative - improving some aspect of its quality
  • perfective - improve some aspect of its usefulness, including adding new features

Maintainability

definition: The degree of effectiveness and efficiency with which maintenance activities can be carried out on the product.

Maintenance costs dominate the lifetime cost of a software product. The more maintainable the software, the lower the maintenance costs, and so the lower the lifetime cost.

Each part of Maintenance is independent. (E.G. some code is “hard” to understand but “easy” to change)

maintenance activities:

  • comprehensibility (analysability) : implementation can be understood.
  • alterability (modifiability) : system can be changed without introducing defects.
  • testability : test criteria can be established.

​

Comprehensibility

PCM

Program Comprehension Model

  • Knowledge base

    • context schema - > domain knowledge

    • design schema -> knowledge of programming language

    • Mental model : current understanding of the code

  • Assimilation Process : use information to update Mental model to understand the program

  • External Representation : code, documents …

How to use PCM

Figure out the occasion affects which links between these parts.

Comprehensibility depends on what is in KB, what is available in the ER, how efficient the assimilation process is.

Alterability

Change Cases: specifying what needs to change:

  • likelihood (the case will occur)
  • potential impact
    • to the requirements, to the behavior of existing system

Assess the alterability of the design

Focus on change cases with high likelihood and high impact

(lower likelihood, lower impact -> less affect on alterability assessment)

  • change one… have less work than multiple…
    • class, method, statement
  • simple work have less work than complex work

less need to change -> more alterable

  • fewer places to deal with -> efficiency
  • less likely to make mistakes -> effectiveness

Testability

Testing Scopes

  • unit testing - single class or subsystem (small number of classes)
  • integration testing - combination of subsystems
  • system testing - the whole system

Test Case

a set of actions performed on the Implementation Under Test (IUT) to determine if it results in the correct behavior

  • pretest state -> inputs -> expected state or behavior

Test Criteria

requirements on a set of test cases must pass

improving testability : controllability, observability -> more classes, methods -> automate

Naming

using PCM to determine better name: easy to add to MM > easily found in KB > avoid going to ER

Meaningful Names

  • Intention-Revealing : the name can explain its function without comments
  • Avoid Encoding : don’t use special prefix or name styles for different types

Code Quality Principles (CQP)

Code Style Mode

Clear Layout -> Clear Presentation

  • clear layout between different elements:
    • characters, space, indentations, brackets, line length

Explanatory Language

  • clear intent and meaning: documentation, names(variables, functions), language feature(annotations)
  • don’t reuse for different purposes within the same scope

Simple Constructs

  • minimize nesting
  • easy flow of control (with few breaks and exceptions)
  • easy expression

Be Consistent

  • consistent implementation for similar jobs
  • using language idiom (documentation, notation) consistently

Avoid Duplication -> Single Definition

  • using loops or methods, classes instead of repeated codes

No Unused Content -> Contributing Content

  • remove unreachable code, redundant branches, unused documentation

Modular Structure

  • placing related code together with a clear task
  • minimal dependence on other modules, and the scope of constants and variables
  • functions aim to a single task, minimum parameters

Congruent Implementation

  • implementation (structure, process) are consistent with the nature of the problem

Maintainable Functions

structured programming: breaks things into separately-callable units

  • small

    • decomposed a large function into several smaller functions

  • do one thing <- identify duplication

    • reduce revisiting

  • explain yourself in code

    • use small function to create “self-documenting” code
    • avoid use of literal values

  • one level of abstraction per function -> small

    • avoid “high level” and “low level” concepts in the same function
    • reduce number of levels of abstraction

  • reading code from top to bottom -> order of functions ▽

    • the first function is the most abstract, call to other

  • use descriptive names <- the one thing it does

  • reduce number of parameters

    • Wrap some of the existing arguments into a class

  • reduce returns

  • have no side effects

Classes

  • Organisation
    • order: constants -> other static -> instance -> constructors -> public methods -> non public methods
  • Encapsulation -> object behavior is consistent with the concept it represents
  • Small
  • Single Responsibility Principle (SRP)
  • Cohesion

Class Quality

  • abstraction (matches a concept from the context or design schema)
  • single responsibility principle
  • cohesion : fields associated with methods (rather than using a few fields in methods)

Tools for comprehending programs

  • program analysis tools

    • AST (abstract syntax tree), identify potential quality issues

  • software visualization tools <- limited evidence shows helps

    • software metrics (number of classes), relationships between elements

  • program languages

  • PMD - linter to detect

  • NcssCount (Non-commenting Source Statements)

  • ControlStatementBraces

  • FieldVisibility : fields should not accessible from other classes

  • Visualizations of object creation

  • Automated “Concept” Extraction (ACE)

    • things to recognise
      • Design (and Architecture) Patterns — common structure organizations
      • Abstractions — high-level ideas (typically from problem)
      • Across Abstractions — characteristics that occur across abstractions
      • Across Packages — characteristics that occur across packages
    • Threat to Validity: Not use meaningful names in a consistent manner
    • Benefits to understanding: Seeing dependencies that between different concepts
    • Extraction Process
      • divide names into component -> keywords
      • look for classes at the top of inheritance corresponding to keywords
      • look for keywords occur in multiple places, corresponding to patterns

  • UML (Unified Modeling Language)

a visual language used to describe software: class, sequence, use case, deployment

  • Remix Programming language

use standard compiler technology to make code readable
