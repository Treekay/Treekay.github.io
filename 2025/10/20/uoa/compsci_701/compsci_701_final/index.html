Creating Maintainable Software

  1. 可维护性的三大支柱

可理解性 (Comprehensibility/Analysability)：快速准确地理解代码的实现和意图。

可修改性 (Alterability/Modifiability)：高效地对产品进行更改，而不引入缺陷或降低现有质量。

可测试性 (Testability)：有效率地建立和执行测试的能力。

  1. 程序理解模型 (PCM)

理解开发者如何通过外部表示（代码、文档）和自身知识（知识库）构建对程序的心理模型。

所有的代码质量实践，其最终目标都是优化 PCM 中的同化过程，减少理解成本。

  1. 面向对象设计的维护性基础

Context Schema（上下文图式）：关注问题域的概念，类设计应与其匹配 。

Encapsulation（封装）：确保对象行为与所代表的概念一致 。

Abstraction（抽象）：类应是对现实或设计概念的良好抽象 。

  1. 代码结构和质量原则 (CQP)

Code Quality Principles (CQP)：提供了一个通用的框架来解释和应用编程约定 。

DRY (Don’t Repeat Yourself)：避免代码重复，重复代码会增加阅读量和维护风险 。

命名原则：使用意图揭示的名称，避免模糊和误导性的名称，以提高可理解性 。

  1. 函数和类的设计原则

  • 函数 (Functions)：

    • “One Thing”：函数应只做一件事 。

    • 统一抽象级别：函数中所有语句都应在同一抽象级别上 。

    • 参数数量：尽量减少参数数量，或使用 Argument Object 来封装相关参数 。

  • 类 (Classes)：

    • 单一职责原则 (SRP)：一个类应只有一个修改的理由 。SRP 是提高可修改性的核心。

    • 高内聚 (High Cohesion)：类中的内部元素应紧密相关地执行同一任务 。

  1. 代码异味与重构实践 (Unencapsulated Collection)

代码异味：Unencapsulated Collection（未封装的集合）是一个重要的设计问题 。

重构方法：通过创建一个集合类，将集合设为私有字段，并将所有针对该集合的业务逻辑移动到该类的方法中，从而实现封装 。

重构收益：显著提高内聚性、增强封装、降低变更风险
