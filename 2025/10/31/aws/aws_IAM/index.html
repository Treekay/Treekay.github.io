IAM policy

题型类别 常见关键词 典型考点
访问控制设计题 “restrict access / manage permissions / different users” IAM Users / Groups / Roles / Policies
安全改进题 “public/private key login / shared credentials / security concern” 改成 IAM Role、Group、最小权限原则
运维管理题 “too many users / need central management” IAM Groups、Policies、Organizations
应用访问 AWS 资源题 “EC2 accessing S3 / Lambda accessing DynamoDB” 绑定 IAM Role 给服务访问权限
多账户与合规题 “multiple departments / compliance” AWS Organizations + Service Control Policies

Example:

Use IAM Roles and Groups instead of individual IAM users or key-based access. Assign permissions through IAM Policies following the principle of least privilege.

Create Groups based on job functions (e.g., Admin, Developer, Auditor). Attach policies defining what each group can access. Attach IAM Roles to AWS resources (like EC2, Lambda) so they can access only required services (e.g., S3, RDS).

Benefits:

  • Centralized permission management
  • Easier to update or revoke access
  • Reduces risk from hardcoded or shared credentials
  • Enforces least privilege and improves compliance

Cons:

  • Requires careful policy planning
  • Initial setup is time-consuming
  • May need training for administrators

关键概念对比表

项目 定义 典型用途 注意点
IAM User 单个用户身份 日常操作、控制台登录 不宜过多创建
IAM Group 用户集合 同类用户权限集中管理 不能嵌套组
IAM Role 临时授权身份 EC2、Lambda 等服务访问 AWS 资源 无长期凭证，更安全
Policy JSON 格式权限规则 限定具体操作与资源 遵循 least privilege
Root User 账户所有者 仅用于账户级操作 禁止日常使用
Access Key 编程访问凭证 CLI/API 操作 应定期轮换或改用 Role
