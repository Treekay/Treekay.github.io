Join Expressions

Concepts

Join operations combine tuples from two or more relations to produce another relation. The core ideas are:

  • Natural join: Equates all attributes with the same name across the two relations and returns one copy of each common attribute. Convenient, but risky when unrelated columns share the same name.
  • Inner join (join): The default join that returns only matching tuples. You can specify:
    • join … using (A1, A2, …): Match on the listed attributes only (and deduplicate those columns in the result).
    • join … on : Match on any Boolean condition (like a WHERE clause just for the join). Does not automatically deduplicate columns.
  • Outer joins: Preserve unmatched tuples using nulls to fill missing columns:
    • left outer join: keep all rows from the left table
    • right outer join: keep all rows from the right table
    • full outer join: keep all rows from both tables

Rule of thumb: Prefer explicit join … using or join … on when schemas share multiple identically named columns or risk accidental equivalence. Use outer joins to retain rows without matches.

Example 1: Natural join for student-course enrollments

Goal: List each student’s name and the course_id values for the courses they have taken. Here, student and takes both have a column named ID used to match.

Explanation: natural join automatically matches on all identically named columns (here, ID). It returns only one copy of ID in the output, along with name and course_id.

Example 2: Avoiding the natural join pitfall with join … on

Goal: List each student’s name and the course title for the courses they have taken.

Suppose student, takes, and course have multiple columns with the same names (e.g., dept_name) that are unrelated for matching. A chained natural join can accidentally equate unrelated attributes. Use join … on to be explicit.

Explanation: We join s to t using the student’s ID and then t to c using course_id, avoiding accidental equivalences on unrelated same-name columns.

Example 3: Inner join using … versus natural join

Goal: List each student’s name and the course title for the courses they have taken.

Both return matching rows, but natural join deduplicates all shared columns; inner join with on does not. using strikes a middle ground (deduplicates the listed attributes).

Explanation: using is concise and prevents duplicated join columns from appearing multiple times, while on gives maximum flexibility at the cost of manual projection to avoid duplicate columns.

Example 4: Outer joins to retain non-matching rows

Goal: List all students and any courses they have taken; for students with no enrollments, course fields appear as null.

Explanation: left outer join preserves all students. If a student has no matching rows in takes, the course-related columns are null. Similarly, right outer join preserves the right table, and full outer join preserves both sides.

Example 5: Chaining multiple joins safely

Goal: Show student names and titles of all courses they have taken, including those who haven’t taken any course (title is null in that case).

Explanation: The first left outer join preserves all students; the second left outer join brings in course titles where available. Unenrolled students appear with null titles.

Views (Virtual Relations)

Concepts

A view is a virtual relation defined by an SQL query. You create it with create view v as . The database stores the definition, not the rows, and computes the result on demand. Views can depend on other views (view expansion resolves the dependencies). Some systems also support materialized views that persist the computed rows and need maintenance when base data changes.

Updating through views is restricted: generally only simple views (derived from a single base table, no aggregates, no distinct, no expressions, and no grouping) are updatable. Complex views (joins, aggregates) are usually read-only unless special rules or triggers are provided.

Transactions

Concepts

A transaction is a sequence of SQL statements that executes as a single atomic unit: either all its effects persist (commit) or none do (rollback). Transactions provide isolation from concurrent work and ensure the database is not left in a partial, inconsistent state.

Integrity Constraints

Concepts

Integrity constraints enforce business rules and protect against incorrect data:

  • NOT NULL: Forbid missing values for listed columns.
  • PRIMARY KEY: Unique identifier for rows; implies NOT NULL.
  • UNIQUE: Ensures no duplicate values (or tuples of values) across rows; can include nulls (DBMS-dependent).
  • CHECK (predicate): Row-level predicate that must be satisfied for every row (e.g., check (semester in (‘Fall’,‘Winter’,‘Spring’,‘Summer’))).
  • FOREIGN KEY: Enforces referential integrity across tables by requiring referenced values to exist in the target table’s key. Authorization

Concepts

Authorization governs who can see or modify which data. Privileges include DML (select, insert, update, delete) and DDL (index, create table, alter table, drop table). DBAs grant and revoke privileges for users or roles. You can grant with grant option to allow recipients to pass privileges onwards.
