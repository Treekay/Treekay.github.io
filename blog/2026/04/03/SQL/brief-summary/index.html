Brief summary for SQL

SQL overview

evolving standards (SQL-86 to SQL:2003); key components include DDL, DML, Views, Transaction control, Authorization, Embedded/Dynamic SQL.

Data Definition (DDL)

CREATE TABLE with types (char, varchar, numeric(p,d)); integrity constraints (PRIMARY KEY, FOREIGN KEY, NOT NULL); composite keys and referential integrity.

Basic queries

SELECT–FROM–WHERE; case-insensitive; duplicates allowed by default; use DISTINCT or ALL; SELECT *; expressions and column renaming with AS; joins via FROM + WHERE; self-joins with table aliases.

Filtering

logical operators (AND/OR/NOT); comparisons (<, <=, >, >=, =, <>); BETWEEN; tuple comparisons.

Strings

LIKE with % (substring) and _ (single char); ESCAPE for literals; functions such as concatenation (||), UPPER/LOWER, LENGTH, SUBSTRING.

Ordering

ORDER BY with ASC/DESC; multiple sort keys supported.

Set operations

UNION, INTERSECT, EXCEPT (and their ALL variants); default behavior removes duplicates unless ALL is used.

Null semantics

NULL means unknown/absent; arithmetic/comparisons with NULL yield unknown; three-valued logic in WHERE; test with IS NULL / IS NOT NULL.

Aggregates

AVG, MIN, MAX, SUM, COUNT; GROUP BY forms groups; HAVING filters groups (vs WHERE filters rows); typical evaluation order: FROM → WHERE → GROUP BY → HAVING → SELECT.

Subqueries:

  • In WHERE: IN/NOT IN, SOME/ALL, EXISTS/NOT EXISTS, UNIQUE; supports correlated subqueries.
  • In FROM: derived tables; WITH (CTE) for temporary named subqueries.
  • In SELECT: scalar subqueries (must return a single value).

Data modification:

  • INSERT with VALUES or INSERT…SELECT; NULLs can be inserted explicitly.
  • DELETE with optional subqueries; beware changing aggregates during deletion (use CTEs to stage results).
  • UPDATE with SET and WHERE; use CASE for conditional updates; scalar subqueries can recompute derived columns.

Schema changes:

DROP TABLE; ALTER TABLE ADD COLUMN (existing rows get NULL) and (limited) DROP COLUMN support.

Portability note:

some/all set-comparison clauses may not be supported in all engines (e.g., SQLite).

Intermediate SQL: Core Components

Join Expressions

Natural join: matches rows on all common attribute names and keeps a single copy; risk of incorrect matches if unrelated columns share names. Inner join (join): use USING (col1, col2, …) to specify match columns, or ON predicate for general conditions; unlike natural join, it does not deduplicate same-named columns.
Outer joins: left/right/full preserve non-matching rows by padding NULLs; use ON or USING to define the join condition.
Key idea: join condition defines how rows match; join type defines how unmatched rows are handled.

Views

Virtual relations defined by create view v as ; used for abstraction and security.
Materialized views store results physically and require maintenance on updates.
Updates generally allowed only on simple views (single base table, no aggregates/distinct/group by); view dependencies expand to underlying queries.

Transactions

Atomic sequence of operations with isolation from concurrent work.
commit makes changes permanent; rollback undoes all changes in the transaction.

Integrity Constraints

Single-table: NOT NULL; PRIMARY KEY (implies NOT NULL); UNIQUE; CHECK (custom predicates, e.g., value in a set).
Multiple tables: FOREIGN KEY enforces referential integrity (typically referencing a primary key).
On update/delete actions: default reject, or CASCADE, SET NULL, SET DEFAULT to control propagation.

SQL Data Types and Schemas

Date/Time types: date, time, timestamp, interval.
Large objects: BLOB (binary) and CLOB/TEXT (character); queries often return locators.
User-defined types/domains via create type and create domain (domains can include constraints).

Index Definition

Indexes speed equality and range searches without changing queries.
Create with create index on

(column_list); used automatically by the optimizer.

Authorization

Privileges for DML/DDL (select, insert, update, delete, create/alter/drop/index) managed via GRANT and REVOKE.
Grant option allows re-granting; revocation can cascade or restrict.
View privileges do not imply privileges on underlying tables.
Roles simplify management: assign privileges to roles, then grant roles to users; roles can chain.
