Storage in Database System
2026-05-04 14:38   label SQL

1) Storage Hierarchy and Performance

Volatile (cache, RAM): fast, byte-addressable, expensive; Non-volatile (SSD, HDD, tape): slower, larger, cheaper, block-addressable.
HDDs: mechanical seek + rotational latency; great sequential bandwidth (≈25–200 MB/s), poor random IOPS (≈50–200).
SSDs: no moving parts; very high random IOPS (10k–100k+), SATA ≈400 MB/s, NVMe ≈2–3+ GB/s; page-out-of-place writes, erase blocks, wear leveling.
Reliability: mean time to failure (MTTF) typically a few years; more disks raise failure probability; hybrids combine flash cache with HDD.
# 2) RAID: Reliability and Parallel Performance
Goals: higher capacity, bandwidth, and reliability via parallelism and redundancy. RAID 0: striping only (high speed, no safety). RAID 1: mirroring (high reliability and reads, 50% capacity overhead). RAID 5: distributed parity (tolerates 1 disk failure; write penalty from read-old-data/parity, compute, write-new).
Choice factors: cost, IOPS/bandwidth, performance during failure/rebuild, rebuild time. Level 1 often better random-write performance than level 5.
# 3) File Organization and Record Representation
Database stored as files of records packed into blocks (I/O unit).
Fixed-length records: simple packing; deletions reuse space via a free list.
Variable-length records: slotted page structure with a header directory of (offset, length); records compacted from page end.
Indirection: external pointers should target the slot directory entry, not the physical record, enabling in-page movement/defragmentation.
File organizations: heap (anywhere), sequential (ordered by key), B+-tree (ordered with efficient inserts/deletes), hashing (direct to bucket).
# 4) Buffer Management
Buffer pool caches disk blocks; buffer manager serves page requests, minimizing I/O.
Pin/unpin: pinned pages can’t be evicted; dirty pages require write-back on eviction (overwrite or write-new-then-invalidate-old).
Replacement: LRU and Clock (approx. LRU). DB-aware policies (e.g., Toss-immediate, MRU) can outperform OS LRU for known access patterns.
# 5) External Sort-Merge (for data larger than memory)
Phase 1 (Run generation): read M blocks, sort in memory, write as a run; repeat.
Phase 2 (Merge): if runs N < M, single N-way merge using N input buffers + 1 output; else do multiple passes merging up to (M−1) runs per pass.
Cost: with relation size br blocks, passes = ⌈logM−1(br/M)⌉; total I/Os = br · (2·⌈logM−1(br/M)⌉ + 1).
Sorting underpins operators like merge join; sequential access patterns reduce seeks and boost throughput.
