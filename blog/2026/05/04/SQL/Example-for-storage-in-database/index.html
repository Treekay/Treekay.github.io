Example for Storage in Database
2026-05-04 15:17   label SQL

1) Introduction to the Physical World and the Storage Hierarchy

Why physical storage matters

Databases ultimately live as bits on physical devices. CPU operations are extremely fast compared to I/O; fetching data from storage is commonly the bottleneck. A well-designed system minimizes slow I/O by organizing data intelligently across a hierarchy of storage devices that trade off speed, capacity, and cost.

The storage hierarchy

  • Volatile storage (loses contents on power loss):
    • CPU caches: fastest, smallest, byte-addressable.
    • Main memory (RAM): fast, byte-addressable random access, but expensive per GB and limited in size for very large datasets.
  • Non-volatile storage (persists without power):
    • Flash memory (SSDs): block-addressable, no moving parts, high random IOPS.
    • Magnetic disks (HDDs): block-addressable, mechanical parts, great for sequential access; lower random IOPS.
    • Magnetic tapes: sequential access, very low cost per TB; used for backups/archival.

Sequential vs random, volatile vs non-volatile

  • Volatile devices are byte-addressable and optimized for random access.

  • Non-volatile devices read/write whole blocks (a.k.a. pages), typically 4–8 KB or larger.

  • Sequential access patterns on disks are much faster than random ones due to reduced seek/rotation overhead. ## Illustrative examples

  • Example: RAM vs SSD vs HDD for a key-value lookup

    • RAM: O(100 ns) access, effectively instant for a single lookup.
    • SSD: tens of microseconds to a fraction of a millisecond per random 4 KB read; tens to hundreds of thousands of IOPS.
    • HDD: 5–10 ms typical random access latency; 50–200 IOPS for 4 KB random reads.
    • Conclusion: indexing and caching matter; pushing frequent lookups to RAM avoids slow I/O.

  • Example: Choosing a device for analytics scans

    • HDD sequential throughput: 25–200 MB/s. SSD sequential: 400 MB/s (SATA3) to multiple GB/s (NVMe PCIe).
    • Large sequential scans (e.g., fact table scans) can be reasonable even on HDDs
    • Random point probes in large tables favor SSDs or memory-resident caches.

2) Performance Measures and Device Behavior

Key metrics

  • Block I/O: Databases transfer data between memory and storage in fixed-size blocks (pages).
  • IOPS (I/O operations per second): capacity for random block operations.
  • Throughput: MB/s, most relevant for sequential scans/transfers.
  • Latency: time per operation; includes HDD seek + rotational latency.
  • MTTF (Mean Time To Failure): expected time until a device fails (e.g., 3–5 years on average for disks).

Representative figures

  • HDD sequential throughput: roughly 25–200 MB/s.
  • SSD sequential throughput: ~400 MB/s (SATA3), 2–3+ GB/s (NVMe PCIe).
  • Random IOPS at 4 KB:
    • HDD: ~50–200.
    • SSD: ~10,000 reads/sec; ~40,000 writes/sec (typical consumer figures vary widely).

Failure scaling

In arrays of many disks, the probability that any disk fails is much higher than the probability that one particular disk fails. For N independent disks, expected time to some disk failing shrinks roughly by a factor of N, making redundancy essential.

Illustrative examples

  • Example: Time to first failure in a 100-disk system
    • Each disk MTTF = 100,000 hours. Expected time to any failure ≈ 100,000 / 100 = 1,000 hours (~41 days).
    • Implication: plan for redundancy and repair workflows.
  • Example: Estimating random I/O latency
    • HDD: 100 IOPS → 10 ms/op on average. Reading 10,000 random blocks ≈ 100 s just in I/O time.
    • SSD: 10,000 IOPS → 0.1 ms/op. Same 10,000 random blocks ≈ 1 s.
  • Example: Sequential scan time
    • Scan 100 GB on HDD at 150 MB/s: ~667 s (~11 min).
    • Scan 100 GB on NVMe SSD at 2 GB/s: ~50 s.

3) RAID: Redundant Arrays of Independent Disks

Goals

  • Capacity: aggregate many disks into one logical volume.
  • Performance: parallelize reads/writes across disks (striping).
  • Reliability: mask disk failures with redundancy (mirrors/parity).

Common RAID levels

  • RAID 0 (Striping): data striped across disks; high performance; no redundancy—any disk failure loses the array.
  • RAID 1 (Mirroring): duplicate each block on two disks; excellent reliability and fast reads; 2× capacity cost.
  • RAID 5 (Distributed parity): stripe data and store parity blocks rotated across disks; tolerates one disk failure; write penalty for small random writes (read-old-data + read-old-parity + compute + write-data + write-parity).

Design considerations

  • Monetary cost: usable capacity vs raw capacity.
  • Performance: read/write bandwidth, random IOPS, small-write behavior.
  • During failure: degraded mode performance can drop (e.g., parity reconstruction on reads).
  • Rebuild time: long rebuilds increase risk of another failure (RAID 5).

Illustrative examples

  • Example: Usable capacity
    • 4 disks × 2 TB each:
    • RAID 0: 8 TB usable; zero redundancy.
    • RAID 1 (two mirrors): 4 TB usable; can survive one disk failure per mirror set.
    • RAID 5: 6 TB usable (N−1 disks); tolerates one disk failure.
  • Example: Small random write in RAID 5
    • To update one 4 KB block: read old 4 KB data + read old parity block; compute new - parity; write 4 KB data + write parity. That’s four I/Os (read-modify-write penalty).
  • Example: When RAID 0 is acceptable
    • Temporary scratch space for recomputable data (e.g., ephemeral staging). High throughput with no safety. Not acceptable for primary data.

4) File Organization and Record Representation

Files, records, and blocks

  • Database data is stored in files; each file is a sequence of fixed-size blocks (pages).
  • Each block holds one or more records; records consist of fields (attributes).

Fixed-length records

  • All records have the same size; packing into blocks is straightforward.
  • Deletion management:
    • Move-last-into-hole: copy the last record into the deleted slot.
    • Free list: maintain a linked list of free slots; new insertions reuse them, avoiding intra-block reshuffling.

Variable-length records and slotted pages

  • Variable-length fields (e.g., varchar) lead to records with different sizes.
  • Slotted page structure:
    • Block header stores a directory: for each slot, the record’s offset and length (or an indicator if free).
    • Records are packed from the end of the page upward; free space accumulates in the middle.
    • External references should point to the slot entry (not the physical byte offset) so the DB can move records within the block to defragment space without breaking pointers.

Illustrative examples

  • Example: Fixed-length free list
    • Deleting record at slot 17: mark slot 17 as free and link it to the head of the free list stored in the block header. Next insertion reuses slot 17 immediately.
  • Example: Slotted page compaction
    • Two variable-length records shrink/grow; the system moves them to keep free space contiguous. Because external pointers reference slot directory entries, no external references break.
  • Example: External pointer target
    • Target the directory slot (logical handle), not the in-page byte offset (physical). This indirection enables safe in-page record movement.

5) Organization of Records in Files (Heap, Sequential, Tree, Hash)

Organization options

  • Heap file: place new records wherever space is available (fast inserts).
  • Sequential file: maintain records sorted on a search key (fast ordered scans, slower inserts/deletes due to reorganization).
  • B+-tree file organization: maintain records in key order with balanced tree structure to support efficient point/range queries and graceful inserts/deletes (covered deeply in indexing).
  • Hashing: apply a hash function on a search key to choose a block/bucket; excellent for equality lookups; not ideal for range queries (covered in indexing).

Trade-offs

  • Heap: best for write-heavy, no-order-needed workloads; scanning may be slower; needs indexes for efficient queries.
  • Sequential: great for range scans and order-by without extra sort; expensive maintenance on updates.
  • B+-tree: balanced performance for inserts/lookups/range queries; extra space and maintenance overhead.
  • Hash: O(1) expected equality lookup; range queries poor; sensitive to hash distribution and load factor.

Illustrative examples

  • Example: Log ingestion table
    • High insert rate; queries often filter by timestamp ranges. A heap file plus a B+-tree index on time balances fast inserts with efficient range reads.
  • Example: Reference table with equality lookups
    • Lookups by ID only; range scans rare. Hash organization or a hash index provides fast equality access.
  • Example: Pre-sorted data consumption
    • Frequent order-by on a key: a sequential file or clustered B+-tree on that key can remove the need to sort at query time.

6) Buffer Management

Buffer pool and manager

  • Buffer pool: region of RAM to cache disk pages and reduce I/O.
  • Buffer manager: serves page requests, loads pages on misses, and decides which page to evict when space is needed.

Operations

  • Pin/unpin: a pinned page cannot be evicted; clients must unpin after use.
  • Dirty pages: pages modified in memory; must be written back before eviction.
  • Write-back: either overwrite the original block or write to a new location and mark the original invalid (common with SSD log-structured designs).

Replacement policies

  • LRU (Least Recently Used): evict the page not used for the longest time (good default heuristic).
  • Clock: approximate LRU with lower overhead.
  • Workload-aware choices:
    • Toss-immediate: pages known to be read once (e.g., sequential scan) can be evicted right after use.
    • MRU (Most Recently Used): can perform better than LRU for certain nested-loop patterns where older pages are more likely to be reused soon.

Illustrative examples

  • Example: Sequential scan of a large table
    • Each page is read once. Toss-immediate prevents the scan from evicting useful pages for concurrent OLTP queries.
  • Example: Nested loop join
    • Inner table pages are repeatedly probed; MRU can outperform LRU if the “most recently used” page has the lowest reuse probability in the immediate future (depending on join access pattern).
  • Example: Dirty page eviction
    • A modified page selected for eviction must be flushed to disk first; group commits and background flushers reduce stalls.

7) Magnetic Disks and Flash Memory (SSDs)

Magnetic disks (HDDs)

  • Rotating platters and mechanical arms with read/write heads.
  • Two dominant costs for random access:
    • Seek time: moving the head to the correct track.
    • Rotational latency: waiting for the sector to rotate under the head.
  • Excellent sequential throughput; low random IOPS.

SSDs (flash memory)

  • No moving parts; very low random access latency; high IOPS.
  • Writes require erasing larger “erase blocks”; pages cannot be overwritten in place.
  • Wear leveling distributes erases across physical blocks to prolong device life.

Illustrative examples

  • Example: Random read workload
    • HDD at 100 IOPS vs SSD at 50,000 IOPS: SSD completes 10,000 random reads in ~0.2 s vs ~100 s on HDD.
  • Example: Log-structured writes on SSDs
    • DBMS issues append-only writes to avoid in-place overwrites, aligning with erase-block constraints and improving endurance/performance.
  • Example: Mixed workload placement
    • Place random-read-heavy indexes on SSD; bulk data scans can remain on HDD to save cost.

8) External Sort-Merge Algorithm

When and why we need external sorting

  • Relations often exceed main memory capacity. External sort-merge sorts data using limited memory by organizing I/O into large sequential transfers.
  • Sorted data enables efficient algorithms like merge join and supports ORDER BY/GROUP BY without large in-memory states.

Two phases

  • Phase 1: Run generation
    • Read M blocks into memory, sort, write out a sorted run. Repeat until the entire relation is processed (producing approximately ceil(br / M) runs for br data blocks and M buffer blocks).
  • Phase 2: Multi-way merge
    • If number of runs N < M: single N-way merge using N input buffers + 1 output buffer.
    • If N ≥ M: multiple passes; each pass merges groups of up to M−1 runs into longer runs; repeat until one final run remains.

Cost analysis

  • Total merge passes: ceil(log_{M−1}(br / M)).
  • Each pass (including run generation) reads and writes all br blocks; conventionally, we count the final read but omit the final write.
  • Total I/O cost: br × (2 × ceil(log_{M−1}(br / M)) + 1).

Illustrative examples

  • Example: 3-way merge with M=4 buffers
    • Use 3 input buffers + 1 output buffer; at each step, pick the smallest front element and append to output; refill buffers when empty.
  • Example: Given br = 13 blocks, M = 4
    • Run generation: ceil(13/4) = 4 runs.
    • Pass 1: 3-way merge of first three runs → one longer run; the 4th run remains.
    • Pass 2: merge the two remaining runs → final sorted run.
    • Passes = ceil(log_{3}(13/4)) = ceil(log3(3.25)) = 2. Total cost = 13 × (2×2 + 1) = 13 × 5 = 65 I/Os.
  • Example: Buffer allocation
    • With M buffers and N runs where N ≥ M, each pass merges up to M−1 runs. Choose M to maximize merge degree and minimize number of passes.
