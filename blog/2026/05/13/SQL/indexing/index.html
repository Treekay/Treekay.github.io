Brief Summary: Database Indexing and B+-Trees

1) Basic Concepts

  • Purpose of Indexing: Speed up data access by minimizing disk I/O versus full scans.
  • Search Key: Attribute(s) used to locate records; supports equality and range queries.
  • Evaluation Metrics: Access time, insertion time, deletion time, and space overhead.

2) Ordered Indices

  • Clustering (Primary) Index: Physical file order matches the index’s search key.
  • Secondary (Non-clustering) Index: Logical order differs from physical file order.
  • Dense vs. Sparse: Dense has an entry per search-key value; sparse has entries for some values and requires the data file to be ordered on the key.
  • Multilevel Indices: Add outer levels when an index is too large for memory to reduce I/O.

3) B+-Tree Indexes

  • Characteristics: Self-balancing, all leaves at the same depth, high fanout m; non-root nodes are at least half-full.
  • Node Types: Inner nodes guide search (keys + child pointers); leaf nodes hold key–value pairs and are linked for fast range scans.
  • Leaf Values: Either record IDs (most common) or tuple data (index-organized storage); secondary indexes often store primary keys as values.
  • B-Tree vs. B+-Tree: B-Tree stores values in all nodes; B+-Tree stores values only in leaves.
  • Operations: Lookups are O(log_m n) I/Os; inserts split nodes and copy/push up middle keys; deletes borrow from siblings or merge and pull keys down.
  • Practical Notes: With fanout ≈ 100 and 10^6 keys, depth ≈ 3–4; inner nodes often stay in buffer; average occupancy ~2/3 (random inserts).

Topic 1. Organization of Records and Basic Indexing Concepts

Why we use indexes

Indexes are auxiliary data structures that allow the database to find the data you want quickly (minimizing disk I/O). Instead of scanning an entire file to find a record (e.g., “student with ID = 22201”), an index maps a search key (one or more attributes) to a pointer that leads directly to the data block containing the target record.

Record organization in files

  • Heap file: Records are placed anywhere; no particular order. Fast for inserts; slow for searches without an index.
  • Sequential file: Records are stored in sorted order on a chosen search key. Efficient for range scans; expensive to maintain under frequent updates.
  • B+-tree file organization: Maintains records in an order that supports inserts and deletes efficiently while staying balanced.
  • Hashing: A hash function maps a search key to a bucket. Very fast for equality search; does not support range queries well.

Search keys and access types

  • Search key: An attribute (or set of attributes) used to look up records.
  • Access types:
    • Equality search: Find records where key = value.
    • Range search: Find records where key is within [low, high] (or <, >, ≤, ≥).

Index types and key ideas

  • Ordered indices: Keys are stored in sorted order (enable equality and range queries).
  • Hash indices: Keys are distributed across buckets by a hash function (excellent for equality, not for ranges).

How we evaluate an index

  • Access time: I/Os to answer a query.
  • Insertion time: I/Os to add a record and update the index.
  • Deletion time: I/Os to remove a record and update the index.
  • Space overhead: Bytes/pages needed in addition to the data itself.

Examples

  • Example 1 (Equality search): A Students(ID, Name, Major, Year) table has 1M rows stored as a heap. Without an index, finding ID=22201 requires scanning many pages. With a B+-tree on ID (fanout around 100), only a few page reads (often 3–4) are needed to reach the target leaf and then the record.
  • Example 2 (Range search): A Salaries(EmployeeID, Amount) table with an ordered index on Amount can answer queries like 60,000 ≤ Amount ≤ 80,000 by seeking to the first qualifying key in the leaves and then following leaf links to read subsequent qualifying entries efficiently.
  • Example 3 (Choosing search keys): If the most common query is “find by email,” make Email the search key for the index. For frequent range queries on timestamps, index on Timestamp with an ordered structure like a B+-tree.

Topic 2. Ordered Indices: Primary/Clustering, Secondary, Dense, Sparse, and Multilevel

Primary (Clustering) vs. Secondary (Non-clustering) indices

  • Clustering (Primary) index: The physical order of the data file matches the index’s search key order. Great for range scans; updates can be costlier if they disturb order.
  • Secondary (Non-clustering) index: The search key order differs from the physical order of the data file. Flexible, supports many lookup patterns; range scans are less sequential on disk unless combined with other techniques.

Dense vs. Sparse indices

  • Dense index: Has an index entry for every search-key value. Fast lookups; higher space overhead.
  • Sparse index: Has entries for only some search-key values (typically one per data block or group). Less space; requires scanning forward in the data from the located block. Requires that the data file be physically sorted on the search key (i.e., clustering).

Multilevel indices

When an index becomes too large for memory and on-disk binary search would require too many I/Os, we can build a small, higher-level index on top of the larger (lower-level) index—recursively, forming a tree of indices. This is the idea that leads to B+-trees.

Examples

  • Example 1 (Clustering index on StudentID): If Students are physically stored sorted by StudentID, then an index on StudentID is clustering. Range queries on ID (e.g., 10000–19999) are efficient because the data blocks are adjacent on disk.
  • Example 2 (Secondary index on Department): If Students are stored by StudentID, an index on Department is secondary. A query Department=‘CS’ uses the secondary index to find matching pointers but resulting records are scattered in the heap.
  • Example 3 (Dense vs. Sparse): For an Instructor(ID, Name, Salary) clustering index on ID, a sparse index may store one entry per data page keyed by the first ID in that page. To find ID=22201, locate the largest index key ≤ 22201, jump to that page, then scan forward within the page(s). A dense index on ID has a one-to-one mapping from every ID to its exact location.
  • Example 4 (Multilevel): Suppose a dense index over 1M keys occupies thousands of pages. A higher-level sparse index over the lower index’s first keys might fit in a handful of pages, offering a quick two- or three-level search without scanning the entire index.

Topic 3. B+-Tree Fundamentals

Definitions and properties

  • B+-tree: A self-balancing, ordered m-way tree optimized for disk I/O.
  • Perfectly balanced: All leaves are at the same depth.
  • Occupancy: Every non-root node is at least half-full. If a node can have up to m pointers, it will have between ⌈m/2⌉ and m pointers (and thus between ⌈m/2⌉−1 and m−1 keys). The root is a special case and may be less occupied.
  • Performance: Lookup, insertion, and deletion each cost O(log_m n) I/Os, where m is fanout and n is keys.

Node types and layouts

  • Inner (non-leaf) nodes: Sorted array of keys that guide navigation, interleaved with child pointers. They contain keys only.
  • Leaf nodes: Sorted key/value pairs and sibling pointers linking leaves for fast ordered scans. “Values” are either:
    • Record IDs (RIDs) pointing to the heap tuple location (common). This is typical for secondary indexes.
    • Tuple data itself (index-organized storage). This is typical for a primary index that stores the actual row in leaves.
  • Null handling: If present, null keys are gathered in a designated first/last leaf to maintain ordering semantics.

B-tree vs. B+-tree

  • B-tree (original): Stores keys and values in both inner and leaf nodes (each key appears once); more space-efficient for keys.
  • B+-tree (common in DBMSs): Stores values only in leaves; inner nodes hold keys to guide search. This design simplifies range scans and bulk I/O on leaves.

Why B+-trees excel on disk

  • High fanout (often ~100) keeps tree height small (e.g., ≈ 3–4 levels for millions of keys).
  • Inner nodes are frequently cached in memory; most operations hit only one leaf I/O.
  • Sequential access across leaves is efficient for range queries due to sibling pointers.

Examples

  • Example 1 (Lookup): To find key 31, start at the root, choose the appropriate child by comparing 31 against node keys, repeat at each level until reaching the leaf that contains 31 (or where it would be). If found, follow the leaf’s value (RID or tuple) pointer.
  • Example 2 (Range scan 10–44): Navigate to key ≥ 10 in the first leaf. Then follow the leaf sibling pointers, emitting all keys up to 44. Only the first seek traverses the inner nodes; subsequent pages read sequentially from leaves.
  • Example 3 (Primary vs. secondary leaves): In a primary B+-tree over the table’s primary key, a leaf may store the full tuple (index-organized). In a secondary B+-tree on a non-key attribute, a leaf stores the (key, primary-key) pair or (key, RID), requiring one extra hop to fetch the full tuple from the heap.

Topic 4. B+-Tree Operations: Insertion, Deletion, and Complexity

Insertion

  • Find the correct leaf L and insert the new (key, value) in sorted order.
  • If L overflows (exceeds capacity), split it into L and L2 and redistribute keys evenly.
  • Copy up the split key into the parent (for a leaf split); insert a new pointer to L2 in the parent.
  • If the parent overflows, split it too, and push up the middle key to its parent; repeat up to the root. If the root splits, the tree grows in height by one.

Deletion

  • Find the target leaf and remove the (key, value).
  • If the leaf falls below minimum occupancy (less than half-full), first try to borrow an entry from a sibling (with the same parent).
  • If borrowing is not possible, merge the leaf with a sibling (coalesce) and remove the separating key from the parent. If the parent underflows, apply the same borrow-or-merge logic upwards. If the root ends up with a single child, the tree shrinks in height.

Complexity and occupancy facts

  • Worst-case I/Os for lookup/insert/delete: O(log_m n).
  • In practice, inner nodes are often cached; most operations pay for one leaf I/O.
  • Average node occupancy: roughly 2/3 with random insertions; can be ~1/2 if keys are inserted in sorted order (which triggers more splits at half-full states).

Examples

  • Example 1 (Leaf split): Insert key 6 into a leaf already full with [5,9,10]. After insertion and split, redistribute, keep leaf nodes balanced (e.g., [5,6] and [9,10]), then copy the split key (e.g., 9) up to the parent to point to the new right leaf.
  • Example 2 (Root split): Repeated inserts can fill the root. When it splits, a new root is created with two child pointers, increasing the height by one.
  • Example 3 (Borrow): Deleting a key makes a leaf underfull. If a sibling has extra entries, borrow one and update the separator key in the parent accordingly.
  • Example 4 (Merge and shrink): If siblings are both near empty, merge them and pull down the separator key from the parent. This can cascade upward; if the root ends with one child, reduce the height.
  • Example 5 (I/O estimate): For n=1,000,000 and fanout m≈100, the height is at most about log base ~50 of 1,000,000, so a lookup touches at most ~4 nodes. With inner nodes cached, often only the leaf I/O is needed.
