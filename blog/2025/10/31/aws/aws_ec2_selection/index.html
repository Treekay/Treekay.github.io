AWS EC2 Selection

Identify Workload Type

  • Long-running stable → predictable workload
  • Temporary tasks / testing → short-term / flexible
  • Compute-intensive → CPU/GPU heavy
  • Infrequently used → low utilization

Choose the Right Pricing Model

Model Use Case Cost Characteristics Example Phrase
On-Demand Temporary testing, development Highest cost, most flexible “Use On-Demand for flexibility.”
Reserved Instances / Savings Plans Long-term stable workloads Lower cost (up to 72% savings) “Use Reserved Instances for predictable workloads.”
Spot Instances Fault-tolerant, batch processing Lowest cost, but interruptible “Use Spot Instances for cost saving in flexible jobs.”
Dedicated / Host Instances Compliance, high security requirements High cost “Use Dedicated for compliance isolation.”

Choose the Right Instance Type

Type Characteristics Use Cases
General Purpose (t, m) Balanced CPU and memory Web servers, application servers
Compute Optimized (c) High CPU performance Batch computing, video encoding
Memory Optimized (r, x) Large memory Databases, caching
Storage Optimized (i, d) High I/O performance Data warehouse, Hadoop
GPU (g) Graphics rendering / AI training Graphic design, ML training

Additional Optimization Strategies

  • Auto Scaling → dynamic scaling
  • Elastic Load Balancing → traffic distribution
  • EBS-backed instances → retain data after stop
  • Custom AMI → consistent configuration
  • AWS Systems Manager / CloudWatch → automated patching and monitoring

Example Template

Company X should match each workload with the appropriate EC2 pricing model.
Use Reserved Instances for long-running tasks, Spot Instances for flexible workloads, and On-Demand for short-term development.
Enable Auto Scaling to handle demand changes.

Use GPU-based instances because designers require high-performance graphics for rendering images and videos.

Use EBS-backed EC2 instances, which allow users to stop and start instances without losing data. Designers can pause work and resume later from the same state.

Create a custom Amazon Machine Image (AMI) from the configured instance. This ensures every new instance launched has the same software and settings for consistent collaboration.

Key factors: cost-effective, compliant, flexible, high-performance, reliable, secure, and long-term data protection.

EC2 Related Core Services Summary

Function Service/Option Description
Pricing Model On-Demand / Reserved / Spot / Dedicated Trade-off between cost and flexibility
Data Persistence EBS-backed vs Instance Store EBS persists data, Instance Store is temporary
Configuration Management AMI (Amazon Machine Image) Stores system settings and environment
Auto Scaling Auto Scaling Group (ASG) Scale instances based on demand
Load Balancing Elastic Load Balancer (ELB) Distributes traffic to prevent overload
Maintenance & Monitoring Systems Manager / CloudWatch Automated patching and performance monitoring

Common Question Types Summary

Question Type Keywords Common Solution
Cost Optimization “cost-effective / optimize / reduce cost” Reserved + Spot + Auto Scaling
High Performance Computing “graphics / video / simulation” GPU instances (G5/G4dn)
Elastic Scaling “traffic spikes / unpredictable workload” Auto Scaling + On-Demand
Periodic Tasks “periodic / batch job” Spot + Lambda
Configuration Consistency “same configuration / resume work” EBS-backed + AMI
Disaster Recovery / High Availability “redundancy / backup” Multi-AZ + Load Balancer
