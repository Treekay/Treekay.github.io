AWS IAM
2025-10-31 00:03   label AWS

IAM Policy

Question Type Common Keywords Key Concepts
Access Control Design “restrict access / manage permissions / different users” IAM Users / Groups / Roles / Policies
Security Improvement “public/private key login / shared credentials / security concern” Use IAM Roles, Groups, and apply least privilege principle
Operations Management “too many users / need central management” IAM Groups, Policies, Organizations
Application Access to AWS Resources “EC2 accessing S3 / Lambda accessing DynamoDB” Attach IAM Roles to services
Multi-Account & Compliance “multiple departments / compliance” AWS Organizations + Service Control Policies

Example

Use IAM Roles and Groups instead of individual IAM users or key-based access.
Assign permissions through IAM Policies following the principle of least privilege.

Create Groups based on job functions (e.g., Admin, Developer, Auditor).
Attach policies to define what each group can access.
Attach IAM Roles to AWS resources (such as EC2, Lambda) so they can access only required services (e.g., S3, RDS).

Benefits

  • Centralized permission management
  • Easier to update or revoke access
  • Reduces risk from hardcoded or shared credentials
  • Enforces least privilege and improves compliance

Cons

  • Requires careful policy planning
  • Initial setup can be time-consuming
  • May require training for administrators

Key Concepts Comparison Table

Item Definition Typical Use Notes
IAM User Individual identity Daily operations, console login Avoid creating too many users
IAM Group Collection of users Manage permissions for similar users Cannot nest groups
IAM Role Temporary identity with permissions Access by AWS services (EC2, Lambda) No long-term credentials, more secure
Policy JSON-based permission rules Define allowed actions and resources Follow least privilege
Root User Account owner Account-level operations only Should not be used daily
Access Key Programmatic credentials CLI/API access Rotate regularly or replace with Roles
