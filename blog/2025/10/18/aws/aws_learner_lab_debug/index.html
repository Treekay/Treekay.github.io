2025-10-18 10:19   label AWS, UoA

AWS Learner Lab Debugging Record

Problem Background

The server instances in the frontend target group connected to the ALB were consistently marked as unhealthy: request timed out (504), and the web content could not be accessed.

Problem Analysis

A 504 error indicates that the request reached the server, but the server did not respond in time. This suggests that the connection from the ALB to the private subnet is functioning correctly.

Possible causes of failed health checks include mismatched check paths or ports between the ALB and the instances. After verification, these were not the issue.

The most likely cause was that the server service was not running, meaning the instance was not properly listening for and handling requests.

Initial Actions

Added userdata to the Auto Scaling Group launch template associated with the ALB target group to install and start Nginx during instance initialization.

However, health checks still failed.

To further investigate, it was necessary to connect to the instance to verify whether Nginx was running. Since the instance was in a private subnet, SSH access required a bastion host in a public subnet.

NAT Instance and Bastion Host Setup

To reduce costs, a single EC2 instance was configured to act as both a NAT instance and a bastion host.

  • Created an Amazon Linux 2023 EC2 instance
  • Assigned NAT security group
  • Placed in a public subnet
  • Enabled key pair (vockey) for SSH access

NAT Instance Network Interface Configuration

The source/destination check must be disabled, otherwise AWS will drop packets not originating from the NAT instance itself (including outbound traffic from private EC2 instances).

Since AWS Learner Lab resets EC2 instances after timeout or restart, this configuration must be reapplied after each restart.

NAT Instance Configuration

  • NAT Security Group

    • Inbound: SSH | My IP
    • Inbound: All traffic | VPC CIDR

  • Frontend Security Group

    • Outbound: All traffic | 0.0.0.0/0

  • Private subnet route table → NAT instance

Note: After lab restart, NAT instance settings may need to be reconfigured due to network interface changes.

Further Analysis

After resolving networking issues, nginx installation still failed.

Suspected cause: In the Learner Lab environment, access to the official S3 repository might be restricted.

Observation:

  • NAT instance (in public subnet) can install nginx successfully
  • Private instance cannot, likely due to restricted outbound access

Alternative Solutions

  1. Build a repo proxy on NAT instance
  2. Use Amazon Linux 2 AMI (potential workaround, but more costly)

Latest Findings

Since the VPC includes endpoints, NAT proxy should not be required to access S3 repositories.

Actual root cause:

  • Userdata executed too early before network initialization completed
  • This caused installation failures during startup

Additional issue:

  • Cross-AZ NAT usage caused connectivity problems
  • Solution: deploy separate NAT instances per AZ with corresponding route tables
