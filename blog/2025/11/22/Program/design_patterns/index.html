Design Patterns
2025-11-22 16:51   label Program

Composite Pattern

Intent:
Treat individual objects and compositions of objects uniformly.

Structure:

  • Component (interface/abstract class)
  • Leaf (individual objects)
  • Composite (contains children)

Use Case:
Tree structures (e.g., file systems, UI components)

Pros:

  • Simplifies client code
  • Easy to add new components

Cons:

  • Harder to restrict component types

Command Pattern

Intent:
Encapsulate a request as an object, allowing parameterization and queuing.

Structure:

  • Command (interface)
  • ConcreteCommand
  • Invoker
  • Receiver

Use Case:
Undo/redo, task queues, remote controls

Pros:

  • Decouples sender and receiver
  • Supports undo/redo

Cons:

  • Increases number of classes

Observer Pattern

Intent:
Define a one-to-many dependency so that when one object changes state, all dependents are notified.

Structure:

  • Subject
  • Observer
  • ConcreteSubject
  • ConcreteObserver

Use Case:
Event systems, UI updates, pub-sub systems

Pros:

  • Loose coupling
  • Easy to add observers

Cons:

  • Potential performance issues with many observers

Adapter Pattern

Intent:
Convert one interface into another expected by the client.

Structure:

  • Target
  • Adapter
  • Adaptee

Use Case:
Integrating incompatible interfaces

Pros:

  • Reuse existing code
  • Improves compatibility

Cons:

  • Adds extra layer of abstraction

Strategy Pattern

Intent:
Define a family of algorithms and make them interchangeable.

Structure:

  • Strategy (interface)
  • ConcreteStrategy
  • Context

Use Case:
Different algorithms (e.g., sorting, payment methods)

Pros:

  • Easy to switch algorithms
  • Eliminates conditional logic

Cons:

  • Increases number of classes

Decorator Pattern

Intent:
Add behavior to objects dynamically without modifying their structure.

Structure:

  • Component
  • ConcreteComponent
  • Decorator
  • ConcreteDecorator

Use Case:
Extending functionality (e.g., adding features to UI elements)

Pros:

  • Flexible alternative to inheritance
  • Supports combination of behaviors

Cons:

  • Can lead to many small objects

Factory Method

Intent:
Define an interface for creating objects, but let subclasses decide which class to instantiate.

Structure:

  • Creator
  • ConcreteCreator
  • Product
  • ConcreteProduct

Use Case:
Object creation logic delegation

Pros:

  • Promotes loose coupling
  • Follows Open/Closed Principle

Cons:

  • Requires more subclasses

Template Method

Intent:
Define the skeleton of an algorithm, letting subclasses override specific steps.

Structure:

  • AbstractClass
  • Template Method
  • ConcreteClass

Use Case:
Processes with fixed steps but variable details

Pros:

  • Code reuse
  • Enforces algorithm structure

Cons:

  • Less flexibility due to inheritance
