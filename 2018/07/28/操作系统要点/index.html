操作系统要点

发表于 2018-07-28 | 分类于 操作系统

第1章 导论

操作系统概念

  • 作为硬件和用户之间的中介程序，是管理计算机硬件的软件
  • 从用户视角：使用方便、性能、资源使用率等考量点决定了操作系统的目标
  • 从系统视角：作为资源分配器，管理资源，解决请求冲突，解决CPU时间、内存空间、文件存储空间、I/O设备等问题
  • 定义操作系统：一直运行在计算机上的程序（内核），其他程序为系统程序和应用程序

计算机系统组织

  • I/O和CPU可以并发执行
  • CPU、设备控制器由总线相连，对共享内存进行访问，协调内存竞争问题，需要内存控制器
  • CPU通过局部缓存与主存交换数据

  • 控制器和设备之间的接口

  • 引导程序(bootstrap program)初始化系统中的所有部分，包括CPU寄存器、设备控制器和内存，并将操作系统装入内存执行

  • 事件发生通过中断表示，硬件可随时通过系统总线向CPU 发出信号触发中断，软件通过系统调用(监视器调用monitor call)触发中断
  • CPU中断时停止当前工作并转到中断服务程序(interrupt service routine)的开始位置
  • 内存容量小，易失性，断电丢失，因而需要辅存(secondary storage)
  • 存储设备层次
    • 寄存器
    • 高速缓存cache
    • 主存
    • 电子磁盘
    • 磁盘
    • 光盘optical disk
    • 磁带
  • I/O的执行，设备驱动程序(device driver)在设备控制器中装载适当的寄存器，设备控制器检查寄存器内容决定执行什么操作。

计算机系统体系结构

  • 单处理器系统，用专用处理器接受主CPU的请求执行自己的磁盘队列和调度算法，以此克服主CPU的磁盘调度超载问题
  • 多处理器系统的优点
    • 增加吞吐量
    • 规模经济
    • 增加可靠性
  • 适度退化(graceful degradation) : 提供与正常工作的硬件成正比的服务的能力
  • 容错（fault tolerant)：超出适度退化的能力
  • 非对称多处理和对称多处理(SMP)
    • 非对称：主处理器控制系统，其他处理器从主处理器获得并做各自特定的任务
    • 对称：所有处理器都要完成操作系统中的所有任务
  • 集群系统(Clustered Systems)将多个CPU集中起来完成计算任务，和多处理器系统的不同之处在于，它是由两个或多个独立的系统耦合起来的。共享存储，通过LAN(局域网络)连接或内部连接
    • 高可用性、热备份模式(hot standby mode)
  • 并行集群：用专门软件来是实现多主机访问共享存储数据，完全访问，需要加锁，分布式锁管理器(DLM)

操作系统结构

  • 多道程序设计(Multiprogramming)：组织作业，同时将多个任务保存在内存
  • 分时系统(Time sharing)：在作业之间切换来执行多个作业
  • 进程：装入到内存并执行的程序
  • 作业池(job pool):等待分配内存的作业
  • 作业调度(job scheduling):多个作业需要调入内存时，要做出的选择
  • CPU调度(CPU scheduling):多个任务需要执行时做出的选择

操作系统操作

操作系统是由中断驱动的。没有进程、I/O服务、用户请求要响应时，操作系统会等待事件发生。

  • 事件是由中断(trap) 或陷阱引起的
  • 陷阱(或异常exception)是一种软件中断，由于出错或用户程序的特别请求引起

双重模式操作

CPU有两种模式：实模式和保护模式

  • 实模式下，用户程序不受限制
  • 保护模式下，用户程序被限制，此时操作系统有下面特点：
    • 双重模式操作
      • 用户模式
      • 内核模式(管理模式、系统模式、特权模式)
    • 模式位(mode bit)为0为内核模式，1为用户模式
    • 一旦出现陷阱或中断，硬件会从用户模式切换到内核模式
    • 提供了I/O保护、内存保护、CPU保护
  • 操作系统得到对计算机的控制时总是内核模式，能引起损害的机器指令作为特权指令，若在用户模式下试图执行特权指令，则视作非法指令不执行，以陷阱的方式通知操作系统。
  • 所有的I/O指令都是特权指令
  • 定时器:定时产生中断，把控制权交给操作系统

        • 通信进程通过彼此之间交换消息来交换信息。直接或间接地通过一个共同的邮箱，消息可以在进程之间得到交换。

        • 共享内存要求两个甚至多个进程都统一取消阻止其他进程访问另一进程的限制，这样它们就可以通过读写公共区域来交换信息
      • 优缺点：消息传递对交换少量数据很有用，这是因为不必避免冲突，对于计算机间的通信，它也比共享内存更容易实现。共享内存允许最大速度地通过通信并且十分方便。这是因为当通信发生在计算机内，它能以内存的速度进行。不过，在保护和同步方面，进程共享内存存在一些问题。

第3章 进程

进程概念

进程

  • 批处理系统中称为作业
  • 分时系统中称为用户程序或任务

  • 进程是执行中的程序(文本段)、当前活动、进程堆栈端、数据段、堆

  • 栈段：编译时编译器计算并设定好，运行时由系统自动分配和释放

  • 堆区：通常由程序员分配释放。若程序员不释放，程序结束时可能由OS回收。与数据结构中的堆是两回事，可看成内存块的(链)表
  • 数据段(含区安居区、静态区和常量区)：全局区或静态区——存放全局变量和静态变量，通常是放在一块的，程序结束后由OS回收。通常初始化的全局变量和静态变量在一块区域，未初始化的全局变量和未初始化的静态变量在相邻的另一块区域。常量区是另一块，存放常量字符串。也由OS回收 。
  • 程序代码段(文本段)：存放程序的二进制代码

进程状态

  • new：进程正在被创建
  • running：进程正在被执行
  • waiting：进程等待某个事件的发生
  • ready：进程等待分配处理器
  • terminated：进程完成执行

进程控制块

  • 进程控制块(PCB)

  • 进程状态：状态可包括新的、就绪、运行、等待、停止等

  • 程序计数器：计数器表示进程要执行的下个指令的地址
  • CPU寄存器：根据计算器体系结构的不同，寄存器的数量和类型也不同。它们包括累计器、索引寄存器、堆栈指针、通用寄存器和其他条件码信息寄存器。与程序计数器一起，这些状态信息在出现中断时也需要保存，以便进程以后能正确地继续执行
  • CPU调度信息：这类信息包括进程优先级、调度队列的指针和其他调度参数
  • 内存管理信息：根据操作系统所使用的内存系统，这类信息包括基址和界限寄存器的值，页表或段表。
  • 记账信息：这类信息包括CPU时间、实际使用时间、时间界限、记账数据、作业或进程数量等
  • I/O状态信息：这类信息包括分配给进程的I/O设备列表、打开的文件列表等

进程调度

调度队列

  • 作业队列(Job queue)：进程进入系统时，会被加到作业队列，该队列包括系统中所有进程
  • 就绪队列(ready queue)：驻留在内存中就绪的、等待运行的进程保存在就绪队列中。该队列通常用链表来实现，其头结点包含指向链表的第一个PCB块和最后一个PCB块的指针，每个PCB包括一个指向就绪队列的下一个PCB的指针域
  • 设备队列(Device queues)：等待特定I/O设备的进程列表称为设备队列，每个设备都有自己的设备队列

当进程分配到CPU并执行时，可能发生几种情况：

  • 进程可能发出一个I/O请求，并被放到I/O队列(设备队列)
  • 进程可能创建一个新的子进程，并等待其结束
  • 进程可能会由于中断而强制释放CPU，并被放回到就绪队列

进程完成后将从所有队列中删除，其PCB和资源将被释放

调度程序

  • 进程的选择由相应的调度程序(scheduler)来执行
  • 长期调度程序(long-term scheduler)或作业调度(job scheduler)：从大容量存储设备(如磁盘)的缓冲池中选择进程，并装入内存以准备执行。长期调度控制多道程序设计的程度(内存中的进程数量)
  • 短期调度程序(short-term scheduler)或CPU调度：从准备执行的进程中选择集成，并为之分配CPU

中期调度：进程可以从内存中被换出

进程操作

  • 进程创建：通过创建进程系统调用
  • 根据唯一的进程标识符pid来识别进程
  • 进程树，顶端是标识符为0的进程
  • 创建新进程后的两种执行可能：
    • 父进程与子进程并发执行；
    • 父进程等待，直到某个或全部子进程执行完
  • 新进程的地址空间也有两种可能：
    • 子进程是父进程的复制品(具有与父进程相同的程序和数据)，继承了父进程的地址空间
    • 子进程装入另一个新程序
  • 通过fork()系统调用创建子(新)进程
    • 对于子进程，fork()的返回值都是0
    • 对于父进程，fork()的返回值非0
  • 子进程通过系统调用exec取代进程的内存空间并开始执行，父进程若无任务则用wait()把自己移出就绪队列来等待子进程的终止
  • 当进程完成执行后用系统调用exit()请求操作系统删除自身，进程终止，并返回状态值给父进程(通过wait())，所有进程资源被释放
  • 级联终止：不允许子进程在父进程已终止的情况下存在

进程间通信

  • 进程独立(independent)，进程协作(cooperating)

  • 协助进程：与其他进程共享内存的进程

  • 信息共享、提高运算速度、模块化、方便

  • 进程间通信机制(IPC, inter process communication)

  • 进程间通信的两种基本模式：

    • 共享内存：建立一块供协作进程共享的内存区域，通过共享区域读或写入数据来交换信息
    • 消息传递：通过在协助进程间交换消息来实现通信

  • 消息传递更易于实现，适用于交换少量的数据，不需要避免冲突，内核介入消耗更多时间

  • 内存共享传递更快，但占用更多资源

消息传递系统

  • 进程命名

直接或间接通信

  • 直接通信(对称寻址)：
    • send(P, message) // 发送消息到进程P
    • receive(Q, message) // 接收来自进程Q的消息
    • 要求：一个通信线路只与两个进程有关
  • 直接通信(非对称寻址):
    • send(P, message) // 发送消息到进程P
    • receive(id, message) // 接收来自任何进程的消息，变量id设置成与其通信的进程名称
  • 间接通信(通过邮箱或端口来发送和接收消息):
    • send(A, message) // 发送消息到邮箱A
    • receive(A, message) // 接收来自邮箱A的消息
    • 要求：两个进程要有共享邮箱，每个线路对应一个邮箱
    • 特点：一个进程可以与多个进程想关联
    • 关于多个进程共享一个邮箱时的接受问题：取决于所选择解决方案
      • 一个线路最多只能两个进程相关联
      • 一次最多允许一个进程执行receive()操作
      • 允许系统随意选择一个进程接受消息

同步或异步通信

  • 同步(阻塞):
    • 阻塞send：发送者阻塞，直到消息被接收
    • 阻塞receive：接收者阻塞，直到有消息可用
  • 异步(非阻塞):
    • 非阻塞send：发送者发送消息并继续其他操作
    • 非阻塞receive：接收者接受一个消息，可能没有消息
  • 当发送者和接收者都是阻塞时，产生一个集合点

自动或显式缓冲

  • 零容量(无缓冲)：队列中不能有任何消息等待，必须使用阻塞发送
  • 有限容量(自动缓冲)：根据线路剩余容量决定是否阻塞发送者
  • 无限容量(自动缓冲)：从不阻塞发送者

RPC和RMI的不同：

第4章 线程

  • 线程是CPU使用的基本单元，它由线程ID、程序计数器、寄存器集合和栈组成。如果进程有多个控制线程，那么它能同时做多个任务
  • 多线程优点
    • 响应度高：即使部分阻塞或执行较冗长的操作，程序能继续执行，从而增加了对用户的响应程度
    • 资源共享：线程默认共享它们所属进程的内存和资源
    • 经济：进程创建所需的内存和资源的分配比较昂贵，线程的创建和切换更为经济，花费的时间也更少
    • 多处理器体系结构的利用：多线程的优点就是充分使用多处理器体系结构，以便进程能并行运行在不同的处理器上(单线程进程只能运行在一个CPU上)

多线程模型

  • 多对一模型
    • 一个内核线程管理多个用户线程
    • 任一时刻只有一个线程能访问内核
    • 多个线程不能并行执行在多处理器上
    • 一个线程阻塞系统调用时，整个进程会被阻塞
  • 一对一模型
    • 每个用户线程映射到一个内核线程
    • 一个线程阻塞系统调用时，其他线程能继续执行
    • 比多对一更好的并发功能
    • 允许多个线程并行地运行在多处理器上
    • 缺点：创建内核线程开销较大，影响性能，且受系统实际支持的线程数限制
  • 多对多模型
    • 可以创建任意多的用户线程
    • 可并发执行
    • 当一个线程执行阻塞系统调用时，内核能调度另一个内核线程来执行
    • 二级模型：允许某个用户线程绑定到内核线程上

线程库

  • 线程库：为程序员提供创建和管理线程的API
  • 两种方法实现：
    • 一：在用户空间中体用一个没有内核支持的库，所有代码和数据结构都存在于用户控件，调用库中的函数是本地函数调用而不是系统调用
    • 二：执行一个由操作系统直接支持的内核级的库，调用库中的一个API函数会导致内核的系统调用

多线程问题

  • fork()和exec()
    • 在多线程程序中，系统调用fork()和exec()的语义有变化
      • 如果一个线程调用fork(), 新进程有复制所有线程，和只有单个线程两种情况
      • 如果一个线程调用exec()，那么exec()参数所指定的程序会替换整个进程，包括所有线程
  • 线程取消(thread cancellation)
    • 要取消的线程称为目标线程
    • 目标线程的取消可在两种情况下发生：
      • 异步取消(asynchronous cancellation)：一个线程立即终止目标线程
      • 延迟取消(deferred cancellation)：目标线程不断地检查它是否应终止，这允许目标线程有机会以有序方式来终止自己
  • 信号处理
    • 信号用来通知进程某个特定事件已发生
      • 信号是由特定事件的发生所产生的
      • 产生的信号要发送到进程
      • 一旦发送，信号必须加以处理
    • 同步信号发送到执行操作而产生信号的同一进程
    • 异步信号被发送到另一个进程(信号由运行进程之外的产生，由运行进程异步接收这一信号)
    • 信号的处理程序
      • 默认信号处理程序
      • 用户定义的信号处理程序
    • 信号发送的对象
      • 信号所引用的线程
      • 进程内的每个线程
      • 进程内的某些固定线程
      • 规定一个特定线程以接收进程的所有信号
  • 线程池：无限制的线程会耗尽系统资源，如CPU时间和内存，线程池在进程开始时创建一定数量的线程，并放入到池中等待工作。
    • 优点
      • 通常用现有线程处理请求要比等待创建新的线程快
      • 线程池限制了在任何时候可用线程的数量，这对那些不能支持大量并发线程的系统非常重要

第5章 CPU调度

CPU调度：操作系统利用等待时间、空闲时间，将CPU使用权交给其他进程使用，减少等待时间的浪费

  • 进程执行由CPU执行和I/O等待周期组成

  • 进程选择由短期调度程序或CPU调度程序执行

    当CPU空闲时，操作系统就必须从就绪队列中选择一个进程来执行

  • CPU调度发生的四个时间：

    • 当一个进程从运行状态切换到等待状态(例如，I/O请求，或者调用wait等待一个子进程终止)
    • 当一个进程从运行状态切换到就绪状态(例如，出现中断)
    • 当一个进程从等待状态切换到就绪状态(例如,I/O完成)
    • 当一个进程终止时
    • 1、4两种情况只有调度，没有选择；2、3两种情况可以进行选择。
    • 当调度值只能发生在1、4时，调度方案是非抢占的(non-preemptive)或协作的
    • 否则调度方案是抢占的(preemptive)

  • 分派程序(dispatcher)：是一个模块，用来将CPU的控制交给由短期调度程序选择的进程，功能包括：

    • 切换上下文
    • 切换到用户模式
    • 跳转到用户程序的合适位置，以重新启动程序

  • 分派延迟(dispatch latency)：分派程序停止一个进程而启动另一个所要花费的时间

调度准则

  • 用于比较CPU调度算法的特征
    • CPU使用率
    • 吞吐量：一个时间单元内所完成进程的数量
    • 周转时间：从进程提交到进程完成的时间
    • 等待时间：进程在就绪队列等待所花费时间
    • 响应时间：从提交请求到产生第一响应的时间(公平原则)

调度算法

FCFS-先到先服务调度

(First-Come, First-Served)

  • 先请求先分配到CPU
  • 用FIFO队列实现，进程进入就绪队列，其PCB链接到队列的尾部，从队头的进程开始分配
  • 缺点
    • 平均等待时间较长
    • 非抢占式，I/O多的大进程会让CPU和设备使用率变得很低，对于分时系统尤其不适用

SJF-最短作业优先调度

(Shortest-Job-First Scheduling)

  • 更适当的表示：最短下一个CPU区间算法
    • 这是因为调度检查进程的下一个CPU区间的长度，而不是其总长度(因为一个进程有很多个CPU区间和I/O区间组成)
    • 相同时间按FCFS
  • 最佳的调度算法，平均等待时间最小
  • 缺点：
    • 难以知道下一个CPU区间的长度
    • 不能在短期CPU调度层次上加以实现
  • 近似SJF调度，根据(该进程)以前的CPU区间的长度来估计下一个CPU区间的预测值
  • 抢占式SJF算法：最短剩余时间优先调度
    • 当一个进程到达就绪队列而以前进程正在执行，就需要选择

优先级调度

(Priority Scheduling)

  • 每个进程都有一个优先级与其关联，最高优先级的进程会被分配到CPU，相同优先级的进程按FCFS顺序调度
    • SJF算法属于优先级调度的一个特例，简单优先级算法，优先级(p)为下一个(预测的)CPU区间的倒数
  • 以大数字或小数字为高优先级没有定论
  • 主要问题：
    • 无穷阻塞(indefinite blocking)或饥饿(starvation)
    • 解决办法：老化(aging)，随等待时间逐渐增加优先级

RR - 轮转法调度

(round-robin)

专门为分时系统设计，类似于FCFS调度，但是增加了抢占以切换进程，

  • 时间片：每间隔一定时间就切换进程，从就绪队列中拿出第一个进程(就绪队列作为循环队列，时间片结束的进程放回到队列尾部)
  • CPU区间短于时间片的进程，在终止后进程本身会自动释放CPU；长与时间片的进程则会被定时器中断，进行上下文切换，将进程加入到就绪队列的尾部
  • 缺点：平均等待时间通常较长
  • 性能依赖于时间片大小，若时间片很大，则无异于FCFS；若时间片很小，则RR算法成为处理器共享
  • 周转时间也依赖于时间片大小

多级队列调度

(Multilevel Queue Scheduling)

  • 将就绪队列分成多个独立队列，根据进程的属性，如内存大小、进程优先级、进程类型，一个进程被永久地分配到一个队列
  • 每个队列有自己的调度算法
  • 另外，队列之间要有调度，通常采用固定优先级抢占调度

多级反馈队列调度

(Multilevel Feedback-Queue Scheduling)

  • 允许进程在队列之间移动
  • 根据不同CPU区间的特点以区分进程
  • CPU时间长的放到优先级低的队列，I/O约束和交互进程放在搞优先级的队列
  • 多级反馈队列调度程序的参数
    • 每个队列的调度算法
    • 用以确定何时升级到更高优先级队列的方法
    • 用以确定何时降级到更低优先级队列的方法
    • 用以确定进程在需要服务时进入哪个队列的方法

多处理器调度

  • 负载分配(load sharing)成为可能，调度问题也更复杂
  • CPU调度算法同单处理器一样，没有最好的解决办法
  • 多处理器调度的方法
    • 对称多处理(SMP)
  • 处理器亲和性：努力使一个进程在同一处理器上运行
  • 负载平衡：设法将工作负载平均地分配到SMP系统中的所有处理器上
  • 对称多线程：超线程技术

线程调度

  • 局部调度：用户线程->LWP
  • 全局调度：内核线程->CPU

竞争条件(race condition)：多个进程并发访问和操作同一数据且执行结果与访问发生的特定顺序有关

临界区问题

  • 临界区：进程中修改共同变量或其他共享数据的代码段
  • 进入区：每个进程必须请求允许才能进入临界区，实现这一请求的代码段称为进入区
  • 退出区：临界区之后有退出区
  • 剩余区：剩余其他代码为剩余区
  • 抢占内核和非抢占内核
    • 非抢占内核不会导致竞争条件
    • 抢占内核需要考虑竞争条件

三项要求

  • 互斥(mutual exclusion)：如果进程在其临界区内执行，那么其他进程都不能在其临界区内执行 (无空等待)
  • 前进(progress)：如果没有进程在其临界区内执行，且有进程需要进入临界区，那么只有那些不在剩余区内执行的进程可参与选择，以确定谁能下一个进入临界区，且这种选择不能无限推迟 (有空让进)
  • 有限等待(bounded waiting)：从一个进程做出进入临界区的请求，直到该请求允许为止，其他进程允许进入其临界区的次数有上限

信号量

  • 用于解决临界区问题

信号量S是个整数变量，除了初始化外，只能通过两个标准原子操作来wait()和signal()来访问

  • 一个进程修改信号量时，不能有其他进程修改同一信号量的值，在wait(S)对S的整型值的测试和修改时也不被中断
  • 计数信号量的值域不受限制，而二进制信号量mutex的值只能为0或1，也称为互斥锁，因为它们可以提供互斥
  • 使用信号量的互斥实现
  • 也可用信号量synch来解决同步问题，令synch初始化为0
  • P2只有在P1调用signal(synch)之后，才会执行S2，因为synch初始化为0
  • 缺点：忙等待
  • 这种类型的信号量也称自旋锁(spinlock)
    • 自旋锁的优点：在等待锁时不进行上下文切换，而上下文切换可能需要花费相当长的时间

  • 进程的重新执行通过wakeup()操作来进行
  • block()和wakeup()是基本系统调用
  • 死锁deadlock：两个或多个进程无限地等待一个事件，而该事件只能由这些等待进程之一来产生
  • 无限期阻塞indefinite blocking 或饥饿starvation 即进程在信号量内无限期等待

第7章 死锁

  • 申请：如果申请不能立即被允许，那么申请进程必须等待，直到它获得该资源为止。
  • 使用：进程对资源进行操作
  • 释放：进程释放资源

死锁特征

  • 必要条件，同时满足4个条件会引起死锁
    • 互斥：至少有一个资源必须处于非共享模式，即一次只有一个进程。如果另一个进程申请该资源，那么申请进程必须等到该资源被释放为止
    • 占有并等待：一个进程必须占有至少一个资源，并等待另一个资源，而该资源为其他进程所占有
    • 非抢占：资源不能被抢占，即资源只能在进程完成任务后自动释放。
    • 循环等待：有一组等待进程，进程间相互等待其他进程占有的资源，等待的进程构成了一个循环

资源分配图

  • 由一个节点集合V和一个边集合E组成，节点集合V可分成两种类型的结点：P={P1，P2，…，Pn}(系统活动进程的结合)和R={R1，R2，…，Rn}（系统所有资源类型的集合)
  • 申请边：由进程Pi到资源Rj的有向边记为Pi->Rj，它表示进程Pi已经申请了资源类型Rj的一个实例，并正在等待该资源
  • 分配边：由资源类型Rj到进程Pi的有向边记为Rj->Pi，它表示资源类型Rj的一个实例已经分配给进程Pi，并被该进程占有
  • 用圆形来表示进程，用矩形来表示资源
  • 资源可能有多个实例，用矩形中的原点表示实例
    • 如果没有环，那么没有进程死锁，
    • 如果有环，就可能有死锁
    • 如果环涉及到的资源都只有一个实例，就有死锁，这种情况下环是死锁的充分条件
    • 如果环设计到的资源不止一个实例，则不一定有死锁，这种情况下环是死锁的必要条件

死锁处理方法

三种方法:

  • 可使用协议以预防或避免死锁，确保系统不会进入死锁状态
  • 可允许系统进入死锁状态，然后检测它，并加以修复
  • 可忽视这个问题，认为死锁不可能在系统内发生

解决方案：

  • 死锁预防(deadlock prevention)：确保至少一个必要条件不成立
  • 死锁避免(deadlock avoidance)：操作系统实现得到有关进程申请资源和使用资源的额外信息，判断是否让进程等待，允许当前申请或是延迟

死锁预防

缺点：低设备使用率和系统吞吐率

  • 互斥：通常不能通过否定互斥条件来预防死锁，有的资源本身就是非共享的
  • 占有并等待
    • 解决方案一：申请到资源再执行，破坏等待条件
    • 解决方案二：必须没有占有资源才能申请资源，破坏占有条件
  • 非抢占：解决：如果一个进程占有资源并申请另一个立即分配的资源，那么其现已分配的资源都可被抢占。(即资源被隐式释放，被占有的资源变得可抢占，等待中的进程其占有的资源也可被抢占)
  • 循环等待：一个解决办法是对所有资源类型进行完全排序，且要求每个进程按递增顺序来申请资源

死锁避免

用算法动态地检测资源分配状态以确保循环等待条件不可能成立

  • 安全序列：进程顺序<P1,P2,…,Pn> 如果对于每个进程仍然可以申请的资源数小于当前可用资源加上所有Pj(其中j<i)所占有的资源，则这一顺序成为安全序列
  • 在这种情况下，即使进程Pi所需要的资源不能立即使用，那么等待直到所有Pj释放其资源即可。
  • 不是所有的不安全状态都能导致死锁状态，所有的死锁状态都是不安全状态，安全状态就不会产生死锁

银行家算法

(对每种资源类型有多个实例的资源分配系统)

有效， 但效率低于资源分配图方案

  • 不能满足所有客户的需要时，银行绝不会分配其现金
  • Available：长度为m的向量，表示每种资源的现有实例数量，如果Available[j] = k，那么资源类型 Rj 现有k个实例
  • Max：n * m矩阵，定义每个进程的最大需求。如果Max【i】【j】= k，那么进程 Pi 最多可申请k个资源类型 Rj 的实例
  • Allocation：n * m 矩阵，定义每个进程现在所分配的各种资源类型的实例数量，如果Allocation【i】【j】= k，那么进程 Pi 现在已分配了 k 个资源类型 Rj 的实例
  • Need：n * m矩阵，表示每个进程还需要的神域的资源。Need【i】【j】= Max【i】【j】 - Allocation【i】【j】

安全性算法

（1） 设 Work 和 Finish 分别为长度为 m 和 n 的向量。按如下方式进行初始化：Work = Available 且对于 i = 0, 1, …, n-1, Finish[i] = false;

（2）查找这样的 i 使其满足

  • Finish[i] = false
  • Need_i <= Work
    • 如果没有这样的 i 存在，那么就转到第4步

（3）Work = Work + Allocation_i

  • Finish[i] = true
  • 返回到第2步

（4）如果对所有 i ，Finish[i] = true，那么系统处于安全状态

这个算法可能需要 m * n^2 数量级的操作以确定系统状态是否安全

资源请求算法

设Request_i为进程Pi的请求向量，如果Request_i[j] = k，那么进程 Pi 需要资源类型 Rj 的实例数量为k。当进程 Pi 作出资源请求时，采取如下动作：

（1）如果Request_i <= Need_i，那么转到第2步。否则，产生出错条件，这是因为进程 Pi 已超过了其最大请求。

（2）如果Request_i <= Available， 那么转到第3步，否则，Pi 必须等待， 这是因为没有可用资源

（3）假定系统可以分配给进程 Pi 所请求的资源，并按如下方式修改状态：

Available = Available - Request_i；

Allocation_i = Allocation_i + Request_i；

Need_i = Need_i - Request_i；

    • 加载时：如果编译时并不知道进程将驻留在内存的什么地方，那么编译器就必须生成可重定位代码，最后绑定会延迟到加载时才进行
    • 执行时：如果进程在执行时可以从一个内存段移到另一个内存段，那么绑定必须延迟到执行时才进行
  • 逻辑地址空间与物理地址空间
    • 逻辑地址：CPU所生成的地址（通常称为虚拟地址）
    • 物理地址：内存单元所看到的地址（即加载到内存地址寄存器中的地址）
  • 逻辑地址空间：由程序所生成的所有逻辑地址的集合
  • 物理地址空间：与这些逻辑地址相对应的所有物理地址的结合称为物理地址空间

  • 重定位寄存器：基地址寄存器

    • 重定位项：在程序中需要修改的位置成为重定位项
    • 重定位因子：程序装入内存中的起始地址
    • 重定位项表：为了支持静态重定位，连接程序在生成统一地址空间和装配
  • 动态重定位：在程序执行过程中进行地址重定位，即在每次访问内存单元前才进行地址变换
  • 静态链接：在生成可执行文件时进行的

第9章 虚拟内存

  • 虚拟内存（virtual memory）：在主存中可只装入最近经常要访问的某些区域的指令和数据，剩余部分就暂时不必装入，等到以后要访问到它们时再调入内存。
  • 如果主存较紧张，必要时可将已不大访问的信息调出内存，再执行调入操作。
  • 由于作业的指令和数据可以存放在外存只有操作系统可以修改中，用户的程序就不受实际内存大小的限制，好像计算机系统向用户系统提供了容量极大的“主存”
  • 这个大容量的“主存”是靠存储管理的软件和硬件通过大容量的辅存作为后援存储器扩充而获得的，是程序设计员感觉到的，而实际上并不存在的存储器，故称虚拟存储器

按需调页

  • 只有程序执行需要时才载入页，那些从未访问的也不会调入到物理内存
  • 懒惰交换：只有在需要页时，才将它调入内存
  • 调页程序
  • 缺页中断：检查进程页表，确定引用是否合法；非法引用则终止进程，引用有效但未调入内存则调入；找到空闲帧；调度磁盘，将页调入空闲帧；修改进程内部表和页表，表示已经调入；重新开始因陷阱而中断的指令

有效访问时间=（1-页错误率） 内存访问时间 + 页错误率 页错误处理时间

  • 三个主要的页错误处理时间，中断处理，读入页和重启进程。
  • 纯粹按需调页：只有在需要时才将页调入内存
  • 页表：该表能够通过有效-无效位或保护位的特定值，将条目设为无效
  • 次级存储器：该次级存储器用来保存不在内存中的页。次级存储器通常为快速磁盘，它通常称为交换设备。用来交换的这部分磁盘称为交换空间
  • 写时复制：允许父进程与子进程开始时共享同一页面。如果任何一个进程需要对页进行写操作，就创建一个共享页的副本

页面置换

  • 基本页置换：如果没有空闲帧，那么就查找当前没有使用的帧，并将其释放。
  • 释放的方式可以是：将其内容写到交换空间，并改变页表以表示该页不再内存中。
  • 包含页置换的页错误处理程序：
    • 查找所需页在磁盘上的位置
    • 查找一个空闲帧
      • 如果有空闲帧，那么就使用它
      • 如果没有空闲帧，那么就使用页置换算法以选择一个“牺牲”帧
      • 将“牺牲”帧的内容写到磁盘上，改变页表和帧表
    • 将所需页读入（新）空闲帧，改变页表和帧表
    • 重启用户进程

FIFO页置换

  • 记录每个页调入内存的时间
  • 选择最旧的页替换，可以不用准确记录而是用一个FIFO队列来管理内存中的页
  • 性能不佳，Belady异常：对于有些置换算法，页错误率会随着分配帧增加而增加

最优置换

最优页置换算法：置换最长时间不会使用的页

  • 产生页错误率最低，绝没有Belady异常的问题
  • 但无法知道后续的序列所以难以实现，需要预测最长时间不会使用的页

LRU页置换

最近最少使用算法：对于已有帧的页码，替换当距离当前页最远（最前）的页

  • 用计数器记录使用时间域
  • 用栈记录，每次使用就从栈中移除并放到顶部
  • 不会有Belady异常，和最优置换一样都是栈算法
  • 需要硬件支持

近似LRU页置换

引用位：每当引用一个页时（无论是对页的字节进行读或写），相应页表的引用位就被硬件置位。开始时清零，用过的置为1，不能知道使用顺序，只能知道哪些页用过

  • 二次机会算法，在FIFO页中拿到一个引用位为0的页就置换，为1的就将引用位清零并给第二次机会，选择下一个FIFO页
  • 增强型二次机会算法：引用位和修改位，使用率从00-01-10-11依次增加

  • 最不经常使用页置换算法（LFU）：置换计数最小的页
  • 最常使用页置换算法（MFU）：置换计数最大的页

帧分配

帧分配算法：

  • 平均分配
  • 根据进程大小，使用比例分配
  • 全局置换：全局置换允许一个进程从所有帧集合中选择一个置换帧，而不管该帧是否已分配给其他进程，即一个进程可以从另一个进程中拿到帧
  • 局部置换要求每个进程仅从自己的分配帧中进行选择

系统颠簸

颠簸（thrashing）：频繁的页调度行为，如果一个进程在换页上用的时间比执行时间多，那么这个进程在颠簸

  • 局部模型：局部是一个经常使用页的集合，一个程序通常由多个不同局部组成，它们可能重叠

    • 低级格式化（物理格式化）：磁盘分成扇区，每个扇区的数据结构由头数据区域和尾部组成。头部和尾部包含了一些磁盘控制器所使用的信息：纠错代码ECC（判断扇区是否损坏及用于找到出错数据）
    • 逻辑格式化：创建文件系统，将初始文件系统数据结构存储到磁盘上，包括空闲和已分配的空间（FAT或inode）和一个初始为空的目录
    • 簇（cluster）：由多个块集中而成的一个大块
  • 引导块:
    • 引导块放在磁盘的第一个扇区（引导扇区）MBR，包含引导代码和分区表
  • 坏块：
    • 控制器维护一个磁盘坏块链表，该链表在出厂前进行低级格式化时就已经初始化了，并在磁盘整个使用过程中不断更新
    • 扇区备用（sector sparing）或转寄（forwarding）：用操作系统看不到的备用块，由控制器用它来逻辑地替代坏块

第13章 I/O输入系统

  • 端口（port）：CPU 通过接口寄存器或特定电路与外设进行数据传送，这些寄存器或特定电路称之为端口

  • 总线（bus）：总线是一种内部结构，它是 cpu、内存、输入、输出设备传递信息的公用通道，主机的各个部件通过总线相连接，外部设备通过相应的接口电路再与总线相连接，计算机各种功能部件之间传送信息的公共通信干线

  • 控制器（controller）：用于操作端口、总线或设备的一组电子器件。按照预定顺序改变主电路或控制电路的接线和改变电路中电阻值来控制电动机的启动、调速、制动和反向的主令装置

  • I/O端口通常由4种寄存器：状态寄存器、控制寄存器、数据输入寄存器与数据输出寄存器
  • 轮询（polling）或忙等待：在该循环中，不断地抽取状态寄存器直到忙位被清除
  • 中断（interrupt）能使外设通知CPU的硬件机制
    • 机制：CPU硬件有一条中断请求线（IRL），CPU在执行完每条指令后，都将检测IRL，当检测到已经有控制器通过IRL发送了信号，CPU将保存当前状态并且跳转到内存固定位置的中断处理程序（interrupt-controller）
      • 进行关键处理时：能够延迟中断处理
      • 更为有效地讲中断分发到合适的中断处理程序，而不是检查所有设备以决定哪个设备引起中断
      • 需要多级中断，根据优先级响应紧迫性更高的中断
  • 非屏蔽中断，可屏蔽中断
  • 软中断或陷阱指令：有一个参数用来标识所需的内核服务
  • I/O应用接口：从I/O设备中抽象出一些通用类型，每个通用类型都可以通过一组标准函数（即接口）来访问
  • DST设备状态表：设备类型，地址和状态
  • 缓冲区：是用来保存两个设备之间或在设备和应用程序之间所传输数据的内存区域
  • 高速缓存（cache）可以保留数据副本的高速存储器
  • 假脱机（spooling）用来保存设备输出的缓冲区（如打印机，不能接收交叉的数据流，一次只能打印一个任务）
  • 为了防止用户执行非法I/O，定义所有I/O指令为特权指令，不能由用户程序发出，必须通过操作系统来进行，用户执行系统调用请求执行
