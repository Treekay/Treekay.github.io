云计算要点

发表于 2018-07-28 | 分类于 云计算

三. 云计算概念

​ 云计算是作为一种服务提供的一种特殊形式的分布式计算，引入效用模型远程地通过网络将可拓展、可测量的资源或软件和信息共享提供给计算机和其他设备。

​ 云计算=数据 (软件 + 平台 + 基础设施) 服务

提供

数据（Data） 传感器、物联网 、原始数据、半结构化数据和经过处理的结构化数据

  • 软件（Software） 检索、发现、关联、处理和创造数据
  • 平台（Platform）： 软件开发API、环境和工具
  • 基础设施（Infrastructure） 存储资源、计算资源、网络资源等
  • 服务（Service） IT服务化：产品、服务 XaaS

商业驱动力

  • 容量规划
  • 领先策略(Lead Strategy)：根据预期增加IT资源的容量
  • 滞后策略(Lag Strategy)：当IT资源达到其最大容量时增加资源容量
  • 匹配策略(Match Strategy)：当需求增加时，小幅增加IT资源容量
  • 购置和维护成本
  • 组织灵活性

技术创新

  • 集群化
  • 网格计算
  • 虚拟化

云

云（cloud）是指一个独特的IT环境，其设计目的是为了远程供给可扩展和可测量的IT资源。

IT资源（IT resource）是指一个与IT相关的物理的或虚拟的事物，它既可以是基于软件的，比如虚拟服务器或定制软件程序，也可以是基于硬件的，比如物理服务器或网络设备。

​ e.g. 物理服务器，虚拟服务器，软件程序，云服务，存储设备，网络

  • 可拓展性: IT资源可以处理增加或减少的使用需求的能力
  • 水平拓展: 向外（内），增加IT资源的数量
  • 垂直拓展: 向上（下），增加IT资源的容量

云服务

云服务（cloud service）是指任何可以通过云远程访问的IT资源。

四. 基本概念与模型

云提供者

提供基于云的IT资源的一方称为云提供者

云用户

使用基于云的IT资源的一方称为云用户，云用户使用云服务用户来访问云服务

云服务用户

云服务用户（cloud service consumer）一个临时的运行时角色，访问云服务的软件程序

云服务拥有者

在法律上拥有云服务的个人或者组织称为云服务拥有者，可以是云用户也可以是该云服务所在的云的云提供者（在云中部署了自己的服务的对象就是云服务拥有者，关键看云中的服务是谁部署的）

云资源管理者

可以是来自云用户或者云提供者，也可以第三方组织，就是签订了合约来管理基于云的IT资源的组织（不称为云服务管理者的原因是这个角色可能要管理不以云服务形式存在但又基于云的IT资源）

了解：云审计者、云代理、云运营商

组织边界

​ 组织边界是一个物理范围，包括由一家组织拥有和管理的IT资源的集合。

信任边界

​ 信任边界是一个逻辑范围，通常会跨越物理边界，表明IT资源受信任的程度

云的特性

  • 按需使用

  • 随处访问

  • 多租户

  • 弹性

  • 可测量的使用

  • 可恢复性

云交付模型

基础设施作为服务（IaaS）

  • 硬件资源作为服务提供给用户；
  • 资源通常是未配置好的，管理的责任直接落在云用户身上；
  • 典型例子：虚拟服务器；
  • 主要技术：虚拟化技术，资源动态管理和调度技术；

平台作为服务（PaaS）

  • 预先定义好的“就绪可用”的环境，已经部署好和配置好的IT资源；
  • 典型例子：运行平台如应用服务器，辅助系统软件如数据库；
  • 关键技术：分布式文件系统、分布式数据库，并行计算技术；
  • 好处：云用户省去了建立和维护裸的基础设施IT资源的管理负担，相对地给予云用户的控制权等级较低，不能访问平台的实现细节；

软件作为服务（SaaS）

  • 把软件程序定位成共享的云服务，将运行于云中的应用程序的功能交付给用户；

用户的管理权限十分有限，云服务用户可以访问玉衡服务，但是不能访问任何底层的IT资源或实现细节；

  • 关键技术：呈现技术，多租户技术；
  • 应用场景：通常是商用，如saleforce的CRM

三种云交付模型的比较

  • 云用户的控制等级：SaaS < PaaS < IaaS
  • 云用户的可用功能：前端用户接口访问 < 对使用平台相关的IT资源的管理控制 < 对基础设施相关的IT资源和可能的底层物理IT资源的完全访问
  • 云用户的行为：使用和配置云服务 < 开发、测试、部署、管理云服务 < 建立和配置裸的基础设施，安装、管理和监控所需的软件
  • 云提供者的行为：实现、管理和维护云服务，监控云用户的使用 < 实现配置好的平台和在需要时提供底层的基础设施中间件和其他所需的IT资源，监控云用户的使用 < 提供和管理需要的物理处理器、存储、网络和托管监控云用户的使用

云交付模型的组合使用

  • IaaS + PaaS

例子：一个提供PaaS环境的云提供者选择租用另一个不同的云提供者提供的IaaS环境; 原因可能是经济因素，也可能是原有的云提供者还服务者其他的云用户而它现有的容量快不够用了，或者其他原因是它的数据被要求物理地存放在某个特定的区域内。

  • IaaS + PaaS + SaaS

云用户组织可以利用这样的模型组合来开发部署自己的SaaS云服务

云部署模型

公有云

​ 由第三方云提供者拥有的可公共访问的云环境

社区云

​ 限制为特定的云用户社区可以访问的云环境

私有云

​ 一家组织单独拥有的，可以集中访问不同部分、位置或部门的IT资源

企业内部分工由单独的部门来承担云提供者的角色，区分“企业内部的”和“基于云的”概念很重要，因为组织和信任边界的定义和应用被改变

混合云

由两个或多个不同的云部署模型组成的云环境，可以将数据按敏感程度分别部署到公有云或私有云上。

缺点：架构的构建和维护复杂和具有挑战性

五. 云使能技术

宽带网络和Internet架构

  • Internet服务提供者ISP
  • 云用户和云提供者通常利用Internet进行通信。Internet以无中心的供给和管理模型为基础，不受任何集中式实体的控制
  • 网络互联架构的主要组件是使用网络路由器和交换机的无连接分组交换与基于路由器的互联
  • 网络带宽和延迟是影响QoS的因素，而网络拥塞对其有巨大影响

数据中心技术

  • 数据中心基本资源：云化的计算集群
  • 硬件：计算设备、存储设备、网络设备、配套设备
  • 软件：操作系统、系统软件、应用软件……
  • 数据中心基本技术：虚拟化，抽象物理计算和网络IT资源以便于资源分配、操作、
  • 释放、监视和控制；

  • 标准化与模块化

  • 自动化（自配置与自恢复）

  • 远程操作与管理

  • 高可用性（采用高冗余度）

  • 数据中心管理软件层次：业务服务管理平台、资源管理平台、基础设施资源池

  • 自动化部署：初次部署、迁移

  • 弹性能力提供：向上/下扩展，向内/外扩展

  • 资源监控，资源调度

虚拟化技术

  • 将物理IT资源转换为虚拟IT资源的过程；

  • 大多数IT资源都可被虚拟化：服务器，存储设备，网络，电源

  • 特性：硬件无关性，服务器整合，资源复制

基于操作系统的虚拟化—寄生架构（Hosted）

  • VM首先被安装在完整的宿主操作系统上，然后被用于产生虚拟机；

  • 在一个已存在的操作系统上安装虚拟化软件

    • 例子：备份和恢复，集成目录服务，安全管理

    • 缺点：

      • 宿主操作系统消耗CPU、内存等硬件IT资源；
      • 来自客户操作系统的硬件相关操作穿过层次多，整体性能降低
      • 每个操作系统需要独立的许可证

    • 优点：使用方便，功能丰富，适用于桌面系统

基于硬件的虚拟化—裸金属架构（Bare-metal）

  • 不再需要另一个宿主操作系统，将虚拟化软件直接安装在物理主机硬件上，

  • 从而绕过宿主操作系统

  • 优点：效率高，安全性高，适用于服务器系统

  • 缺点：高级功能少，使用不够方便

系统虚拟化技术

  • 硬件仿真，属于hosted架构
  • 全虚拟化，hosted和bare-metal都有；

在客户操作系统和硬件之间捕获和处理那些对虚拟化敏感的特权指令，使客户操作系统无需修改就能运行

  • 半虚拟化，bare-metal模式；

利用VMM实现对底层硬件的访问，Guest OS集成与半虚拟化有关的代码以配合VMM，无需重新编译或捕获特权指令

  • 硬件辅助虚拟化：结合到全/半虚拟化中使用，对软件技术进行硬件化来提高性能
  • 操作系统级虚拟化：对服务器操作系统进行隔离实现虚拟化（只有一份操作系统）

CPU 虚拟化

  • CPU 全虚拟化（模拟执行）

​ 优先级压缩技术，二进制代码翻译技术

  • CPU半虚拟化（操作系统辅助）
  • CPU硬件辅助虚拟化

内存虚拟化

做好虚拟机内存之间的隔离，使每个虚拟机都认为自己拥有了整个内存地址，并且效率也能接近物理机；影子页表，页表写入，EPT

I/O虚拟化

使虚拟机能访问到I/O资源并相互隔离，减轻虚拟化带来的开销

存储虚拟化

实现与应用/网络无关的数据存储和管理

设备层虚拟化：实现统一寻址；

块聚合层虚拟化：整合不同的物理存储设备

网络虚拟化

将网络资源和功能集成到一个软件中统一管控，增加容量、应对横向流量、大缓存低延迟等；两种形式：横向整合：多个变一个；纵向分割：VLAN等

  • Web技术
  • 多租户技术

设计多租户应用的目的是使得多个用户（租户）在逻辑上同时访问同一个应用，同时每个租户都不会意识到还有其他租户正在使用该应用，确保每个租户都不会访问到不属于自己的数据和配置信息

  • 服务技术

SOA模型

面向服务架构，将应用程序的不同功能单元（称为服务）通过接口和契约联系起来；接口是采用中立的方式进行定义的，独立于实现服务的硬件平台、操作系统和编程语言。构建在各种这样的系统中的服务以统一和通用的方式进行交互。

三种角色：服务提供者，服务请求者，服务代理；

媒介：服务总线；

三个操作：发布、查找、绑定

Web Services

  • 实现SOA的方式之一，平台无关、语言无关、通用性强；
  • 目标是即时装配、松散耦合以及自动集成

微服务架构

（一种软件架构风格）通过将传统的单块架构的应用拆分成一组细粒度的微服务，每个微服务运行在独立的进程中，可以单独部署，采用不同 的编程语言，通过语言无关的轻量级的通信机制进行相互调用，实现软件系统的松耦合。

五a. 云计算与大数据处理

云数据处理关键技术

  • 分布式数据管理
    • 分布式文件系统
    • 分布式数据库
  • 云并行计算技术
  • 分布式协同管理
  • 集群管理

分布式计算技术

云平台的分布式计算特性

  • CAP理论：Consistency、Availability、Partition-tolerance
  • 传统分布式计算注重“C”
  • 云计算平台注重“A”和”P”

MapReduce计算模型

  • Google发明的分布式计算模型
  • 适于海量数据处理
    • 传统分布式计算适于“计算”密集型任务
  • 基于数据分割

分布式协同管理

  • 共享资源的并行操作->数据不一致
  • 通过同步机制控制并发操作
  • 常用的并发控制方法
    • 基于锁的并发控制
      • 两阶段锁协议
      • 多副本的锁机制：读-写全法、多数法、主副本法、单一管理法
    • 基于时间戳的并发控制
      • 基于全局唯一的时间戳
    • 乐观并发控制
    • 基于版本的并发控制

集群和平台管理

  • 集群的自动化部署
    • 安装配置OS、DFS、分布式计算程序、作业管理软件、系统管理软件等
  • 集群作业调度
    • 基于资源管理器调度作业的运行节点和时间
    • 调度管理MapReduce任务
  • 远程控制
    • 开关机控制
    • 远程控制台
  • 应用管理与度量计费

分布式文件系统

基本特征

  • 透明性、并发访问、高可用性

基本需求

  • 数据冗余、异构性、一致性、高效性、安全性

基本架构

  • 按块存储、并行读取、效率高
  • 自动复制、多层次容错、原子操作保证一致性

HDFS

  • GFS的开源实现
  • 容量大：Terabytes or Petabytes
    • 将数据保存到大量的节点当中
    • 支持很大单个文件
  • 高可靠性、快速访问、高可扩展
    • 大量的数据复制
    • 简单加入更多服务器

HDFS适应的场景

  • HDFS是针对MapReduce设计
    • 数据尽可能根据其本地局部性进行访问与计算
  • 大量地从文件中顺序读
    • HDFS对顺序读进行了优化
    • 随机的访问负载较高
  • 数据支持一次写入，多次读取
    • 不支持数据更新（但可以直接进行文件替换）
  • 数据不进行本地缓存
    • 文件很大，且顺序读

HDFS的设计

  • 基于块的文件存储
  • 块进行复制的形式放置，按照块的方式随机选择存储节点
  • 副本的默认数目是3
  • 默认的块的大小是64MB
    • 减少元数据的量
    • 有利于顺序读写（在磁盘上数据顺序存放）

HDFS设计要点

  • 名字空间
  • 副本选择
  • 安全模式
    • 刚启动的时候，等待每一个DataNode报告情况
    • 退出安全模式的时候才进行副本复制操作
  • NameNode有自己的FsImage和EditLog，前者有自己的文件系统状态，后者是还没有更新的记录

HDFS可靠性

  • 磁盘数据错误

HDFS程序接口

  • 在MapReduce程序中使用HDFS
    • 通过fs.default.name的配置选项关联NameNode
  • 在程序中使用HDFS接口
    • 命令行接口
    • Hadoop MapReduce Job的隐含的输入
    • Java程序直接操作
    • lib/hdfs从c/c++程序中操作

HDFS权限控制与安全特性

  • 类似POSIX的安全特性
  • 不完全，主要预防操作失误
  • 不是一个强的安全模型，不能保证操作的完全安全性
  • 用户：当前登录的用户名，即使用Linux自身设定的用户与组的概念

负载均衡

  • 加入一个新节点的步骤
    • 配置新节点上的hadoop程序
    • 在Master的slaves文件中加入新的slave节点
    • 启动slave节点上的DataNode，会自动去联系NameNode，加入到集群中
  • Balancer类用来做负载均衡
    • 默认的均衡参数是10%范围内
    • bin/start-balancer.sh -threshold 5

分布式拷贝

  • bin/hadoop distcp

    hdfs://SomeNameNode:9000/foo/bar/
    hdfs://OtherNameNode:2000/baz/quux/

  • 目标也可以是s3://bucket-name/key

分布式数据库

  • 数据存储量和交易率的巨大需求
  • 数据库的可伸缩性
    • 方法1：MyCask或其他缓存，可扩展性有限
    • 方法2：使用现有的并行数据库，昂贵且最并行的数据库是为决策支持而不是OLTP设计的。
    • 方法3：用数据库构建并行存储

缩放关系数据库管理系统 - 分割法

  • “碎片”
    • 在一些廉价数据库(MySQL/PostgreSQL)中划分数据
    • 管理应用程序中的并行访问
    • 适合读写
    • 不透明，应用程序需要分区感知

CAP 定理

共享数据的三个性质：

  • 一致性（Consistency）：所有的副本的值相同
  • 可用性（Availability）：总能进行读写
  • 分区容错性（Partition-tolerance）：即使网络故障妨碍了一些设备之间的通信，一致性和/或可用性仍然有效

布鲁尔的CAP定理

  • 对于任何系统共享数据，不能保证同时满足三个性质
  • 最多有两个性质
  • 传统的数据库管理系统更注重”C”

云系统，分区是不可避免的

  • “C”和”A”都可以从中选择
  • 几乎在所有情况下都会选择”A”

NoSQL

主要特点

  • 非关系型，不需要模式
  • 数据被复制到多个节点并且可以分区：
    • 下行节点容易替换
    • 没有单一的故障点
  • 水平可缩放
  • 廉价，容易实现
  • 大规模写入性能
  • 快速键值存取

缺点

  • 不完全支持关系特性
    • 没有join，group by，order by 操作；
    • 没有声明性查询语言（如：SQL）
    • Relaxed ACID
    • 不容易与其他支持SQL的应用结合

类型

  • 键值存储数据库
    • 如：DynamoDB, Voldermort, Scalaris
  • 列存储数据库
    • 如：MongoDB, CouchDB
  • 文档型数据库
    • 如：BigTable, Cassandra, Hbased
  • 图形数据库
    • Neo4J, InfoGrid

“No-schema”是大多数NoSQL存储系统的共同特点，提供”灵活“数据类型

MapReduce技术

  • 一种新的编程框架/模型与运行环境
  • 适于大数据分析处理
  • 高度并行化、可扩展、透明、容错

动力：大规模数据处理

  • 处理大量数据（>1 TB）
  • 在千百级CPU之间并行执行

Word Count

Map
  • 接受输入键值/对
  • 发出中间键值/对
Reduce
  • 接受中间键值/对
  • 发出输出键值/对
InputSplits
  • 定义了输入到单个Map任务的输入数据
  • 一个MapReduce程序被统称为一个Job，可能有上百个任务构成
  • InputSplit将文件分为64MB的大小
    • hadoop-site.xml中的mapred.min.split.size参数控制这个大小
  • mapred.tasktracker.map.taks.maximum用来控制某一个节点上所有map任务的最大数目
RecordReader
  • InputSplit定义了一项工作的大小，但是没有定义如何读取数据
  • RecordReader实际上定义了如何从数据上转化为一个（key，value）对，从而输出到Mapper类中
  • TextInputFormat提供了LineRecordReader
Mapper
  • 来自数据源的记录
    • 从文件中逐行读出、或从数据库的行获得，等
    • 键值对：如（filename，line）
  • map（）函数从输入的数据中产生一个或多个中间值以及输出键
  • 每一个Mapper类的示例生成了一个Java进程
    • 在某一个InputSplit上执行
  • 有两个额外的参数OutputCollector以及Reporter
    • 前者用来收集中间结果
    • 后者用来获得环境参数以及设置当前执行的状态
Reducer
  • 在map阶段结束后，给定输出键的所有中间值都被组合成一个列表
  • reduce()函数将这些中间值组合为同一输出键的一个或多个最终值
Partition&Shuffle
  • 在Map工作完成之后，每一个Map函数会将结果传到对应的Reducer所在的节点
  • 用户可以提供一个Partitioner类，用来决定一个给定的（key，value）对传输的具体位置
Sort
  • 传输到每一个节点上的所有的Reduce函数接收到Key，value对会被Hadoop自动排序（即Map生成的结果传送到某一个节点的时候，会被自动培排序）
  • 默认：hash（key）mod R
  • 保证：
    • 相对平衡的分区
    • 分区内排序保证

六. 基本云安全

  • 保密性：只有被授权方才能访问

  • 完整性：云服务传送过程中数据不被未授权方篡改

  • 真实性：唯一性，真实可靠

  • 可用性：在特定的时间段内可以访问和可以使用

  • 威胁：潜在的安全型违反，可能视图破坏隐私并/或导致危害，以此挑战防护

  • 风险：执行一个行为带来损失或危害的可能性

威胁作用者

  • 匿名攻击者

  • 恶意服务作用者

  • 授信的攻击者（恶意的租户）

  • 恶意的内部人员

云安全威胁

  • 流量窃听：数据在传输到云中或在云内部传输时被恶意的服务作用者被动地截获
    • 目的：破坏数据的保密性，破坏云用户和云提供者之间关系的保密性
    • 特点：由于攻击被动的本质，容易长时间进行而不被发现

  • 恶意媒介：消息被恶意服务作用者截获并且篡改，破坏消息的保密性和完整性

  • 拒绝服务：使IT资源过载至无法正确运行

  • 授权不足：错误地授予了攻击者访问权限或是授权太宽泛，导致攻击者能够访问到本应 该受到保护的IT资源

  • 虚拟攻击：利用虚拟化平台中的漏洞来危害虚拟化平台的保密性。完整性和可用性。

  • 信任边界重叠

七. 云基础设施机制

逻辑网络边界

将一个网络环境与通信网络的其他部分隔离开来，形成了一个虚拟网络边界。

用于：将云中的IT资源与非授权用户或非用户、云用户隔离，控制被隔离IT资源的可用带宽；

虚拟防火墙

一种IT资源，可以主动过滤杯隔离网络的网络流量，并控制其与Internet的交互。

虚拟网络

一般通过VLAN形成，这种IT资源用来隔离数据中心基础设施内的网络环境。

虚拟服务器

模拟物理服务器的虚拟化软件，是最基本的云环境构建块。每个虚拟服务器都可以存储大量的IT资源、基于云的解决方案和各种其他的云计算机制。从映像文件进行虚拟服务器的实例化是一个可以快速且按需完成的资源分配过程。

云存储设备

  • 云存储设备是指专门为基于云配置所设计的存储设备。

  • 支持按使用计费，提供固定增幅的容量分配

  • 主要问题是数据的完整性，保密性，安全性

  • 常见数据存储逻辑单元：文件、块、数据集、对象
  • 三种技术接口：网络存储接口（文件或块）、对象存储接口（Web资源），数据库存储接口（分为关系数据存储和非关系数据存储NoSQL）

云使用监控

​ 三种常见的实现形式：监控代理、资源代理、轮询代理

​ 监控代理是一个中间的事件驱动程序，作为服务代理

​ 资源代理是一种处理模块，轮询代理是一种处理模块

资源复制：对同一个IT资源创建多个实例，加强IT资源的可用性和性能，复制基于云的IT资源

已就绪环境：已就绪环境机制是PaaS云交付模型的定义组件，代表的是预定义的基于云的平台，该平台由一组已安装的IT资源组成，可以云用户使用和定制。云用户可以使用这些环境在云内远程开发和配置自身的服务与应用程序。

典型的已就绪环境包括预安装的IT资源，如数据库、中间件、开发工具等

八. 特殊云机制

自动伸缩监听器

  • 一个服务代理，它监控和追踪云服务用户和云服务之间的通信，用以动态自动伸缩。

  • 通常部署在靠近防火墙的位置，来自动追踪负载状态信息

  • 负载量可以由云用户产生的请求量或某种类型的请求印发的后端处理需求量来决定

  • 对于不同负载波动的条件，自动伸缩监听器可以提供不同类型的响应：

    • 自动伸缩IT资源：根据事先定义的参数

    • 自动通知云用户：负载过高或过低时

负载均衡器

  • 一个运行时代理，主要用于把负载在两个或更多的IT资源上做负载

  • 负载分配功能：

    • 非对称分配
    • 负载优先级
    • 上下文感知的分配

  • 目标：优化IT资源的使用，避免过载并最大化吞吐量

  • 机制：

    • 多层网络交换机
    • 专门的硬件设备
    • 专门的基于软件的系统（在服务器操作系统中比较常见）
    • 服务代理（通常由云管理软件控制）

  • 负载均衡器通常位于产生负载的IT资源和执行负载处理的IT资源之间的通信路径上

  • 负载均衡器可以被设计成一个透明的代理或是一个代理的组件

SLA监控器

  • 被用来专门观察云服务的运行时性能，确保它们履行了SLA中报告的约定的QoS需求
  • SLA监控器收集的数据由SLA管理系统处理并集成到SLA报告的标准中

（PS.详见第九章)

按使用付费监控器

  • 机制，按照预先定义好的定价参数测量基于云的IT资源使用，并生成使用日志用于计算费用
  • 使用数据由计费管理系统进行处理
  • 一些典型的监控变量包括：
    • 请求/响应消息数量
    • 传送的数据量
    • 带宽消耗
  • 实现方式
    • 资源代理
    • 监控代理

审计监控器

  • 用来收集网络和IT资源的审计记录数据，用以满足管理需要或者合同义务

故障转移系统

  • 通过使用现有的集群技术提供冗余的实现来增加IT资源的可靠性和可用性
  • 只要当前活跃的IT资源变得不可用时，便会自动切换到冗余的或待机IT资源实例上
  • 类型：
    • 主动-主动
      • 多个实例都处于活动状态，同时提供服务
      • IT资源的冗余实现和负载均衡器是必须要的
    • 主动-被动
      • 一个处于活动状态，一个待机或闲置
      • 当IT资源变得不可用的时候，就会激活待机实例来接管工作
      • 相应的工作负载就会被重定向到接管操作的这个实例上

虚拟机监控器

  • 虚拟化基础设施的最基本部分，主要用来在物理服务器上生成虚拟服务器实例。
  • 通常受限于一台物理服务器（即一个虚拟机监控器由一台物理服务器来管理）
  • VIM提供了一组特性来管理跨物理服务器的多虚拟机监控器

资源集群

  • 把多个IT资源实例分为一组，使得他们能像一个IT资源那样进行操作
  • 通过高速专用网络链接或者集群结点是实现工作负载、任务调度、数据共享和系统同步等通讯要求
  • 作用：增强了集群化IT资源的组合计算能力、负载均衡能力和可用性
  • 常见的资源集群类型包括：
    • 服务器集群：提高性能和可用性
    • 数据库集群：提高数据可用性，维持数据的一致性
    • 大数据集集群：数据的分区和分布
  • 两种基本类型:
    • 负载均衡的集群：保持集中管理的特性下实现了在集群结点中的分布式工作负载
    • 高可用集群：在多节点失效的情况下保持系统的可用性，需要冗余实现和故障转移机制

多设备代理

  • 用来帮助运行时的数据转换
  • 克服云服务和多样性的云服务用户之间的不兼容性
  • 使得云服务能够被更广泛的云服务用户程序和设备所使用
  • 需要创建映射逻辑来改变运行时交换的信息
  • 通常是作为网关存在的：
    • XML网关
    • 云存储网关
    • 移动设备网关

状态管理数据库

  • 是一种存储设备，用来暂时地持久化软件程序的状态数据，软件程序可以把状态数据卸载到数据库汇总
  • 替代状态数据缓存在内存中的一种办法，用以降低程序占用的运行时的内存量
  • 状态管理数据库使得软降程序和周边的基础设施都具有更大的可拓展性

九. 云管理机制

远程管理系统

  • 远程管理系统机制向外部云资源管理者提供工具和用户界面来配置并管理基于云的IT资源
  • 远程管理系统能够建立一个入口以便访问各种底层系统的控制与管理功能，包括资源管理、SLA管理和计费管理
  • 远程管理系统提供的工具和API一般被云提供者用来开发和定制在线入口，这些入口向云用户提供各种管理控制
  • 主要创建如下两种类型的入口：
    • 使用与管理入口：一种通用入口，集中管理不同的基于云的IT资源，并提供IT资源使用报告
    • 自主服务入口：本质上是一个购买门户，允许云用户搜索云提供者提供的最新云服务和IT资源列表
  • 通过远程管理控制奥体，云用户通常能够执行的任务包括：
    • 配置与建立云服务
    • 为按需云服务提供和释放IT资源
    • 监控云服务的状态、使用和性能
    • 监控QoS和SLA的实行
    • 管理租赁成本与使用费用
    • 管理用户账户、安全凭证、授权和访问控制
    • 跟踪对租赁服务内部与外部的访问
    • 规划与评估IT资源供给
    • 容量规划

资源管理系统

  • 资源管理系统机制帮助协调IT资源，以便相应云用户和云提供者执行的管理操作
  • 此系统的核心是虚拟基础设施管理器（VIM），用于协调服务器硬件，这样就可以从最合适的底层物理服务器创建虚拟服务器实例
  • VIM是一个商业化产品，用于管理一系列跨多个物理服务器的IT资源
  • 通常通过资源管理系统自动化并是吸纳的任务包括：
    • 管理虚拟IT资源模板，如虚拟服务器映像
    • 分配和释放虚拟IT资源，比如开始、暂停、继续和终止
    • 在有其他机制参与的条件下，协调IT资源，比如资源复制、负载均衡和故障转移系统
    • 在云服务实例的生命周期内，强制执行使用策略和安全规定
    • 监控IT资源的操作条件

SLA管理系统

  • SLA管理系统机制代表的是一系列商品化的可用云管理产品，这些产品提供的功能包括：SLA数据的管理、收集、存储、报告以及运行时通知
  • 部署SLA管理系统时，常常会包含一个库，用于存储和检索被收集的基于预定义指标和报告参数的SLA数据

计费管理系统

  • 计费管理系统机制专门用于手机和处理使用数据，它涉及云提供者的结算和云用户的计费
  • 计费管理系统依靠按使用付费监控器来收集运行时使用数据，用于计费、报告和开发票等目的
  • 计费管理系统允许指制定不同的定价规则，还可以针对每个云用户或每个IT资源自定义定价模型
  • 定价模型可以是传统的按使用付费模型，也可以是固定费率或按分配付费模型，还可以是它们的组合
  • 计费是根据使用前支付和使用后支付来安排的

十. 云安全机制

  • 安全属性：
    • 保密性：是指只有被授权方才能访问的特性
    • 完整性：是指未被未授权方篡改的特性
    • 真实性：是指事务是由经过授权的源提供的这一特性
    • 可用性：是在特定的时间段内可以访问和可以使用的特性

加密

  • 加密
    • 是一种数据编码系统，专门用来保护数据的保密性和完整性
    • 用于对抗流量窃听、恶意媒介、授权不足和信任边界重叠这样一些安全威胁
  • 加密部件
    • 加密用的标准化算法，把原始的明文数据转换成加密的密文数据
  • Key - 密钥

加密的类型

  • 基于流的密码
    • 一次一个
    • 将明文与密钥流混合
    • 有利于实时服务
  • 基于块的密码
    • 取代和换位

加密的方法

对称加密

  • 加密和解密的相同密钥
  • 密钥分配问题

非对称加密

  • 基于无法逆向求解的数学问题
    • 整数分解、离散对数、椭圆曲线
  • 通过一个大的随机数产生一对密钥
    • 私钥、公钥

混合加密

  • 结合了两种的优点
  • 非对称分配对称密钥
    • 也称为会话密钥
  • 对称提供批量加密

对称 VS. 非对称

  • 对称加密保证了数据的保密性，但是并不保证数据的不可否认性
  • 非对称加密依赖于使用两个不同的密钥，称为私钥和公钥
    • 用自己的私钥加密给对方，对方用公钥解开
      • 真实性、不可否认性和完整性保护（不提供保密性）
    • 用公钥加密送给私钥拥有者
      • 保证了保密性（不保证完整性、真实性的保护）

哈希

  • 哈希
    • 用来获得消息的哈希代码或者消息摘要，通常是固定的长度，小于原始的消息大小
    • 单向的、不可逆转的数据保护
    • 主要属性：计算简单、不可逆、不重复
  • 典型应用
    • 数据保密：如密码哈希
    • 恶意媒介和授权不足
    • 服务滥用：（如DoS）Proof-of-work
    • 文件标识：P2P文件系统、软件版本管理工具

数字签名

  • 数字签名
    • 将消息进行哈希，然后用私钥进行加密
  • 作用
    • 通过身份验证和不可否认性来提供数据真实性和数据完整性
    • 类似于日常生活中的“签名”
  • 算法
    • RSA是最常用的签名算法
  • 扩展应用
    • 数字证书：个人安全邮件证书
    • 智能卡：将数字签名放入便携物理存储介质

公钥基础设施

  • 公钥基础设施
    • 是一个由协议、数据格式、规则和实施组成的系统
    • 是用于大规模、公共系统的公钥加密技术
  • 数字证书
    • 带数字签名的数据结构，验证证书拥有者身份以及相关信息
    • 通常是由第三方证书颁发机构签发的
  • PKI与云计算
    • 实现非对称加密
    • 管理云用户和云提供者身份信息
    • 防御恶意中介和不充分的授权威胁

身份与访问管理IAM

  • 身份与访问管理
    • 控制和追踪用户身份以及IT资源、环境、系统访问特权
  • 四个主要部分组成：
    • 认证
    • 授权
    • 用户管理
    • 证书管理
  • IAM机制功能：
    • 分配用户特权等级
    • 访问控制和策略
  • IAM机制的用途
    • 对抗授权不足、拒绝服务供给和信任边界重叠等威胁
    • 比PKI机制要更全面

单一登录

  • 单一登录机制
    • 使得一个云服务用户能够被一个安全代理认证并建立起一个安全上下文
    • 当云用户要访问其他云服务或资源的时候，这个上下文会被持久化
  • 实现跨云服务的用户认证和授权
  • 允许互相独立的云服务和IT资源产生并流通运行时认证和授权证书
  • 不直接对抗任何一个云安全威胁
  • 增强了基于云的环境的访问并管理分布式IT资源和解决方案的可用性

基于云的安全组

  • 云资源分割
    • 为不同用户和组创建各自的物理和虚拟IT环境的过程
    • 云资源分割是虚拟化的基础
  • 基于云的安全组
    • 基于云的资源分割，网络被分成逻辑的基于云的安全组，形成逻辑网络边界
    • 运行在同一个物理服务器上的多个虚拟服务器逻辑上可以是不同基于云的安全组的成员
    • 基于云的安全组能限制对资源的未被授权的访问
    • 用来对抗拒绝服务、授权不足和信任边界重叠等威胁

强化的虚拟服务器映像

  • 虚拟服务器镜像
    • 一个从模板配置创建的虚拟服务器镜像
    • 比原始标准映像更加安全的虚拟服务器模板
  • 强化
    • 把不必要的软件从系统中剥离出来。限制可能被攻击者利用的潜在漏洞的过程
    • 去除冗余的程序，关闭不必要的服务器端口，关闭不使用的服务，内部root用户和guest访问等
  • 对抗拒绝服务、授权不足和信任边界重叠等威胁

十一. 基本云架构

负载分布架构

  • 负载分布架构
    • 通过增加一个或多个相同的IT资源可以进行IT资源水平扩展，而提供运行时逻辑的负载均衡器能够在可用IT资源上均衡分配工作负载
    • 以来负载均衡算法和运行时逻辑
  • 分布式IT资源
    • 多个同类型IT资源同时提供服务
    • 水平扩展：增加/减少相同的IT资源
  • 负载均衡
    • 在可用IT资源上均匀分配工作负载
  • 负载均衡器
    • 资源入口处执行负载均衡的实体
    • 调度方法
      • 基于DNS
      • 基于虚拟IP
      • 基于链路聚合：用于整合链路提高网络传输能力
      • 基于应用：用于分配到分布式调度器
    • 调度策略
      • 轮转、负载水平,……
      • 同一用户的多个请求调度到同一服务器
      • 同一租户的请求调度到尽量少的一组服务器
      • 尽量实现不同类型负载的互补
  • 负载分布架构的多个构成部分
    • IT资源
      • 资源副本
      • 资源集群
    • 服务入口
      • 负载均衡器
      • 逻辑网络边界
    • 监控与审计
      • 使用监控器
      • VM监控器
      • 审计监控器

服务负载均衡架构

  • 服务负载架构
    • 可以被认为是工作负载分布架构的一个特殊变种
    • 专门针对扩展云服务实现
  • 基于云服务的冗余部署
    • 部署在虚拟机上或容器中的应用程序或软件
    • 多实例构成资源池
  • 两种形式
    • 独立于云设备及主机服务器
    • 作为应用程序/服务器环境的内置组件
  • 其他机制
    • 云使用监控器
    • 资源集群
    • 资源复制

资源池架构

  • 资源池架构
    • 使用一个或多个资源池为基础，其中相同的IT资源由一个系统进行分组和维护，以确保它们保持同步
  • 常见的资源池有：
    • 物理服务器池
    • 虚拟服务器池
    • 存储池
    • 网络池
    • CPU池
    • 内存池
  • 可以每种类型的IT资源创建专用池，也可以单个或多个池集合成一个更大的池

层次资源池架构

  • 资源池可以建立层次结构，形成父子、兄弟和嵌套关系，利于构成不同的资源池需求

同级资源池

  • 通常来自物理上的同一组IT资源

  • 同级资源池之间是相互隔离的，云用户只能访问各自的资源池

嵌套资源池

  • 较大的资源池被分解成较小的资源池，每个小资源池分别包含了与大资源池相同类型的IT资源

  • 嵌套资源池可以用于向同一个云用户组织的不同部门或者不同组分配资源池

资源池使能技术

  • 云存储设备和虚拟服务器
  • 辅助技术：
    • 审计监控器
    • 云使用监控器
    • 虚拟机监控器
    • 逻辑网络边界
    • 按使用付费监控器
    • 远程管理系统
    • 资源管理系统
    • 资源复制

动态可扩展架构

  • 动态可扩展架构
    • 基于预先定义的扩展条件从资源池中动态分配IT资源
    • 云资源弹性管理的核心机制
  • 自动扩展监听器
    • 是实现动态扩展的核心部件
    • 监听负载状态以决定何时扩展
  • 动态扩展类型
    • 动态水平扩展
      • 向内或向外扩展IT资源实例
      • 自动扩展监听器请求资源复制，并发信号启动IT资源复制
    • 动态垂直扩展
      • 调整单个IT资源的处理容量
      • 向上或向下扩展IT资源实例
    • 动态重定位
      • 将IT资源重放置到更大/更小容量的主机上
  • 使用的机制
    • 云使用监控器
    • 虚拟机监控器
    • 按使用付费监控器

弹性资源容量架构

  • 弹性资源容量架构
    • 主要与虚拟服务器的动态供给相关
    • 根据负载变化分配和回收CPU与RAM资源
  • 扩展逻辑在智能自动化引擎
    • 自动扩展监听器发信号给智能引擎(一种脚本)
    • 智能引擎运行工作流逻辑，触发扩展
  • 扩展属于垂直扩展
  • 依赖一些额外机制
    • 云使用监控器
    • 按使用付费监控器
    • 资源复制
  • 变种
    • 包含工作流逻辑的脚本通过VIM发送其扩展请求
    • 而不是直接发送给虚拟机监控器

云爆发架构

  • 动态扩展架构的特殊形式
  • 云数据中心和企业IT资源协同
    • 企业内部IT资源<- -="">云IT资源
  • 基于云的IT资源是冗余性预部署
    • 保持非活跃状态，直到发生云爆发
  • 架构模型基础
    • 自动扩展监听器
    • 资源复制机制

弹性磁盘供给架构

  • 建立了一个动态存储供给系统，确保按照云用户实际使用的存储量进行精确计费
  • 基于自动精简供给技术
  • 辅助机制
    • 云使用监控器
    • 资源复制
  • 通常的收费方式：按照预定义的磁盘容量收费

冗余存储架构

  • 引入了复制的辅云存储设备作为故障系统的一部分，与主云存储设备中的数据保持同步
  • 主设备失效时，存储设备网关把用户请求转向辅设备
  • 主要依靠的的存储复制系统
    • 使得主云存储设备与其复制的辅云存储谁被保持同步
  • 通常部署在不同的地理区域
    • 抗灾容错
    • 经济成本
    • 法律问题

十二. 高级云架构

虚拟机监控器集群架构

  • 虚拟机监控器集群
    • 建立了一个跨多个物理服务器的高可用虚拟机监控器集群
    • 如果一个给定的虚拟机监控器或其底层物理服务器变得不可用，则被其托管的虚拟机服务器可迁移到另一物理服务器或虚拟机监控器上来保持运行时操作
  • 虚拟机监控器集群由中心VIM控制
    • 通过常规心跳消息来检测虚拟机监控器的状态
    • 当物理机或虚拟机监控器失效时进行在线迁移
    • 使用共享云存储设备实现在线迁移

粗粒度负载均衡与资源预留

  • 负载均衡的粒度有很多
  • 用户请求
    • 前端LB把用户请求分发给不同的云服务实例
  • 虚拟机服务器实例
    • 把虚拟服务器放置到不同的物理机
  • 云负载均衡
    • 数据中心间的负载均衡

负载均衡的虚拟服务器实例架构

  • 由于物理服务器之间保持跨服务器的工作负载均衡是很难实现的，运行和管理是相互隔离的，需要建立一个容量看门狗系统
  • 包含的机制
    • 虚拟机监控器
    • 资源集群
    • 虚拟服务器
    • (容量看门狗)云使用监控器
    • 自动伸缩监控器
    • 负载均衡器
    • 逻辑网络边界
    • 资源复制

容量看门狗系统

  • 动态地计算虚拟服务器实例及其相关的工作负载，再把处理任务分配到可用的物理服务器主机
  • 组成
    • 一个云使用监控器
    • VM在线迁移程序
    • 容量计划器

云负载均衡架构

  • 云负载均衡架构
    • 一个特殊的架构模型
    • IT资源可以在多个云之间进行负载均衡
  • 目的
    • 提高IT资源的性能和可扩展性
    • 增加IT资源的可用性和可靠性
    • 改进负载均衡和IT资源优化
  • 基础
    • 自动伸缩监听器
    • 故障转移系统

资源预留架构

  • 资源受限
    • 太多并发访问可能会导致运行时异常
    • IT资源没有足够的容量
  • 资源预留
    • 专门为给定的云用户保留
      • 单个IT资源
      • 一个IT资源的一部分
      • 多个IT资源
  • 依赖多个部件和机制
    • 审计监控器
    • 云使用监控器
    • 虚拟机监控器
    • 逻辑网路边界
    • 资源复制

云服务容错架构

  • 云服务不可用的原因
    • 运行时需求超出处理能力
    • 维护更新导致的暂时中断
    • 云服务迁移
    • 物理机失效、宕机
  • 应对机制
    • 服务迁移：不中断服务重定位架构
    • 服务故障检测与恢复：动态故障检测与恢复架构
    • 物理机容错

动态故障检测与恢复架构

  • 建立一个弹性的看门狗系统，以监控范围广泛的预先定义的故障场景，并对之做出相应
  • 弹性看门狗系统
    • 五个核心功能
      • 监视
      • 选定事件
      • 对事件做出反应
      • 报告
      • 升级处理
    • 智能看门狗监控器升级一个问题的处理最常采用的措施包括
      • 运行一个批处理文件
      • 发送一个控制台消息
      • 发送一条短信消息
      • 发送一份电子邮件消息
      • 发送一个SNMP陷阱
      • 记录一个通知单
  • 零宕机架构
    • 物理机不中断机制
    • 服务器会聚成一组，由容错系统控制
    • 虚拟服务器都存储在共享介质

裸机供给与快速供给架构

  • 裸机
    • 指没有预装操作系统或其他任何软件的物理服务器

裸机供给架构

  • 弹性增加裸机
  • 基于ROM提供的远程安装支持自动安装系统
    • 连接默认或DHCP配置IP
    • 通过Web或专有接口连接到物理机
    • 自动安装操作系统或软件
  • 主要构成
    • 发现代理
      • 一种监控代理
      • 搜索并找到可用的物理机
    • 部署代理
      • 裸机供给的客户端
      • 加载到物理服务器RAM中
    • 部署组件
      • 用于安装操作系统

快速供给架构

  • 快速供给架构
    • 实现大范围的IT资源供给的自动化
    • 单个IT资源或者复合IT资源
    • 大用户量或者大资源量
  • 主要构成部分
    • 顺序管理器
      • 组织自动化供给任务的顺序
        顺序日志记录器

存储负载管理架构

  • 存储负载管理架构
    • 使得LUN可以均匀地分布在可用的云存储设备上
  • LUN迁移
    • 把LUN从一个存储设备都能到另一个上而无需中断
    • 同时还对云用于保持透明
  • 存储容量系统
    • 则用来确保运行时工作负载均匀地分布在LUN上
  • 云存储设备合并成一个组
    • LUN数据在可用的存储主机上均匀地分布
  • 放置一个自动伸缩监听器
    • 监控并且动态分配云存储设备之间的运行时工作负载
  • 其他支持机制
    • 审计监控器
    • 负载均衡器
    • 云使用监控器
    • 自动伸缩监听器
    • 逻辑网络边界
