操作系统要点

发表于 2018-07-28 | 分类于 操作系统

第1章 导论

操作系统概念

  • 作为硬件和用户之间的中介程序，是管理计算机硬件的软件
  • 从用户视角：使用方便、性能、资源使用率等考量点决定了操作系统的目标
  • 从系统视角：作为资源分配器，管理资源，解决请求冲突，解决CPU时间、内存空间、文件存储空间、I/O设备等问题
  • 定义操作系统：一直运行在计算机上的程序（内核），其他程序为系统程序和应用程序

计算机系统组织

  • I/O和CPU可以并发执行
  • CPU、设备控制器由总线相连，对共享内存进行访问，协调内存竞争问题，需要内存控制器
  • CPU通过局部缓存与主存交换数据

  • 控制器和设备之间的接口

  • 引导程序(bootstrap program)初始化系统中的所有部分，包括CPU寄存器、设备控制器和内存，并将操作系统装入内存执行

  • 事件发生通过中断表示，硬件可随时通过系统总线向CPU 发出信号触发中断，软件通过系统调用(监视器调用monitor call)触发中断
  • CPU中断时停止当前工作并转到中断服务程序(interrupt service routine)的开始位置
  • 内存容量小，易失性，断电丢失，因而需要辅存(secondary storage)
  • 存储设备层次
    • 寄存器
    • 高速缓存cache
    • 主存
    • 电子磁盘
    • 磁盘
    • 光盘optical disk
    • 磁带
  • I/O的执行，设备驱动程序(device driver)在设备控制器中装载适当的寄存器，设备控制器检查寄存器内容决定执行什么操作。

计算机系统体系结构

  • 单处理器系统，用专用处理器接受主CPU的请求执行自己的磁盘队列和调度算法，以此克服主CPU的磁盘调度超载问题
  • 多处理器系统的优点
    • 增加吞吐量
    • 规模经济
    • 增加可靠性
  • 适度退化(graceful degradation) : 提供与正常工作的硬件成正比的服务的能力
  • 容错（fault tolerant)：超出适度退化的能力
  • 非对称多处理和对称多处理(SMP)
    • 非对称：主处理器控制系统，其他处理器从主处理器获得并做各自特定的任务
    • 对称：所有处理器都要完成操作系统中的所有任务
  • 集群系统(Clustered Systems)将多个CPU集中起来完成计算任务，和多处理器系统的不同之处在于，它是由两个或多个独立的系统耦合起来的。共享存储，通过LAN(局域网络)连接或内部连接
    • 高可用性、热备份模式(hot standby mode)
  • 并行集群：用专门软件来是实现多主机访问共享存储数据，完全访问，需要加锁，分布式锁管理器(DLM)

操作系统结构

  • 多道程序设计(Multiprogramming)：组织作业，同时将多个任务保存在内存
  • 分时系统(Time sharing)：在作业之间切换来执行多个作业
  • 进程：装入到内存并执行的程序
  • 作业池(job pool):等待分配内存的作业
  • 作业调度(job scheduling):多个作业需要调入内存时，要做出的选择
  • CPU调度(CPU scheduling):多个任务需要执行时做出的选择

操作系统操作

操作系统是由中断驱动的。没有进程、I/O服务、用户请求要响应时，操作系统会等待事件发生。

  • 事件是由中断(trap) 或陷阱引起的
  • 陷阱(或异常exception)是一种软件中断，由于出错或用户程序的特别请求引起

双重模式操作

CPU有两种模式：实模式和保护模式

  • 实模式下，用户程序不受限制
  • 保护模式下，用户程序被限制，此时操作系统有下面特点：
    • 双重模式操作
      • 用户模式
      • 内核模式(管理模式、系统模式、特权模式)
    • 模式位(mode bit)为0为内核模式，1为用户模式
    • 一旦出现陷阱或中断，硬件会从用户模式切换到内核模式
    • 提供了I/O保护、内存保护、CPU保护
  • 操作系统得到对计算机的控制时总是内核模式，能引起损害的机器指令作为特权指令，若在用户模式下试图执行特权指令，则视作非法指令不执行，以陷阱的方式通知操作系统。
  • 所有的I/O指令都是特权指令
  • 定时器:定时产生中断，把控制权交给操作系统

第3章 进程

进程概念

进程

  • 批处理系统中称为作业
  • 分时系统中称为用户程序或任务

  • 进程是执行中的程序(文本段)、当前活动、进程堆栈端、数据段、堆

  • 栈段：编译时编译器计算并设定好，运行时由系统自动分配和释放

  • 堆区：通常由程序员分配释放。若程序员不释放，程序结束时可能由OS回收。与数据结构中的堆是两回事，可看成内存块的(链)表
  • 数据段(含区安居区、静态区和常量区)：全局区或静态区——存放全局变量和静态变量，通常是放在一块的，程序结束后由OS回收。通常初始化的全局变量和静态变量在一块区域，未初始化的全局变量和未初始化的静态变量在相邻的另一块区域。常量区是另一块，存放常量字符串。也由OS回收 。
  • 程序代码段(文本段)：存放程序的二进制代码

进程状态

  • new：进程正在被创建
  • running：进程正在被执行
  • waiting：进程等待某个事件的发生
  • ready：进程等待分配处理器
  • terminated：进程完成执行

进程控制块

  • 进程控制块(PCB)

  • 进程状态：状态可包括新的、就绪、运行、等待、停止等

  • 程序计数器：计数器表示进程要执行的下个指令的地址
  • CPU寄存器：根据计算器体系结构的不同，寄存器的数量和类型也不同。它们包括累计器、索引寄存器、堆栈指针、通用寄存器和其他条件码信息寄存器。与程序计数器一起，这些状态信息在出现中断时也需要保存，以便进程以后能正确地继续执行
  • CPU调度信息：这类信息包括进程优先级、调度队列的指针和其他调度参数
  • 内存管理信息：根据操作系统所使用的内存系统，这类信息包括基址和界限寄存器的值，页表或段表。
  • 记账信息：这类信息包括CPU时间、实际使用时间、时间界限、记账数据、作业或进程数量等
  • I/O状态信息：这类信息包括分配给进程的I/O设备列表、打开的文件列表等

进程调度

调度队列

  • 作业队列(Job queue)：进程进入系统时，会被加到作业队列，该队列包括系统中所有进程
  • 就绪队列(ready queue)：驻留在内存中就绪的、等待运行的进程保存在就绪队列中。该队列通常用链表来实现，其头结点包含指向链表的第一个PCB块和最后一个PCB块的指针，每个PCB包括一个指向就绪队列的下一个PCB的指针域
  • 设备队列(Device queues)：等待特定I/O设备的进程列表称为设备队列，每个设备都有自己的设备队列

当进程分配到CPU并执行时，可能发生几种情况：

  • 进程可能发出一个I/O请求，并被放到I/O队列(设备队列)
  • 进程可能创建一个新的子进程，并等待其结束
  • 进程可能会由于中断而强制释放CPU，并被放回到就绪队列

进程完成后将从所有队列中删除，其PCB和资源将被释放

调度程序

  • 进程的选择由相应的调度程序(scheduler)来执行
  • 长期调度程序(long-term scheduler)或作业调度(job scheduler)：从大容量存储设备(如磁盘)的缓冲池中选择进程，并装入内存以准备执行。长期调度控制多道程序设计的程度(内存中的进程数量)
  • 短期调度程序(short-term scheduler)或CPU调度：从准备执行的进程中选择集成，并为之分配CPU

中期调度：进程可以从内存中被换出

进程操作

  • 进程创建：通过创建进程系统调用
  • 根据唯一的进程标识符pid来识别进程
  • 进程树，顶端是标识符为0的进程
  • 创建新进程后的两种执行可能：
    • 父进程与子进程并发执行；
    • 父进程等待，直到某个或全部子进程执行完
  • 新进程的地址空间也有两种可能：
    • 子进程是父进程的复制品(具有与父进程相同的程序和数据)，继承了父进程的地址空间
    • 子进程装入另一个新程序
  • 通过fork()系统调用创建子(新)进程
    • 对于子进程，fork()的返回值都是0
    • 对于父进程，fork()的返回值非0
  • 子进程通过系统调用exec取代进程的内存空间并开始执行，父进程若无任务则用wait()把自己移出就绪队列来等待子进程的终止
  • 当进程完成执行后用系统调用exit()请求操作系统删除自身，进程终止，并返回状态值给父进程(通过wait())，所有进程资源被释放
  • 级联终止：不允许子进程在父进程已终止的情况下存在

进程间通信

  • 进程独立(independent)，进程协作(cooperating)

  • 协助进程：与其他进程共享内存的进程

  • 信息共享、提高运算速度、模块化、方便

  • 进程间通信机制(IPC, inter process communication)

  • 进程间通信的两种基本模式：

    • 共享内存：建立一块供协作进程共享的内存区域，通过共享区域读或写入数据来交换信息
    • 消息传递：通过在协助进程间交换消息来实现通信

  • 消息传递更易于实现，适用于交换少量的数据，不需要避免冲突，内核介入消耗更多时间

  • 内存共享传递更快，但占用更多资源

消息传递系统

  • 进程命名

直接或间接通信

  • 直接通信(对称寻址)：
    • send(P, message) // 发送消息到进程P
    • receive(Q, message) // 接收来自进程Q的消息
    • 要求：一个通信线路只与两个进程有关
  • 直接通信(非对称寻址):
    • send(P, message) // 发送消息到进程P
    • receive(id, message) // 接收来自任何进程的消息，变量id设置成与其通信的进程名称
  • 间接通信(通过邮箱或端口来发送和接收消息):
    • send(A, message) // 发送消息到邮箱A
    • receive(A, message) // 接收来自邮箱A的消息
    • 要求：两个进程要有共享邮箱，每个线路对应一个邮箱
    • 特点：一个进程可以与多个进程想关联
    • 关于多个进程共享一个邮箱时的接受问题：取决于所选择解决方案
      • 一个线路最多只能两个进程相关联
      • 一次最多允许一个进程执行receive()操作
      • 允许系统随意选择一个进程接受消息

同步或异步通信

  • 同步(阻塞):
    • 阻塞send：发送者阻塞，直到消息被接收
    • 阻塞receive：接收者阻塞，直到有消息可用
  • 异步(非阻塞):
    • 非阻塞send：发送者发送消息并继续其他操作
    • 非阻塞receive：接收者接受一个消息，可能没有消息
  • 当发送者和接收者都是阻塞时，产生一个集合点

自动或显式缓冲

  • 零容量(无缓冲)：队列中不能有任何消息等待，必须使用阻塞发送
  • 有限容量(自动缓冲)：根据线路剩余容量决定是否阻塞发送者
  • 无限容量(自动缓冲)：从不阻塞发送者

RPC和RMI的不同：

  • RPC支持子程序编程，只能调用远程的子程序或函数
  • RMI基于对象，支持调用远程对象的方法
  • RPC远程过程的参数是普通数据结构
  • RMI可以将对象作为参数传递给远程方法

第4章 线程

  • 线程是CPU使用的基本单元，它由线程ID、程序计数器、寄存器集合和栈组成。如果进程有多个控制线程，那么它能同时做多个任务
  • 多线程优点
    • 响应度高：即使部分阻塞或执行较冗长的操作，程序能继续执行，从而增加了对用户的响应程度
    • 资源共享：线程默认共享它们所属进程的内存和资源
    • 经济：进程创建所需的内存和资源的分配比较昂贵，线程的创建和切换更为经济，花费的时间也更少
    • 多处理器体系结构的利用：多线程的优点就是充分使用多处理器体系结构，以便进程能并行运行在不同的处理器上(单线程进程只能运行在一个CPU上)

多线程模型

  • 多对一模型
    • 一个内核线程管理多个用户线程
    • 任一时刻只有一个线程能访问内核
    • 多个线程不能并行执行在多处理器上
    • 一个线程阻塞系统调用时，整个进程会被阻塞
  • 一对一模型
    • 每个用户线程映射到一个内核线程
    • 一个线程阻塞系统调用时，其他线程能继续执行
    • 比多对一更好的并发功能
    • 允许多个线程并行地运行在多处理器上
    • 缺点：创建内核线程开销较大，影响性能，且受系统实际支持的线程数限制
  • 多对多模型
    • 可以创建任意多的用户线程
    • 可并发执行
    • 当一个线程执行阻塞系统调用时，内核能调度另一个内核线程来执行
    • 二级模型：允许某个用户线程绑定到内核线程上

线程库

  • 线程库：为程序员提供创建和管理线程的API
  • 两种方法实现：
    • 一：在用户空间中体用一个没有内核支持的库，所有代码和数据结构都存在于用户控件，调用库中的函数是本地函数调用而不是系统调用
    • 二：执行一个由操作系统直接支持的内核级的库，调用库中的一个API函数会导致内核的系统调用

多线程问题

  • fork()和exec()
    • 在多线程程序中，系统调用fork()和exec()的语义有变化
      • 如果一个线程调用fork(), 新进程有复制所有线程，和只有单个线程两种情况
      • 如果一个线程调用exec()，那么exec()参数所指定的程序会替换整个进程，包括所有线程
  • 线程取消(thread cancellation)
    • 要取消的线程称为目标线程
    • 目标线程的取消可在两种情况下发生：
      • 异步取消(asynchronous cancellation)：一个线程立即终止目标线程
      • 延迟取消(deferred cancellation)：目标线程不断地检查它是否应终止，这允许目标线程有机会以有序方式来终止自己
  • 信号处理
    • 信号用来通知进程某个特定事件已发生
      • 信号是由特定事件的发生所产生的
      • 产生的信号要发送到进程
      • 一旦发送，信号必须加以处理
    • 同步信号发送到执行操作而产生信号的同一进程
    • 异步信号被发送到另一个进程(信号由运行进程之外的产生，由运行进程异步接收这一信号)
    • 信号的处理程序
      • 默认信号处理程序
      • 用户定义的信号处理程序
    • 信号发送的对象
      • 信号所引用的线程
      • 进程内的每个线程
      • 进程内的某些固定线程
      • 规定一个特定线程以接收进程的所有信号
  • 线程池：无限制的线程会耗尽系统资源，如CPU时间和内存，线程池在进程开始时创建一定数量的线程，并放入到池中等待工作。
    • 优点
      • 通常用现有线程处理请求要比等待创建新的线程快
      • 线程池限制了在任何时候可用线程的数量，这对那些不能支持大量并发线程的系统非常重要
  • 线程特定数据：每个线程独有的数据副本
  • 调度程序激活
    • LWP轻量级进程：实现多对多模型时在用户和内核线程之间设置的一个中间数据结构
    • 解决用户线程库与内核间的通信的方法：调度器激活
    • upcall

第5章 CPU调度

CPU调度：操作系统利用等待时间、空闲时间，将CPU使用权交给其他进程使用，减少等待时间的浪费

  • 进程执行由CPU执行和I/O等待周期组成

  • 进程选择由短期调度程序或CPU调度程序执行

    当CPU空闲时，操作系统就必须从就绪队列中选择一个进程来执行

  • CPU调度发生的四个时间：

    • 当一个进程从运行状态切换到等待状态(例如，I/O请求，或者调用wait等待一个子进程终止)
    • 当一个进程从运行状态切换到就绪状态(例如，出现中断)
    • 当一个进程从等待状态切换到就绪状态(例如,I/O完成)
    • 当一个进程终止时
    • 1、4两种情况只有调度，没有选择；2、3两种情况可以进行选择。
    • 当调度值只能发生在1、4时，调度方案是非抢占的(non-preemptive)或协作的
    • 否则调度方案是抢占的(preemptive)

  • 分派程序(dispatcher)：是一个模块，用来将CPU的控制交给由短期调度程序选择的进程，功能包括：

    • 切换上下文
    • 切换到用户模式
    • 跳转到用户程序的合适位置，以重新启动程序

  • 分派延迟(dispatch latency)：分派程序停止一个进程而启动另一个所要花费的时间

调度准则

  • 用于比较CPU调度算法的特征
    • CPU使用率
    • 吞吐量：一个时间单元内所完成进程的数量
    • 周转时间：从进程提交到进程完成的时间
    • 等待时间：进程在就绪队列等待所花费时间
    • 响应时间：从提交请求到产生第一响应的时间(公平原则)

调度算法

FCFS-先到先服务调度

(First-Come, First-Served)

  • 先请求先分配到CPU
  • 用FIFO队列实现，进程进入就绪队列，其PCB链接到队列的尾部，从队头的进程开始分配
  • 缺点
    • 平均等待时间较长
    • 非抢占式，I/O多的大进程会让CPU和设备使用率变得很低，对于分时系统尤其不适用

SJF-最短作业优先调度

(Shortest-Job-First Scheduling)

  • 更适当的表示：最短下一个CPU区间算法
    • 这是因为调度检查进程的下一个CPU区间的长度，而不是其总长度(因为一个进程有很多个CPU区间和I/O区间组成)
    • 相同时间按FCFS
  • 最佳的调度算法，平均等待时间最小
  • 缺点：
    • 难以知道下一个CPU区间的长度
    • 不能在短期CPU调度层次上加以实现
  • 近似SJF调度，根据(该进程)以前的CPU区间的长度来估计下一个CPU区间的预测值
  • 抢占式SJF算法：最短剩余时间优先调度
    • 当一个进程到达就绪队列而以前进程正在执行，就需要选择

优先级调度

(Priority Scheduling)

  • 每个进程都有一个优先级与其关联，最高优先级的进程会被分配到CPU，相同优先级的进程按FCFS顺序调度
    • SJF算法属于优先级调度的一个特例，简单优先级算法，优先级(p)为下一个(预测的)CPU区间的倒数
  • 以大数字或小数字为高优先级没有定论
  • 主要问题：
    • 无穷阻塞(indefinite blocking)或饥饿(starvation)
    • 解决办法：老化(aging)，随等待时间逐渐增加优先级

RR - 轮转法调度

(round-robin)

专门为分时系统设计，类似于FCFS调度，但是增加了抢占以切换进程，

  • 时间片：每间隔一定时间就切换进程，从就绪队列中拿出第一个进程(就绪队列作为循环队列，时间片结束的进程放回到队列尾部)
  • CPU区间短于时间片的进程，在终止后进程本身会自动释放CPU；长与时间片的进程则会被定时器中断，进行上下文切换，将进程加入到就绪队列的尾部
  • 缺点：平均等待时间通常较长
  • 性能依赖于时间片大小，若时间片很大，则无异于FCFS；若时间片很小，则RR算法成为处理器共享
  • 周转时间也依赖于时间片大小

多级队列调度

(Multilevel Queue Scheduling)

  • 将就绪队列分成多个独立队列，根据进程的属性，如内存大小、进程优先级、进程类型，一个进程被永久地分配到一个队列
  • 每个队列有自己的调度算法
  • 另外，队列之间要有调度，通常采用固定优先级抢占调度

多级反馈队列调度

(Multilevel Feedback-Queue Scheduling)

  • 允许进程在队列之间移动
  • 根据不同CPU区间的特点以区分进程
  • CPU时间长的放到优先级低的队列，I/O约束和交互进程放在搞优先级的队列
  • 多级反馈队列调度程序的参数
    • 每个队列的调度算法
    • 用以确定何时升级到更高优先级队列的方法
    • 用以确定何时降级到更低优先级队列的方法
    • 用以确定进程在需要服务时进入哪个队列的方法

多处理器调度

  • 负载分配(load sharing)成为可能，调度问题也更复杂
  • CPU调度算法同单处理器一样，没有最好的解决办法
  • 多处理器调度的方法
    • 对称多处理(SMP)
  • 处理器亲和性：努力使一个进程在同一处理器上运行
  • 负载平衡：设法将工作负载平均地分配到SMP系统中的所有处理器上
  • 对称多线程：超线程技术

线程调度

  • 局部调度：用户线程->LWP
  • 全局调度：内核线程->CPU

第6章 进程同步

竞争条件(race condition)：多个进程并发访问和操作同一数据且执行结果与访问发生的特定顺序有关

进程同步(process synchronization)

协调(coordination)

临界区问题

  • 临界区：进程中修改共同变量或其他共享数据的代码段
  • 进入区：每个进程必须请求允许才能进入临界区，实现这一请求的代码段称为进入区
  • 退出区：临界区之后有退出区
  • 剩余区：剩余其他代码为剩余区
  • 抢占内核和非抢占内核
    • 非抢占内核不会导致竞争条件
    • 抢占内核需要考虑竞争条件

三项要求

  • 互斥(mutual exclusion)：如果进程在其临界区内执行，那么其他进程都不能在其临界区内执行 (无空等待)
  • 前进(progress)：如果没有进程在其临界区内执行，且有进程需要进入临界区，那么只有那些不在剩余区内执行的进程可参与选择，以确定谁能下一个进入临界区，且这种选择不能无限推迟 (有空让进)
  • 有限等待(bounded waiting)：从一个进程做出进入临界区的请求，直到该请求允许为止，其他进程允许进入其临界区的次数有上限

Peterson算法

  • Peterson算法适用于两个进程在剩余区和临界区交替执行
  • 两个共享数据项

  • 变量turn表示哪个进程可以进入其临界区，即如果turn == i，那么进程 Pi 允许在其临界区内执行

  • 数组flag表示哪个进程想要进入其临界区，如果flag[i]为true，即进程 Pi 想要进入其临界区

  • 当只有两个进程时，用 Pi 表示当前进程， Pj 表示另一个进程，j = 1 - i；

硬件同步

  • 用锁来避免竞争条件
  • 特殊硬件指令可以原子地(不被中断)进行

信号量

  • 用于解决临界区问题

信号量S是个整数变量，除了初始化外，只能通过两个标准原子操作来wait()和signal()来访问

  • 一个进程修改信号量时，不能有其他进程修改同一信号量的值，在wait(S)对S的整型值的测试和修改时也不被中断
  • 计数信号量的值域不受限制，而二进制信号量mutex的值只能为0或1，也称为互斥锁，因为它们可以提供互斥
  • 使用信号量的互斥实现
  • 也可用信号量synch来解决同步问题，令synch初始化为0
  • P2只有在P1调用signal(synch)之后，才会执行S2，因为synch初始化为0
  • 缺点：忙等待
  • 这种类型的信号量也称自旋锁(spinlock)
    • 自旋锁的优点：在等待锁时不进行上下文切换，而上下文切换可能需要花费相当长的时间
  • 解决办法：当信号量值不为正时，该进程不是忙等待而是阻塞自己，将该进程放入到与信号量有关的等待队列中，并将该进程的状态切换为等待，控制转到CPU调度程序，以选择另一个进程来执行。
  • 进程的重新执行通过wakeup()操作来进行
  • block()和wakeup()是基本系统调用
  • 死锁deadlock：两个或多个进程无限地等待一个事件，而该事件只能由这些等待进程之一来产生
  • 无限期阻塞indefinite blocking 或饥饿starvation 即进程在信号量内无限期等待

第7章 死锁

  • 申请：如果申请不能立即被允许，那么申请进程必须等待，直到它获得该资源为止。
  • 使用：进程对资源进行操作
  • 释放：进程释放资源

死锁特征

  • 必要条件，同时满足4个条件会引起死锁
    • 互斥：至少有一个资源必须处于非共享模式，即一次只有一个进程。如果另一个进程申请该资源，那么申请进程必须等到该资源被释放为止
    • 占有并等待：一个进程必须占有至少一个资源，并等待另一个资源，而该资源为其他进程所占有
    • 非抢占：资源不能被抢占，即资源只能在进程完成任务后自动释放。
    • 循环等待：有一组等待进程，进程间相互等待其他进程占有的资源，等待的进程构成了一个循环

资源分配图

  • 由一个节点集合V和一个边集合E组成，节点集合V可分成两种类型的结点：P={P1，P2，…，Pn}(系统活动进程的结合)和R={R1，R2，…，Rn}（系统所有资源类型的集合)
  • 申请边：由进程Pi到资源Rj的有向边记为Pi->Rj，它表示进程Pi已经申请了资源类型Rj的一个实例，并正在等待该资源
  • 分配边：由资源类型Rj到进程Pi的有向边记为Rj->Pi，它表示资源类型Rj的一个实例已经分配给进程Pi，并被该进程占有
  • 用圆形来表示进程，用矩形来表示资源
  • 资源可能有多个实例，用矩形中的原点表示实例
    • 如果没有环，那么没有进程死锁，
    • 如果有环，就可能有死锁
    • 如果环涉及到的资源都只有一个实例，就有死锁，这种情况下环是死锁的充分条件
    • 如果环设计到的资源不止一个实例，则不一定有死锁，这种情况下环是死锁的必要条件

死锁处理方法

三种方法:

  • 可使用协议以预防或避免死锁，确保系统不会进入死锁状态
  • 可允许系统进入死锁状态，然后检测它，并加以修复
  • 可忽视这个问题，认为死锁不可能在系统内发生

解决方案：

  • 死锁预防(deadlock prevention)：确保至少一个必要条件不成立
  • 死锁避免(deadlock avoidance)：操作系统实现得到有关进程申请资源和使用资源的额外信息，判断是否让进程等待，允许当前申请或是延迟

死锁预防

缺点：低设备使用率和系统吞吐率

  • 互斥：通常不能通过否定互斥条件来预防死锁，有的资源本身就是非共享的
  • 占有并等待
    • 解决方案一：申请到资源再执行，破坏等待条件
    • 解决方案二：必须没有占有资源才能申请资源，破坏占有条件
  • 非抢占：解决：如果一个进程占有资源并申请另一个立即分配的资源，那么其现已分配的资源都可被抢占。(即资源被隐式释放，被占有的资源变得可抢占，等待中的进程其占有的资源也可被抢占)
  • 循环等待：一个解决办法是对所有资源类型进行完全排序，且要求每个进程按递增顺序来申请资源

死锁避免

用算法动态地检测资源分配状态以确保循环等待条件不可能成立

  • 安全序列：进程顺序<P1,P2,…,Pn> 如果对于每个进程仍然可以申请的资源数小于当前可用资源加上所有Pj(其中j<i)所占有的资源，则这一顺序成为安全序列
  • 在这种情况下，即使进程Pi所需要的资源不能立即使用，那么等待直到所有Pj释放其资源即可。
  • 不是所有的不安全状态都能导致死锁状态，所有的死锁状态都是不安全状态，安全状态就不会产生死锁

  • Available：长度为m的向量，表示每种资源的现有实例数量，如果Available[j] = k，那么资源类型 Rj 现有k个实例
  • Max：n * m矩阵，定义每个进程的最大需求。如果Max【i】【j】= k，那么进程 Pi 最多可申请k个资源类型 Rj 的实例
  • Allocation：n * m 矩阵，定义每个进程现在所分配的各种资源类型的实例数量，如果Allocation【i】【j】= k，那么进程 Pi 现在已分配了 k 个资源类型 Rj 的实例
  • Need：n * m矩阵，表示每个进程还需要的神域的资源。Need【i】【j】= Max【i】【j】 - Allocation【i】【j】

安全性算法

（1） 设 Work 和 Finish 分别为长度为 m 和 n 的向量。按如下方式进行初始化：Work = Available 且对于 i = 0, 1, …, n-1, Finish[i] = false;

（2）查找这样的 i 使其满足

  • Finish[i] = false
  • Need_i <= Work
    • 如果没有这样的 i 存在，那么就转到第4步

（3）Work = Work + Allocation_i

  • Finish[i] = true
  • 返回到第2步

（4）如果对所有 i ，Finish[i] = true，那么系统处于安全状态

这个算法可能需要 m * n^2 数量级的操作以确定系统状态是否安全

资源请求算法

设Request_i为进程Pi的请求向量，如果Request_i[j] = k，那么进程 Pi 需要资源类型 Rj 的实例数量为k。当进程 Pi 作出资源请求时，采取如下动作：

（1）如果Request_i <= Need_i，那么转到第2步。否则，产生出错条件，这是因为进程 Pi 已超过了其最大请求。

（2）如果Request_i <= Available， 那么转到第3步，否则，Pi 必须等待， 这是因为没有可用资源

（3）假定系统可以分配给进程 Pi 所请求的资源，并按如下方式修改状态：

Available = Available - Request_i；

Allocation_i = Allocation_i + Request_i；

Need_i = Need_i - Request_i；

如果所产生的资源分配状态是安全的，那么交易完成且进程 Pi 可分配到其所需要资源。然而，如果新状态不安全，那么进程 Pi 必须等待 Request_i 并恢复到原来资源分配状态。

第8章 内存管理

CPU访问内存通常需要暂停(stall)，由于内存访问频繁，解决方法是在CPU与内存之间增加告诉缓存(cache)

  • 基地址寄存器：含有最小的合法物理内存地址
  • 界限地址寄存器：决定了范围的大小
  • 输入队列：在磁盘上等待调入内存以便执行的进程形成输入队列
  • 地址绑定：编译器将（符号地址）源程序中的地址绑定在可重定位的地址
    • 编译时：如果在编译时就知道进程将在内存中的驻留地址，那么就可以生成绝对代码
    • 加载时：如果编译时并不知道进程将驻留在内存的什么地方，那么编译器就必须生成可重定位代码，最后绑定会延迟到加载时才进行
    • 执行时：如果进程在执行时可以从一个内存段移到另一个内存段，那么绑定必须延迟到执行时才进行
  • 逻辑地址空间与物理地址空间
    • 逻辑地址：CPU所生成的地址（通常称为虚拟地址）
    • 物理地址：内存单元所看到的地址（即加载到内存地址寄存器中的地址）
  • 逻辑地址空间：由程序所生成的所有逻辑地址的集合
  • 物理地址空间：与这些逻辑地址相对应的所有物理地址的结合称为物理地址空间
  • 内存管理单元：运行时从虚拟地址到物理地址的映射是由称为内存管理单元的硬件设备来完成

  • 重定位寄存器：基地址寄存器

  • 动态加载：一个子程序只有在调用时才被加载。所有子程序都以可重定位的形式保存在磁盘上

  • 静态重定位：在程序执行之前进行重定位，根据装配模块将要装入的内存起始地址，直接修改装配模块中的有关使用地址的指令
    • 重定位项：在程序中需要修改的位置成为重定位项
    • 重定位因子：程序装入内存中的起始地址
    • 重定位项表：为了支持静态重定位，连接程序在生成统一地址空间和装配
  • 动态重定位：在程序执行过程中进行地址重定位，即在每次访问内存单元前才进行地址变换
  • 静态链接：在生成可执行文件时进行的
  • 动态链接：在装入或运行时进行链接
  • 通常被连接的共享代码成为动态链接库
  • 动态链接的优缺点
    • 优点：共享DLL，部分装入，便于局部代码修改，便于运行环境适应
    • 缺点：链接开销，管理开销

内存分配

  • 动态存储分配：可变分区
  • 孔：可用于用户进程的一大块内存
  • 从一组可用孔中选择一个空闲孔的方法：
    • 首次适应(first-fit)：分配第一个足够大的孔，查找可以从头开始，也可以从上次首次适应结束时开始。
    • 最佳适应(best-fit)：分配最小的足够大的孔。必须查找整个表，除非列表按大小排序。
    • 最差适应(worst-fit)：分配最大的孔。
  • 首次适应和最佳适应在空间利用方面较好
  • 首次适应在执行时间上最快
  • 最差适应在空间、时间上都最差

碎片

  • 首次适应方法和最佳适应方法都有外部碎片问题：空闲内存空间被分为小片段，但并不连续

  • 紧缩：移动内存内容，把所有空闲空间合并成一整块。

第9章 虚拟内存

  • 虚拟内存（virtual memory）：在主存中可只装入最近经常要访问的某些区域的指令和数据，剩余部分就暂时不必装入，等到以后要访问到它们时再调入内存。
  • 如果主存较紧张，必要时可将已不大访问的信息调出内存，再执行调入操作。
  • 由于作业的指令和数据可以存放在外存只有操作系统可以修改中，用户的程序就不受实际内存大小的限制，好像计算机系统向用户系统提供了容量极大的“主存”
  • 这个大容量的“主存”是靠存储管理的软件和硬件通过大容量的辅存作为后援存储器扩充而获得的，是程序设计员感觉到的，而实际上并不存在的存储器，故称虚拟存储器

按需调页

  • 只有程序执行需要时才载入页，那些从未访问的也不会调入到物理内存
  • 懒惰交换：只有在需要页时，才将它调入内存
  • 调页程序
  • 缺页中断：检查进程页表，确定引用是否合法；非法引用则终止进程，引用有效但未调入内存则调入；找到空闲帧；调度磁盘，将页调入空闲帧；修改进程内部表和页表，表示已经调入；重新开始因陷阱而中断的指令

有效访问时间=（1-页错误率） 内存访问时间 + 页错误率 页错误处理时间

  • 三个主要的页错误处理时间，中断处理，读入页和重启进程。
  • 纯粹按需调页：只有在需要时才将页调入内存
  • 页表：该表能够通过有效-无效位或保护位的特定值，将条目设为无效
  • 次级存储器：该次级存储器用来保存不在内存中的页。次级存储器通常为快速磁盘，它通常称为交换设备。用来交换的这部分磁盘称为交换空间
  • 写时复制：允许父进程与子进程开始时共享同一页面。如果任何一个进程需要对页进行写操作，就创建一个共享页的副本

页面置换

  • 基本页置换：如果没有空闲帧，那么就查找当前没有使用的帧，并将其释放。
  • 释放的方式可以是：将其内容写到交换空间，并改变页表以表示该页不再内存中。
  • 包含页置换的页错误处理程序：
    • 查找所需页在磁盘上的位置
    • 查找一个空闲帧
      • 如果有空闲帧，那么就使用它
      • 如果没有空闲帧，那么就使用页置换算法以选择一个“牺牲”帧
      • 将“牺牲”帧的内容写到磁盘上，改变页表和帧表
    • 将所需页读入（新）空闲帧，改变页表和帧表
    • 重启用户进程

FIFO页置换

  • 记录每个页调入内存的时间
  • 选择最旧的页替换，可以不用准确记录而是用一个FIFO队列来管理内存中的页
  • 性能不佳，Belady异常：对于有些置换算法，页错误率会随着分配帧增加而增加

最优置换

最优页置换算法：置换最长时间不会使用的页

  • 产生页错误率最低，绝没有Belady异常的问题
  • 但无法知道后续的序列所以难以实现，需要预测最长时间不会使用的页

LRU页置换

最近最少使用算法：对于已有帧的页码，替换当距离当前页最远（最前）的页

  • 用计数器记录使用时间域
  • 用栈记录，每次使用就从栈中移除并放到顶部
  • 不会有Belady异常，和最优置换一样都是栈算法
  • 需要硬件支持

近似LRU页置换

引用位：每当引用一个页时（无论是对页的字节进行读或写），相应页表的引用位就被硬件置位。开始时清零，用过的置为1，不能知道使用顺序，只能知道哪些页用过

  • 二次机会算法，在FIFO页中拿到一个引用位为0的页就置换，为1的就将引用位清零并给第二次机会，选择下一个FIFO页
  • 增强型二次机会算法：引用位和修改位，使用率从00-01-10-11依次增加

基于计数的页置换

  • 最不经常使用页置换算法（LFU）：置换计数最小的页
  • 最常使用页置换算法（MFU）：置换计数最大的页

帧分配

帧分配算法：

  • 平均分配
  • 根据进程大小，使用比例分配
  • 全局置换：全局置换允许一个进程从所有帧集合中选择一个置换帧，而不管该帧是否已分配给其他进程，即一个进程可以从另一个进程中拿到帧
  • 局部置换要求每个进程仅从自己的分配帧中进行选择

系统颠簸

颠簸（thrashing）：频繁的页调度行为，如果一个进程在换页上用的时间比执行时间多，那么这个进程在颠簸

  • CPU 处理多道程序时，内存没有足够的帧供多个进程同时运行，导致需要不断置换页，连续产生页错误，使页置换时间大于执行时间
  • 局部模型：局部是一个经常使用页的集合，一个程序通常由多个不同局部组成，它们可能重叠
  • 工作集模型（working-set model）：基于局部性假设的用于研究一个进程实际正使用多少帧的模型，定义一个△表示最近使用的△个页的集合作为工作集

  • 每个文件占用连续的内存单元，优点是访问简单而且支持直接访问，缺点是会产生动态内存分配问题，即碎片，以及文件的可扩展性太差。

链接分配（linked）

  • 用链表连接起文件的各个块。优点是没有外部碎片，只要有空闲快就能够扩展文件，而不需要合并磁盘空间；缺点是不支持直接访问且指针需要占用空间，指针的可靠性不强。

索引分配（indexed）

  • 将连接分配中每个文件的所有指针放在一起组成索引块，类似于页表。优点是支持直接访问且没有外部碎片问题，但是需要索引块需要内存空间，且比链接分配的指针效率要低

空闲空间管理

位向量（bit vector）

  • 每个块都用一个二进制位表示其是否空闲，有多少块就位图就有多大。

链表（linked list）

  • 将所有空闲快用链表连接。

组和计数（group&count）

网络文件系统（NFS）

  • NFS 允许一个系统在网络上与他人共享目录和文件。通过使用 NFS，用户和程序可以像访问本地文件一样访问远端系统上的文件。优点在于本地工作站使用更少磁盘空间，因为数据可以通过网络在别的机器上访问；不必再每个用户机器建立一个 home目录

第12章 磁盘调度

FCFS调度

  • 先来先服务，公平
  • 不能提供最快速度
  • 访问空间的跨度很大

SSTF调度

最短寻道时间优先算法

  • 可以大大减少寻道时间，但是由于请求是随时到达的，所以可能有些请求一直得不到服务

SCAN调度

电梯算法

  • 从第一个服务开始先向正方向扫描，处理完路上所有请求，然后反向扫描处理剩下的请求

C-SCAN调度

  • 提供更均匀的等待时间，当磁头扫描到尾部的时候立即返回磁盘起始点再扫描。

LOOK调度

  • 只移动到一个方向上的最远请求的SCAN

C-LOOK调度

  • 只移动到一个方向上的最远请求的C-SCAN

磁盘管理

  • 磁盘格式化
    • 低级格式化（物理格式化）：磁盘分成扇区，每个扇区的数据结构由头数据区域和尾部组成。头部和尾部包含了一些磁盘控制器所使用的信息：纠错代码ECC（判断扇区是否损坏及用于找到出错数据）
    • 逻辑格式化：创建文件系统，将初始文件系统数据结构存储到磁盘上，包括空闲和已分配的空间（FAT或inode）和一个初始为空的目录
    • 簇（cluster）：由多个块集中而成的一个大块
  • 引导块:
    • 引导块放在磁盘的第一个扇区（引导扇区）MBR，包含引导代码和分区表
  • 坏块：
    • 控制器维护一个磁盘坏块链表，该链表在出厂前进行低级格式化时就已经初始化了，并在磁盘整个使用过程中不断更新
    • 扇区备用（sector sparing）或转寄（forwarding）：用操作系统看不到的备用块，由控制器用它来逻辑地替代坏块

RAID结构

磁盘冗余阵列（RAID），用于提高性能和可靠性

  • RAID级别：

    （C 表示 数据的第二副本，P表示差错纠正位）

    • RAID 0：无冗余的磁盘阵列
  • RAID 1：磁盘镜像

第13章 I/O输入系统

  • 端口（port）：CPU 通过接口寄存器或特定电路与外设进行数据传送，这些寄存器或特定电路称之为端口

  • 总线（bus）：总线是一种内部结构，它是 cpu、内存、输入、输出设备传递信息的公用通道，主机的各个部件通过总线相连接，外部设备通过相应的接口电路再与总线相连接，计算机各种功能部件之间传送信息的公共通信干线

  • 控制器（controller）：用于操作端口、总线或设备的一组电子器件。按照预定顺序改变主电路或控制电路的接线和改变电路中电阻值来控制电动机的启动、调速、制动和反向的主令装置

  • I/O端口通常由4种寄存器：状态寄存器、控制寄存器、数据输入寄存器与数据输出寄存器
  • 轮询（polling）或忙等待：在该循环中，不断地抽取状态寄存器直到忙位被清除
  • 中断（interrupt）能使外设通知CPU的硬件机制
    • 机制：CPU硬件有一条中断请求线（IRL），CPU在执行完每条指令后，都将检测IRL，当检测到已经有控制器通过IRL发送了信号，CPU将保存当前状态并且跳转到内存固定位置的中断处理程序（interrupt-controller）
      • 进行关键处理时：能够延迟中断处理
      • 更为有效地讲中断分发到合适的中断处理程序，而不是检查所有设备以决定哪个设备引起中断
      • 需要多级中断，根据优先级响应紧迫性更高的中断
  • 非屏蔽中断，可屏蔽中断
  • 软中断或陷阱指令：有一个参数用来标识所需的内核服务
  • I/O应用接口：从I/O设备中抽象出一些通用类型，每个通用类型都可以通过一组标准函数（即接口）来访问
  • DST设备状态表：设备类型，地址和状态
  • 缓冲区：是用来保存两个设备之间或在设备和应用程序之间所传输数据的内存区域
  • 高速缓存（cache）可以保留数据副本的高速存储器
  • 假脱机（spooling）用来保存设备输出的缓冲区（如打印机，不能接收交叉的数据流，一次只能打印一个任务）
  • 为了防止用户执行非法I/O，定义所有I/O指令为特权指令，不能由用户程序发出，必须通过操作系统来进行，用户执行系统调用请求执行

三. 云计算概念

​ 云计算是作为一种服务提供的一种特殊形式的分布式计算，引入效用模型远程地通过网络将可拓展、可测量的资源或软件和信息共享提供给计算机和其他设备。

​ 云计算=数据 (软件 + 平台 + 基础设施) 服务

提供

数据（Data） 传感器、物联网 、原始数据、半结构化数据和经过处理的结构化数据

  • 软件（Software） 检索、发现、关联、处理和创造数据
  • 平台（Platform）： 软件开发API、环境和工具
  • 基础设施（Infrastructure） 存储资源、计算资源、网络资源等
  • 服务（Service） IT服务化：产品、服务 XaaS

商业驱动力

  • 容量规划
  • 领先策略(Lead Strategy)：根据预期增加IT资源的容量
  • 滞后策略(Lag Strategy)：当IT资源达到其最大容量时增加资源容量
  • 匹配策略(Match Strategy)：当需求增加时，小幅增加IT资源容量
  • 购置和维护成本
  • 组织灵活性

技术创新

  • 集群化
  • 网格计算
  • 虚拟化

云

云（cloud）是指一个独特的IT环境，其设计目的是为了远程供给可扩展和可测量的IT资源。

IT资源（IT resource）是指一个与IT相关的物理的或虚拟的事物，它既可以是基于软件的，比如虚拟服务器或定制软件程序，也可以是基于硬件的，比如物理服务器或网络设备。

​ e.g. 物理服务器，虚拟服务器，软件程序，云服务，存储设备，网络

  • 可拓展性: IT资源可以处理增加或减少的使用需求的能力
  • 水平拓展: 向外（内），增加IT资源的数量
  • 垂直拓展: 向上（下），增加IT资源的容量

云服务

云服务（cloud service）是指任何可以通过云远程访问的IT资源。

四. 基本概念与模型

云提供者

提供基于云的IT资源的一方称为云提供者

云用户

使用基于云的IT资源的一方称为云用户，云用户使用云服务用户来访问云服务

云服务用户

云服务用户（cloud service consumer）一个临时的运行时角色，访问云服务的软件程序

云服务拥有者

在法律上拥有云服务的个人或者组织称为云服务拥有者，可以是云用户也可以是该云服务所在的云的云提供者（在云中部署了自己的服务的对象就是云服务拥有者，关键看云中的服务是谁部署的）

云资源管理者

可以是来自云用户或者云提供者，也可以第三方组织，就是签订了合约来管理基于云的IT资源的组织（不称为云服务管理者的原因是这个角色可能要管理不以云服务形式存在但又基于云的IT资源）

了解：云审计者、云代理、云运营商

组织边界

​ 组织边界是一个物理范围，包括由一家组织拥有和管理的IT资源的集合。

信任边界

​ 信任边界是一个逻辑范围，通常会跨越物理边界，表明IT资源受信任的程度

云的特性

  • 按需使用

  • 随处访问

  • 多租户

  • 弹性

  • 可测量的使用

  • 可恢复性

云交付模型

基础设施作为服务（IaaS）

  • 硬件资源作为服务提供给用户；
  • 资源通常是未配置好的，管理的责任直接落在云用户身上；
  • 典型例子：虚拟服务器；
  • 主要技术：虚拟化技术，资源动态管理和调度技术；

平台作为服务（PaaS）

  • 预先定义好的“就绪可用”的环境，已经部署好和配置好的IT资源；
  • 典型例子：运行平台如应用服务器，辅助系统软件如数据库；
  • 关键技术：分布式文件系统、分布式数据库，并行计算技术；
  • 好处：云用户省去了建立和维护裸的基础设施IT资源的管理负担，相对地给予云用户的控制权等级较低，不能访问平台的实现细节；

软件作为服务（SaaS）

  • 把软件程序定位成共享的云服务，将运行于云中的应用程序的功能交付给用户；

用户的管理权限十分有限，云服务用户可以访问玉衡服务，但是不能访问任何底层的IT资源或实现细节；

  • 关键技术：呈现技术，多租户技术；
  • 应用场景：通常是商用，如saleforce的CRM

三种云交付模型的比较

  • 云用户的控制等级：SaaS < PaaS < IaaS
  • 云用户的可用功能：前端用户接口访问 < 对使用平台相关的IT资源的管理控制 < 对基础设施相关的IT资源和可能的底层物理IT资源的完全访问
  • 云用户的行为：使用和配置云服务 < 开发、测试、部署、管理云服务 < 建立和配置裸的基础设施，安装、管理和监控所需的软件
  • 云提供者的行为：实现、管理和维护云服务，监控云用户的使用 < 实现配置好的平台和在需要时提供底层的基础设施中间件和其他所需的IT资源，监控云用户的使用 < 提供和管理需要的物理处理器、存储、网络和托管监控云用户的使用

云交付模型的组合使用

  • IaaS + PaaS

例子：一个提供PaaS环境的云提供者选择租用另一个不同的云提供者提供的IaaS环境; 原因可能是经济因素，也可能是原有的云提供者还服务者其他的云用户而它现有的容量快不够用了，或者其他原因是它的数据被要求物理地存放在某个特定的区域内。

  • IaaS + PaaS + SaaS

云用户组织可以利用这样的模型组合来开发部署自己的SaaS云服务

云部署模型

公有云

社区云

​ 限制为特定的云用户社区可以访问的云环境

私有云

​ 一家组织单独拥有的，可以集中访问不同部分、位置或部门的IT资源

企业内部分工由单独的部门来承担云提供者的角色，区分“企业内部的”和“基于云的”概念很重要，因为组织和信任边界的定义和应用被改变

混合云

由两个或多个不同的云部署模型组成的云环境，可以将数据按敏感程度分别部署到公有云或私有云上。

缺点：架构的构建和维护复杂和具有挑战性

      • 每个操作系统需要独立的许可证

HDFS适应的场景

  • HDFS是针对MapReduce设计
    • 数据尽可能根据其本地局部性进行访问与计算
  • 大量地从文件中顺序读
    • HDFS对顺序读进行了优化

  • 数据支持一次写入，多次读取
    • 不支持数据更新（但可以直接进行文件替换）
  • 数据不进行本地缓存
    • 文件很大，且顺序读

HDFS的设计

  • 副本的默认数目是3
  • 默认的块的大小是64MB
    • 减少元数据的量
    • 有利于顺序读写（在磁盘上数据顺序存放）

CAP 定理

共享数据的三个性质：

  • 一致性（Consistency）：所有的副本的值相同
  • 可用性（Availability）：总能进行读写
  • 分区容错性（Partition-tolerance）：即使网络故障妨碍了一些设备之间的通信，一致性和/或可用性仍然有效

布鲁尔的CAP定理

  • 对于任何系统共享数据，不能保证同时满足三个性质
  • 最多有两个性质
  • 传统的数据库管理系统更注重”C”

云系统，分区是不可避免的

  • “C”和”A”都可以从中选择
  • 几乎在所有情况下都会选择”A”

Word Count

Map
  • 接受输入键值/对
  • 发出中间键值/对
Reduce
  • 接受中间键值/对
  • 发出输出键值/对
InputSplits
  • 定义了输入到单个Map任务的输入数据
  • 一个MapReduce程序被统称为一个Job，可能有上百个任务构成
  • InputSplit将文件分为64MB的大小
    • hadoop-site.xml中的mapred.min.split.size参数控制这个大小
  • mapred.tasktracker.map.taks.maximum用来控制某一个节点上所有map任务的最大数目
RecordReader
  • InputSplit定义了一项工作的大小，但是没有定义如何读取数据
  • RecordReader实际上定义了如何从数据上转化为一个（key，value）对，从而输出到Mapper类中
  • TextInputFormat提供了LineRecordReader
Mapper
  • 来自数据源的记录
    • 从文件中逐行读出、或从数据库的行获得，等
    • 键值对：如（filename，line）
  • map（）函数从输入的数据中产生一个或多个中间值以及输出键
  • 每一个Mapper类的示例生成了一个Java进程
    • 在某一个InputSplit上执行
  • 有两个额外的参数OutputCollector以及Reporter
    • 前者用来收集中间结果
    • 后者用来获得环境参数以及设置当前执行的状态
Reducer
  • 在map阶段结束后，给定输出键的所有中间值都被组合成一个列表
  • reduce()函数将这些中间值组合为同一输出键的一个或多个最终值
Partition&Shuffle
  • 在Map工作完成之后，每一个Map函数会将结果传到对应的Reducer所在的节点
  • 用户可以提供一个Partitioner类，用来决定一个给定的（key，value）对传输的具体位置
Sort
  • 传输到每一个节点上的所有的Reduce函数接收到Key，value对会被Hadoop自动排序（即Map生成的结果传送到某一个节点的时候，会被自动培排序）
  • 默认：hash（key）mod R
  • 保证：
    • 相对平衡的分区
    • 分区内排序保证

六. 基本云安全

  • 保密性：只有被授权方才能访问

  • 完整性：云服务传送过程中数据不被未授权方篡改

  • 真实性：唯一性，真实可靠

  • 可用性：在特定的时间段内可以访问和可以使用

  • 威胁：潜在的安全型违反，可能视图破坏隐私并/或导致危害，以此挑战防护

  • 风险：执行一个行为带来损失或危害的可能性

威胁作用者

  • 匿名攻击者

  • 恶意服务作用者

  • 授信的攻击者（恶意的租户）

  • 恶意的内部人员

云安全威胁

  • 流量窃听：数据在传输到云中或在云内部传输时被恶意的服务作用者被动地截获
    • 目的：破坏数据的保密性，破坏云用户和云提供者之间关系的保密性
    • 特点：由于攻击被动的本质，容易长时间进行而不被发现

  • 恶意媒介：消息被恶意服务作用者截获并且篡改，破坏消息的保密性和完整性

  • 拒绝服务：使IT资源过载至无法正确运行

  • 授权不足：错误地授予了攻击者访问权限或是授权太宽泛，导致攻击者能够访问到本应 该受到保护的IT资源

  • 虚拟攻击：利用虚拟化平台中的漏洞来危害虚拟化平台的保密性。完整性和可用性。

  • 信任边界重叠

七. 云基础设施机制

逻辑网络边界

将一个网络环境与通信网络的其他部分隔离开来，形成了一个虚拟网络边界。

用于：将云中的IT资源与非授权用户或非用户、云用户隔离，控制被隔离IT资源的可用带宽；

虚拟防火墙

一种IT资源，可以主动过滤杯隔离网络的网络流量，并控制其与Internet的交互。

虚拟网络

一般通过VLAN形成，这种IT资源用来隔离数据中心基础设施内的网络环境。

虚拟服务器

模拟物理服务器的虚拟化软件，是最基本的云环境构建块。每个虚拟服务器都可以存储大量的IT资源、基于云的解决方案和各种其他的云计算机制。从映像文件进行虚拟服务器的实例化是一个可以快速且按需完成的资源分配过程。

云存储设备

  • 云存储设备是指专门为基于云配置所设计的存储设备。

  • 支持按使用计费，提供固定增幅的容量分配

  • 主要问题是数据的完整性，保密性，安全性

  • 常见数据存储逻辑单元：文件、块、数据集、对象
  • 三种技术接口：网络存储接口（文件或块）、对象存储接口（Web资源），数据库存储接口（分为关系数据存储和非关系数据存储NoSQL）

云使用监控

​ 三种常见的实现形式：监控代理、资源代理、轮询代理

​ 监控代理是一个中间的事件驱动程序，作为服务代理

​ 资源代理是一种处理模块，轮询代理是一种处理模块

资源复制：对同一个IT资源创建多个实例，加强IT资源的可用性和性能，复制基于云的IT资源

已就绪环境：已就绪环境机制是PaaS云交付模型的定义组件，代表的是预定义的基于云的平台，该平台由一组已安装的IT资源组成，可以云用户使用和定制。云用户可以使用这些环境在云内远程开发和配置自身的服务与应用程序。

典型的已就绪环境包括预安装的IT资源，如数据库、中间件、开发工具等

八. 特殊云机制

自动伸缩监听器

  • 一个服务代理，它监控和追踪云服务用户和云服务之间的通信，用以动态自动伸缩。

  • 通常部署在靠近防火墙的位置，来自动追踪负载状态信息

  • 负载量可以由云用户产生的请求量或某种类型的请求印发的后端处理需求量来决定

  • 对于不同负载波动的条件，自动伸缩监听器可以提供不同类型的响应：

    • 自动伸缩IT资源：根据事先定义的参数

    • 自动通知云用户：负载过高或过低时

负载均衡器

  • 一个运行时代理，主要用于把负载在两个或更多的IT资源上做负载

  • 负载分配功能：

    • 非对称分配
    • 负载优先级
    • 上下文感知的分配

  • 目标：优化IT资源的使用，避免过载并最大化吞吐量

  • 机制：

    • 多层网络交换机
    • 专门的硬件设备
    • 专门的基于软件的系统（在服务器操作系统中比较常见）
    • 服务代理（通常由云管理软件控制）

  • 负载均衡器通常位于产生负载的IT资源和执行负载处理的IT资源之间的通信路径上

  • 负载均衡器可以被设计成一个透明的代理或是一个代理的组件

SLA监控器

  • 被用来专门观察云服务的运行时性能，确保它们履行了SLA中报告的约定的QoS需求
  • SLA监控器收集的数据由SLA管理系统处理并集成到SLA报告的标准中

（PS.详见第九章)

按使用付费监控器

  • 机制，按照预先定义好的定价参数测量基于云的IT资源使用，并生成使用日志用于计算费用
  • 使用数据由计费管理系统进行处理
  • 一些典型的监控变量包括：
    • 请求/响应消息数量
    • 传送的数据量
    • 带宽消耗
  • 实现方式
    • 资源代理
    • 监控代理

审计监控器

  • 用来收集网络和IT资源的审计记录数据，用以满足管理需要或者合同义务

故障转移系统

  • 通过使用现有的集群技术提供冗余的实现来增加IT资源的可靠性和可用性
  • 只要当前活跃的IT资源变得不可用时，便会自动切换到冗余的或待机IT资源实例上
  • 类型：
    • 主动-主动
      • 多个实例都处于活动状态，同时提供服务
      • IT资源的冗余实现和负载均衡器是必须要的
    • 主动-被动
      • 一个处于活动状态，一个待机或闲置
      • 当IT资源变得不可用的时候，就会激活待机实例来接管工作
      • 相应的工作负载就会被重定向到接管操作的这个实例上

虚拟机监控器

  • 虚拟化基础设施的最基本部分，主要用来在物理服务器上生成虚拟服务器实例。
  • 通常受限于一台物理服务器（即一个虚拟机监控器由一台物理服务器来管理）
  • VIM提供了一组特性来管理跨物理服务器的多虚拟机监控器

资源集群

  • 把多个IT资源实例分为一组，使得他们能像一个IT资源那样进行操作
  • 通过高速专用网络链接或者集群结点是实现工作负载、任务调度、数据共享和系统同步等通讯要求
  • 作用：增强了集群化IT资源的组合计算能力、负载均衡能力和可用性
  • 常见的资源集群类型包括：
    • 服务器集群：提高性能和可用性
    • 数据库集群：提高数据可用性，维持数据的一致性
    • 大数据集集群：数据的分区和分布
  • 两种基本类型:
    • 负载均衡的集群：保持集中管理的特性下实现了在集群结点中的分布式工作负载
    • 高可用集群：在多节点失效的情况下保持系统的可用性，需要冗余实现和故障转移机制

多设备代理

  • 用来帮助运行时的数据转换
  • 克服云服务和多样性的云服务用户之间的不兼容性
  • 使得云服务能够被更广泛的云服务用户程序和设备所使用
  • 需要创建映射逻辑来改变运行时交换的信息
  • 通常是作为网关存在的：
    • XML网关
    • 云存储网关
    • 移动设备网关

状态管理数据库

  • 是一种存储设备，用来暂时地持久化软件程序的状态数据，软件程序可以把状态数据卸载到数据库汇总
  • 替代状态数据缓存在内存中的一种办法，用以降低程序占用的运行时的内存量
  • 状态管理数据库使得软降程序和周边的基础设施都具有更大的可拓展性

九. 云管理机制

远程管理系统

  • 远程管理系统机制向外部云资源管理者提供工具和用户界面来配置并管理基于云的IT资源
  • 远程管理系统能够建立一个入口以便访问各种底层系统的控制与管理功能，包括资源管理、SLA管理和计费管理
  • 远程管理系统提供的工具和API一般被云提供者用来开发和定制在线入口，这些入口向云用户提供各种管理控制
  • 主要创建如下两种类型的入口：
    • 使用与管理入口：一种通用入口，集中管理不同的基于云的IT资源，并提供IT资源使用报告
    • 自主服务入口：本质上是一个购买门户，允许云用户搜索云提供者提供的最新云服务和IT资源列表
  • 通过远程管理控制奥体，云用户通常能够执行的任务包括：
    • 配置与建立云服务
    • 为按需云服务提供和释放IT资源
    • 监控云服务的状态、使用和性能
    • 监控QoS和SLA的实行
    • 管理租赁成本与使用费用
    • 管理用户账户、安全凭证、授权和访问控制
    • 跟踪对租赁服务内部与外部的访问
    • 规划与评估IT资源供给
    • 容量规划

资源管理系统

  • 资源管理系统机制帮助协调IT资源，以便相应云用户和云提供者执行的管理操作
  • 此系统的核心是虚拟基础设施管理器（VIM），用于协调服务器硬件，这样就可以从最合适的底层物理服务器创建虚拟服务器实例
  • VIM是一个商业化产品，用于管理一系列跨多个物理服务器的IT资源
  • 通常通过资源管理系统自动化并是吸纳的任务包括：
    • 管理虚拟IT资源模板，如虚拟服务器映像
    • 分配和释放虚拟IT资源，比如开始、暂停、继续和终止
    • 在有其他机制参与的条件下，协调IT资源，比如资源复制、负载均衡和故障转移系统
    • 在云服务实例的生命周期内，强制执行使用策略和安全规定
    • 监控IT资源的操作条件

SLA管理系统

  • SLA管理系统机制代表的是一系列商品化的可用云管理产品，这些产品提供的功能包括：SLA数据的管理、收集、存储、报告以及运行时通知
  • 部署SLA管理系统时，常常会包含一个库，用于存储和检索被收集的基于预定义指标和报告参数的SLA数据

计费管理系统

  • 计费管理系统机制专门用于手机和处理使用数据，它涉及云提供者的结算和云用户的计费
  • 计费管理系统依靠按使用付费监控器来收集运行时使用数据，用于计费、报告和开发票等目的
  • 计费管理系统允许指制定不同的定价规则，还可以针对每个云用户或每个IT资源自定义定价模型
  • 定价模型可以是传统的按使用付费模型，也可以是固定费率或按分配付费模型，还可以是它们的组合
  • 计费是根据使用前支付和使用后支付来安排的

十. 云安全机制

  • 安全属性：
    • 保密性：是指只有被授权方才能访问的特性
    • 完整性：是指未被未授权方篡改的特性
    • 真实性：是指事务是由经过授权的源提供的这一特性
    • 可用性：是在特定的时间段内可以访问和可以使用的特性

加密

  • 加密
    • 是一种数据编码系统，专门用来保护数据的保密性和完整性
    • 用于对抗流量窃听、恶意媒介、授权不足和信任边界重叠这样一些安全威胁
  • 加密部件
    • 加密用的标准化算法，把原始的明文数据转换成加密的密文数据
  • Key - 密钥

加密的类型

  • 基于流的密码
    • 一次一个
    • 将明文与密钥流混合
    • 有利于实时服务
  • 基于块的密码
    • 取代和换位

加密的方法

对称加密

  • 加密和解密的相同密钥
  • 密钥分配问题

非对称加密

  • 基于无法逆向求解的数学问题
    • 整数分解、离散对数、椭圆曲线
  • 通过一个大的随机数产生一对密钥
    • 私钥、公钥

混合加密

  • 结合了两种的优点
  • 非对称分配对称密钥
    • 也称为会话密钥
  • 对称提供批量加密

对称 VS. 非对称

  • 对称加密保证了数据的保密性，但是并不保证数据的不可否认性
  • 非对称加密依赖于使用两个不同的密钥，称为私钥和公钥
    • 用自己的私钥加密给对方，对方用公钥解开
      • 真实性、不可否认性和完整性保护（不提供保密性）
    • 用公钥加密送给私钥拥有者
      • 保证了保密性（不保证完整性、真实性的保护）

哈希

  • 哈希
    • 用来获得消息的哈希代码或者消息摘要，通常是固定的长度，小于原始的消息大小
    • 单向的、不可逆转的数据保护
    • 主要属性：计算简单、不可逆、不重复
  • 典型应用
    • 数据保密：如密码哈希
    • 恶意媒介和授权不足
    • 服务滥用：（如DoS）Proof-of-work
    • 文件标识：P2P文件系统、软件版本管理工具

数字签名

  • 数字签名
    • 将消息进行哈希，然后用私钥进行加密
  • 作用
    • 通过身份验证和不可否认性来提供数据真实性和数据完整性
    • 类似于日常生活中的“签名”
  • 算法
    • RSA是最常用的签名算法
  • 扩展应用
    • 数字证书：个人安全邮件证书
    • 智能卡：将数字签名放入便携物理存储介质

公钥基础设施

  • 公钥基础设施
    • 是一个由协议、数据格式、规则和实施组成的系统
    • 是用于大规模、公共系统的公钥加密技术
  • 数字证书
    • 带数字签名的数据结构，验证证书拥有者身份以及相关信息
    • 通常是由第三方证书颁发机构签发的
  • PKI与云计算
    • 实现非对称加密
    • 管理云用户和云提供者身份信息
    • 防御恶意中介和不充分的授权威胁

身份与访问管理IAM

  • 身份与访问管理
    • 控制和追踪用户身份以及IT资源、环境、系统访问特权
  • 四个主要部分组成：
    • 认证
    • 授权
    • 用户管理
    • 证书管理
  • IAM机制功能：
    • 分配用户特权等级
    • 访问控制和策略
  • IAM机制的用途
    • 对抗授权不足、拒绝服务供给和信任边界重叠等威胁
    • 比PKI机制要更全面

单一登录

  • 单一登录机制
    • 使得一个云服务用户能够被一个安全代理认证并建立起一个安全上下文
    • 当云用户要访问其他云服务或资源的时候，这个上下文会被持久化
  • 实现跨云服务的用户认证和授权
  • 允许互相独立的云服务和IT资源产生并流通运行时认证和授权证书
  • 不直接对抗任何一个云安全威胁
  • 增强了基于云的环境的访问并管理分布式IT资源和解决方案的可用性

基于云的安全组

  • 云资源分割
    • 为不同用户和组创建各自的物理和虚拟IT环境的过程
    • 云资源分割是虚拟化的基础
  • 基于云的安全组
    • 基于云的资源分割，网络被分成逻辑的基于云的安全组，形成逻辑网络边界
    • 运行在同一个物理服务器上的多个虚拟服务器逻辑上可以是不同基于云的安全组的成员
    • 基于云的安全组能限制对资源的未被授权的访问
    • 用来对抗拒绝服务、授权不足和信任边界重叠等威胁

强化的虚拟服务器映像

  • 虚拟服务器镜像
    • 一个从模板配置创建的虚拟服务器镜像
    • 比原始标准映像更加安全的虚拟服务器模板
  • 强化
    • 把不必要的软件从系统中剥离出来。限制可能被攻击者利用的潜在漏洞的过程
    • 去除冗余的程序，关闭不必要的服务器端口，关闭不使用的服务，内部root用户和guest访问等
  • 对抗拒绝服务、授权不足和信任边界重叠等威胁

十一. 基本云架构

负载分布架构

  • 负载分布架构
    • 通过增加一个或多个相同的IT资源可以进行IT资源水平扩展，而提供运行时逻辑的负载均衡器能够在可用IT资源上均衡分配工作负载
    • 以来负载均衡算法和运行时逻辑
  • 分布式IT资源
    • 多个同类型IT资源同时提供服务
    • 水平扩展：增加/减少相同的IT资源
  • 负载均衡
    • 在可用IT资源上均匀分配工作负载
  • 负载均衡器
    • 资源入口处执行负载均衡的实体
    • 调度方法
      • 基于DNS
      • 基于虚拟IP
      • 基于链路聚合：用于整合链路提高网络传输能力
      • 基于应用：用于分配到分布式调度器
    • 调度策略
      • 轮转、负载水平,……
      • 同一用户的多个请求调度到同一服务器
      • 同一租户的请求调度到尽量少的一组服务器
      • 尽量实现不同类型负载的互补
  • 负载分布架构的多个构成部分
    • IT资源
      • 资源副本
      • 资源集群
    • 服务入口
      • 负载均衡器
      • 逻辑网络边界
    • 监控与审计
      • 使用监控器
      • VM监控器
      • 审计监控器

服务负载均衡架构

  • 服务负载架构
    • 可以被认为是工作负载分布架构的一个特殊变种
    • 专门针对扩展云服务实现
  • 基于云服务的冗余部署
    • 部署在虚拟机上或容器中的应用程序或软件
    • 多实例构成资源池
  • 两种形式
    • 独立于云设备及主机服务器
    • 作为应用程序/服务器环境的内置组件
  • 其他机制
    • 云使用监控器
    • 资源集群
    • 资源复制

资源池架构

  • 资源池架构
    • 使用一个或多个资源池为基础，其中相同的IT资源由一个系统进行分组和维护，以确保它们保持同步
  • 常见的资源池有：
    • 物理服务器池
    • 虚拟服务器池
    • 存储池
    • 网络池
    • CPU池
    • 内存池
  • 可以每种类型的IT资源创建专用池，也可以单个或多个池集合成一个更大的池

层次资源池架构

  • 资源池可以建立层次结构，形成父子、兄弟和嵌套关系，利于构成不同的资源池需求

同级资源池

  • 通常来自物理上的同一组IT资源

  • 同级资源池之间是相互隔离的，云用户只能访问各自的资源池

嵌套资源池

  • 较大的资源池被分解成较小的资源池，每个小资源池分别包含了与大资源池相同类型的IT资源

  • 嵌套资源池可以用于向同一个云用户组织的不同部门或者不同组分配资源池

资源池使能技术

  • 云存储设备和虚拟服务器
  • 辅助技术：
    • 审计监控器
    • 云使用监控器
    • 虚拟机监控器
    • 逻辑网络边界
    • 按使用付费监控器
    • 远程管理系统
    • 资源管理系统
    • 资源复制

动态可扩展架构

  • 动态可扩展架构
    • 基于预先定义的扩展条件从资源池中动态分配IT资源
    • 云资源弹性管理的核心机制
  • 自动扩展监听器
    • 是实现动态扩展的核心部件
    • 监听负载状态以决定何时扩展
  • 动态扩展类型
    • 动态水平扩展
      • 向内或向外扩展IT资源实例
      • 自动扩展监听器请求资源复制，并发信号启动IT资源复制
    • 动态垂直扩展
      • 调整单个IT资源的处理容量
      • 向上或向下扩展IT资源实例
    • 动态重定位
      • 将IT资源重放置到更大/更小容量的主机上
  • 使用的机制
    • 云使用监控器
    • 虚拟机监控器
    • 按使用付费监控器

弹性资源容量架构

  • 弹性资源容量架构
    • 主要与虚拟服务器的动态供给相关
    • 根据负载变化分配和回收CPU与RAM资源
  • 扩展逻辑在智能自动化引擎
    • 自动扩展监听器发信号给智能引擎(一种脚本)
    • 智能引擎运行工作流逻辑，触发扩展
  • 扩展属于垂直扩展
  • 依赖一些额外机制
    • 云使用监控器
    • 按使用付费监控器
    • 资源复制
  • 变种
    • 包含工作流逻辑的脚本通过VIM发送其扩展请求
    • 而不是直接发送给虚拟机监控器

云爆发架构

  • 动态扩展架构的特殊形式
  • 云数据中心和企业IT资源协同
    • 企业内部IT资源<- -="">云IT资源
  • 基于云的IT资源是冗余性预部署
    • 保持非活跃状态，直到发生云爆发
  • 架构模型基础
    • 自动扩展监听器
    • 资源复制机制

弹性磁盘供给架构

  • 建立了一个动态存储供给系统，确保按照云用户实际使用的存储量进行精确计费
  • 基于自动精简供给技术
  • 辅助机制
    • 云使用监控器
    • 资源复制
  • 通常的收费方式：按照预定义的磁盘容量收费

冗余存储架构

  • 引入了复制的辅云存储设备作为故障系统的一部分，与主云存储设备中的数据保持同步
  • 主设备失效时，存储设备网关把用户请求转向辅设备
  • 主要依靠的的存储复制系统
    • 使得主云存储设备与其复制的辅云存储谁被保持同步
  • 通常部署在不同的地理区域
    • 抗灾容错
    • 经济成本
    • 法律问题

十二. 高级云架构

虚拟机监控器集群架构

  • 虚拟机监控器集群
    • 建立了一个跨多个物理服务器的高可用虚拟机监控器集群
    • 如果一个给定的虚拟机监控器或其底层物理服务器变得不可用，则被其托管的虚拟机服务器可迁移到另一物理服务器或虚拟机监控器上来保持运行时操作
  • 虚拟机监控器集群由中心VIM控制
    • 通过常规心跳消息来检测虚拟机监控器的状态
    • 当物理机或虚拟机监控器失效时进行在线迁移
    • 使用共享云存储设备实现在线迁移

粗粒度负载均衡与资源预留

  • 负载均衡的粒度有很多
  • 用户请求
    • 前端LB把用户请求分发给不同的云服务实例
  • 虚拟机服务器实例
    • 把虚拟服务器放置到不同的物理机
  • 云负载均衡
    • 数据中心间的负载均衡

负载均衡的虚拟服务器实例架构

  • 由于物理服务器之间保持跨服务器的工作负载均衡是很难实现的，运行和管理是相互隔离的，需要建立一个容量看门狗系统
  • 包含的机制
    • 虚拟机监控器
    • 资源集群
    • 虚拟服务器
    • (容量看门狗)云使用监控器
    • 自动伸缩监控器
    • 负载均衡器
    • 逻辑网络边界
    • 资源复制

容量看门狗系统

  • 动态地计算虚拟服务器实例及其相关的工作负载，再把处理任务分配到可用的物理服务器主机
  • 组成
    • 一个云使用监控器
    • VM在线迁移程序
    • 容量计划器

云负载均衡架构

  • 云负载均衡架构
    • 一个特殊的架构模型
    • IT资源可以在多个云之间进行负载均衡
  • 目的
    • 提高IT资源的性能和可扩展性
    • 增加IT资源的可用性和可靠性
    • 改进负载均衡和IT资源优化
  • 基础
    • 自动伸缩监听器
    • 故障转移系统

资源预留架构

  • 资源受限
    • 太多并发访问可能会导致运行时异常
    • IT资源没有足够的容量
  • 资源预留
    • 专门为给定的云用户保留
      • 单个IT资源
      • 一个IT资源的一部分
      • 多个IT资源
  • 依赖多个部件和机制
    • 审计监控器
    • 云使用监控器
    • 虚拟机监控器
    • 逻辑网路边界
    • 资源复制

云服务容错架构

  • 云服务不可用的原因
    • 运行时需求超出处理能力
    • 维护更新导致的暂时中断
    • 云服务迁移
    • 物理机失效、宕机
  • 应对机制
    • 服务迁移：不中断服务重定位架构
    • 服务故障检测与恢复：动态故障检测与恢复架构
    • 物理机容错

动态故障检测与恢复架构

  • 建立一个弹性的看门狗系统，以监控范围广泛的预先定义的故障场景，并对之做出相应
  • 弹性看门狗系统
    • 五个核心功能
      • 监视
      • 选定事件
      • 对事件做出反应
      • 报告
      • 升级处理
    • 智能看门狗监控器升级一个问题的处理最常采用的措施包括
      • 运行一个批处理文件
      • 发送一个控制台消息
      • 发送一条短信消息
      • 发送一份电子邮件消息
      • 发送一个SNMP陷阱
      • 记录一个通知单
  • 零宕机架构
    • 物理机不中断机制
    • 服务器会聚成一组，由容错系统控制
    • 虚拟服务器都存储在共享介质

裸机供给与快速供给架构

    • 指没有预装操作系统或其他任何软件的物理服务器

裸机供给架构

快速供给架构

    • 使得LUN可以均匀地分布在可用的云存储设备上
